## 1. Consultas Recursivas y Estructuras de Grafo (Recursive CTEs)

Las CTEs recursivas (`WITH RECURSIVE`) permiten recorrer relaciones jerárquicas o estructuras de grafos (árboles de decisión, organigramas de RRHH o listas de componentes) mediante la iteración matemática sobre un conjunto de tuplas hasta que la intersección de recursividad sea vacía ($\emptyset$).

$$\text{CTE}_{\text{recursiva}} = \text{Miembro Ancla} \cup \text{Miembro Recursivo}(\text{CTE}_{\text{recursiva}})$$

> [!IMPORTANT]
> **Anotaciones del Príncipe de las Bases de Datos:**
> **Cómo procesa físicamente PostgreSQL una CTE Recursiva:**
> 1. **Fase de Ancla (Anchor):** El motor ejecuta el primer `SELECT` (el caso base, ej. empleados que no reportan a nadie) y llena una tabla temporal interna llamada *Working Table*.
> 2. **Fase de Recursión:** Ejecuta el segundo `SELECT` (el miembro recursivo) uniendo la *Working Table* con la tabla física para producir un conjunto intermedio. Reemplaza el contenido de la *Working Table* con este conjunto intermedio y acumula las tuplas en la *Intermediate Table*.
> 3. **Finalización:** Repite el paso 2 de forma iterativa. El ciclo finaliza estrictamente cuando la *Working Table* queda vacía. *¡Truco del Maestro:* Usa siempre `UNION ALL` en lugar de `UNION` para evitar que PostgreSQL aplique un proceso de ordenamiento y eliminación de duplicados sumamente costoso en cada iteración del bucle.

1. **Reconstruir la jerarquía completa de la organización Northwind, mostrando los niveles jerárquicos desde la presidencia:**
   ```sql
   WITH RECURSIVE organigrama AS (
       -- Miembro Ancla: Empleados de máximo nivel (Andrew Fuller, reports_to IS NULL)
       SELECT 
           employee_id, 
           first_name || ' ' || last_name AS nombre_empleado, 
           reports_to, 
           0 AS nivel_jerarquico
       FROM employees
       WHERE reports_to IS NULL

       UNION ALL

       -- Miembro Recursivo: Une la iteración anterior con sus subordinados directos
       SELECT 
           e.employee_id, 
           e.first_name || ' ' || e.last_name, 
           e.reports_to, 
           o.nivel_jerarquico + 1
       FROM employees e
       INNER JOIN organigrama o ON e.reports_to = o.employee_id
   )
   SELECT * 
   FROM organigrama 
   ORDER BY nivel_jerarquico, reports_to;
   ```

## 2. Modelado y Pivote de Datos (Fila a Columna)

El pivote transforma registros verticales en vectores horizontales para análisis comparativos rápidos.

### Simulación Manual con CASE y GROUP BY

Sintaxis altamente compatible y portable entre cualquier base de datos relacional estándar.

1. **Obtener el total de pedidos anuales por cliente, representando cada año como una columna independiente:**
   ```sql
   SELECT
       c.company_name AS cliente,
       SUM(CASE WHEN EXTRACT(YEAR FROM o.order_date) = 1996 THEN 1 ELSE 0 END) AS pedidos_1996,
       SUM(CASE WHEN EXTRACT(YEAR FROM o.order_date) = 1997 THEN 1 ELSE 0 END) AS pedidos_1997,
       SUM(CASE WHEN EXTRACT(YEAR FROM o.order_date) = 1998 THEN 1 ELSE 0 END) AS pedidos_1998
   FROM customers c
   INNER JOIN orders o ON c.customer_id = o.customer_id
   GROUP BY c.company_name
   ORDER BY cliente;
   ```

### Pivote Físico con Extensión `tablefunc` (`crosstab`)

PostgreSQL ofrece una optimización nativa a través de funciones del sistema para pivotar datos velozmente sin evaluar cientos de cláusulas `CASE`.

1. **Instalar el módulo oficial en la base de datos:**
   ```sql
   CREATE EXTENSION IF NOT EXISTS tablefunc;
   ```

2. **Generar un reporte matricial de cantidad de pedidos mensuales por cliente usando `crosstab`:**
   ```sql
   SELECT * FROM crosstab(
       'SELECT 
           customer_id, 
           EXTRACT(MONTH FROM order_date)::int AS mes,
           COUNT(*)::int AS total_pedidos
        FROM orders 
        GROUP BY 1, 2 
        ORDER BY 1, 2',
       'SELECT generate_series(1, 12)'
   ) AS (
       customer_id VARCHAR, 
       ene INT, feb INT, mar INT, abr INT, may INT, jun INT, 
       jul INT, ago INT, sep INT, oct INT, nov INT, dic INT
   );
   ```

## 3. Analítica Avanzada con Marcos de Ventana (`WINDOW FRAMES`)

El marco de la ventana (`ROWS BETWEEN`) permite delimitar físicamente un subconjunto dinámico de filas alrededor de la fila activa para cómputos financieros complejos.

$$\text{Moving Average} = \frac{1}{K} \sum_{i=0}^{K-1} \text{total\_sale}_{t-i}$$

> [!TIP]
> **Anotaciones del Príncipe de las Bases de Datos:**
> **Entendiendo la Sintaxis del Marco de Ventana:**
> * `ROWS BETWEEN 2 PRECEDING AND CURRENT ROW`: Agrupa las 2 filas anteriores y la fila actual (total 3 filas). Óptimo para promedios móviles rápidos.
> * `ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW`: Rango desde el inicio del cursor hasta la fila actual. Es el comportamiento por defecto de `SUM(...) OVER (ORDER BY ...)`.
> * *Truco de Memoria:* Si la partición de tu ventana supera el tamaño físico de la memoria `work_mem` asignada a tu sesión, PostgreSQL escribirá archivos temporales en disco, degradando la consulta. Incrementa temporalmente `SET work_mem = '64MB';` antes de ejecutar analítica de millones de registros.

1. **Calcular el promedio móvil de facturación diaria considerando los 2 días anteriores y el día actual (Promedio Móvil de 3 Días):**
   ```sql
   WITH facturacion_diaria AS (
       SELECT o.order_date,
              SUM(od.quantity * od.unit_price * (1 - od.discount)) AS total_venta
       FROM orders o
       INNER JOIN order_details od ON o.order_id = od.order_id
       GROUP BY o.order_date
   )
   SELECT order_date, ROUND(total_venta::numeric, 2) AS venta_dia,
          ROUND(AVG(total_venta) OVER (
              ORDER BY order_date 
              ROWS BETWEEN 2 PRECEDING AND CURRENT ROW
          )::numeric, 2) AS promedio_movil_3_dias
   FROM facturacion_diaria;
   ```

## 4. Consultas Autoreferenciales Complejas (`SELF JOIN`)

Unir una relación consigo misma permite realizar comparaciones relacionales intraclase de alto nivel.

1. **Identificar todas las combinaciones únicas de clientes que comparten la misma ciudad de residencia:**
   ```sql
   SELECT c1.company_name AS cliente_a, c2.company_name AS cliente_b, c1.city
   FROM customers c1
   INNER JOIN customers c2 ON c1.city = c2.city AND c1.customer_id < c2.customer_id
   ORDER BY c1.city;
   ```

2. **Análisis de Canasta de Compra (Market Basket Analysis): Encontrar pares de productos que frecuentemente se compran juntos en la misma transacción:**
   ```sql
   SELECT p1.product_name AS producto_a, p2.product_name AS producto_b, COUNT(*) AS frecuencia_coocurrencia
   FROM order_details od1
   INNER JOIN order_details od2 ON od1.order_id = od2.order_id AND od1.product_id < od2.product_id
   INNER JOIN products p1 ON od1.product_id = p1.product_id
   INNER JOIN products p2 ON od2.product_id = p2.product_id
   GROUP BY p1.product_name, p2.product_name
   ORDER BY frecuencia_coocurrencia DESC
   LIMIT 10;
   ```

## 5. Análisis Comportamental de Cohortes (Retention Analytics)

El análisis de cohortes evalúa el comportamiento recurrente de grupos de usuarios adquiridos en un periodo común.

$$\text{Edad de Cohorte} = (\text{Año}_{\text{Orden}} - \text{Año}_{\text{Cohorte}}) \times 12 + (\text{Mes}_{\text{Orden}} - \text{Mes}_{\text{Cohorte}})$$

1. **Construir la matriz analítica de retención mensual de clientes por cohortes de adquisición:**
   ```sql
   WITH cohorte_adquisicion AS (
       -- Identificar la primera transacción del cliente (Fecha de Adquisición)
       SELECT customer_id, MIN(order_date) AS fecha_adquisicion
       FROM orders
       GROUP BY customer_id
   ),
   transacciones_mensuales AS (
       -- Agrupar las compras por meses únicos por cliente
       SELECT DISTINCT customer_id, DATE_TRUNC('month', order_date)::date AS mes_compra
       FROM orders
   ),
   metricas_cohorte AS (
       SELECT 
           DATE_TRUNC('month', ca.fecha_adquisicion)::date AS cohorte,
           -- Calcular el desplazamiento en meses (edad de la cohorte)
           EXTRACT(YEAR FROM AGE(tm.mes_compra, ca.fecha_adquisicion)) * 12 +
           EXTRACT(MONTH FROM AGE(tm.mes_compra, ca.fecha_adquisicion)) AS meses_desde_adquisicion,
           COUNT(DISTINCT tm.customer_id) AS clientes_recurrentes
       FROM cohorte_adquisicion ca
       INNER JOIN transacciones_mensuales tm ON ca.customer_id = tm.customer_id
       GROUP BY 1, 2
   )
   SELECT cohorte, meses_desde_adquisicion, clientes_recurrentes
   FROM metricas_cohorte
   ORDER BY cohorte, meses_desde_adquisicion;
   ```

## 6. Auditoría de Consistencia y Calidad de Datos

Identificación de anomalías físicas y violaciones de restricciones lógicas del negocio.

1. **Detectar pedidos huérfanos que no poseen registros de detalle en `order_details`:**
   ```sql
   SELECT o.order_id, o.customer_id, o.order_date
   FROM orders o
   LEFT JOIN order_details od ON o.order_id = od.order_id
   WHERE od.order_id IS NULL;
   ```

2. **Verificar la integridad del inventario de productos (evitar valores negativos no permitidos):**
   ```sql
   SELECT product_id, product_name, units_in_stock
   FROM products
   WHERE units_in_stock < 0;
   ```

## 7. Observabilidad, Planes de Ejecución y E/S (`EXPLAIN ANALYZE`)

La optimización de bases de datos requiere auditar el plan físico del *Query Planner* mediante la instrumentación de consultas.

> [!CAUTION]
> **Anotaciones del Príncipe de las Bases de Datos:**
> *¡Advertencia de Seguridad Crítica!* Cuando ejecutas `EXPLAIN`, el motor solo estima los costos matemáticos. Pero cuando ejecutas **`EXPLAIN ANALYZE`**, la consulta **SE EJECUTA REALMENTE** en la base de datos para medir los tiempos de CPU en milisegundos. 
> Por lo tanto, si aplicas un `EXPLAIN ANALYZE` sobre un comando `DELETE` o `UPDATE` destructivo, borrarás o alterarás los datos de producción de forma irreversible. Si auditas DML de escritura, envuélvelo siempre en una transacción que luego deshagas:
> `BEGIN; EXPLAIN ANALYZE UPDATE ...; ROLLBACK;`

1. **Diseccionar el costo físico de una consulta compleja con uniones multitabla:**
   ```sql
   EXPLAIN (ANALYZE, BUFFERS, COSTS, VERBOSE)
   SELECT c.company_name, p.product_name, SUM(od.quantity) AS total_comprado
   FROM customers c
   INNER JOIN orders o ON c.customer_id = o.customer_id
   INNER JOIN order_details od ON o.order_id = od.order_id
   INNER JOIN products p ON od.product_id = p.product_id
   WHERE c.country = 'Germany'
   GROUP BY c.company_name, p.product_name;
   ```

   **Cómo interpretar las métricas del plan de ejecución:**
   * **Seq Scan:** Escaneo secuencial (lectura completa de la tabla en disco). Debe evitarse en tablas masivas creando índices selectivos.
   * **Index Scan / Index Only Scan:** Acceso ultra veloz a través de la estructura jerárquica del índice (B-Tree).
   * **Buffers: shared hit=40 read=2:** Indica que 40 páginas de 8KB se leyeron directamente de la memoria caché RAM de PostgreSQL (`shared_buffers`) y solo 2 tuvieron que ser extraídas físicamente del disco. El éxito del DBA es lograr una tasa de acierto de caché (*Cache Hit Ratio*) superior al 99%.

## 8. Escenarios de Élite y Diseño de Entrevistas (Senior Data Architect)

1. **Obtener el segundo empleado que mayores ingresos acumulados por ventas ha generado:**
   ```sql
   WITH ventas_por_empleado AS (
       SELECT 
           e.employee_id,
           e.first_name || ' ' || e.last_name AS nombre_empleado,
           SUM(od.quantity * od.unit_price * (1 - od.discount)) AS total_ingresos,
           DENSE_RANK() OVER (ORDER BY SUM(od.quantity * od.unit_price * (1 - od.discount)) DESC) AS ranking_ventas
       FROM employees e
       INNER JOIN orders o ON e.employee_id = o.employee_id
       INNER JOIN order_details od ON o.order_id = od.order_id
       GROUP BY e.employee_id
   )
   SELECT nombre_empleado, ROUND(total_ingresos::numeric, 2) AS facturacion
   FROM ventas_por_empleado
   WHERE ranking_ventas = 2;
   ```

2. **Superpoder Lateral (`LATERAL JOIN`): Obtener de forma extremadamente eficiente los 3 productos más costosos por cada categoría registrada:**
   ```sql
   -- El operador LATERAL actúa como un bucle "FOR EACH" en SQL, permitiendo que la subconsulta interna
   -- haga referencia a atributos de la tabla externa en cada iteración de manera ultra veloz.
   SELECT c.category_name, p_lat.product_name, p_lat.unit_price
   FROM categories c
   CROSS JOIN LATERAL (
       SELECT p.product_name, p.unit_price
       FROM products p
       WHERE p.category_id = c.category_id
       ORDER BY p.unit_price DESC
       LIMIT 3
   ) AS p_lat;
   ```

3. **Calcular la tasa de contribución porcentual individual de cada producto sobre las ventas globales de la compañía:**
   ```sql
   WITH ventas_productos AS (
       SELECT p.product_name, SUM(od.quantity * od.unit_price * (1 - od.discount)) AS ingresos_producto
       FROM order_details od
       INNER JOIN products p ON od.product_id = p.product_id
       GROUP BY p.product_name
   ),
   ventas_globales AS (
       SELECT SUM(ingresos_producto) AS total_global FROM ventas_productos
   )
   SELECT 
       vp.product_name,
       ROUND(vp.ingresos_producto::numeric, 2) AS facturacion_individual,
       ROUND(((vp.ingresos_producto / vg.total_global) * 100)::numeric, 4) AS porcentaje_contribucion
   FROM ventas_productos vp, ventas_globales vg
   ORDER BY porcentaje_contribucion DESC;
   ```

## 9. Superpoderes Específicos de PostgreSQL

### Generación de Series Temporales (`generate_series`)

Permite fabricar tuplas dinámicas al vuelo, siendo una herramienta vital para la creación de dimensiones temporales en Data Warehousing.

1. **Generar un rango continuo de fechas correspondientes a los últimos 30 días naturales para reportes diarios vacíos:**
   ```sql
   SELECT generate_series(
       CURRENT_DATE - INTERVAL '30 days',
       CURRENT_DATE,
       '1 day'::interval
   )::date AS fecha_calendario;
   ```

### Inserciones y Modificaciones Retornables (`RETURNING`)

PostgreSQL permite recuperar los atributos generados (como valores auto-incrementados por secuencias) durante comandos DML sin tener que lanzar una consulta adicional al motor.

1. **Registrar un nuevo cliente en el sistema y capturar inmediatamente los datos creados:**
   ```sql
   INSERT INTO customers (customer_id, company_name, contact_name)
   VALUES ('NUEVO', 'PostgreSQL Inc.', 'Ing. de Datos')
   RETURNING customer_id, company_name, contact_name;
   ```

### Fusiones No Conflictivas (`UPSERT` / `ON CONFLICT`)

Manejo avanzado de colisiones de claves primarias o restricciones de unicidad.

1. **Intentar registrar un producto y, si ya existe por ID primario, actualizar únicamente su precio unitario al vuelo:**
   ```sql
   INSERT INTO products (product_id, product_name, unit_price)
   VALUES (1, 'Chai Reforzado', 19.50)
   ON CONFLICT (product_id)
   DO UPDATE SET unit_price = EXCLUDED.unit_price;
   ```

# Nivel 3 — Avanzado (Guía de estudio con ejercicios resueltos)

Objetivo: dominar CTEs recursivas, pivot, marcos de ventana avanzados, LATERAL, cohort analysis y EXPLAIN ANALYZE para optimización.

1) CTE recursiva — reconstruir organigrama (ejemplo resuelto)

```sql
WITH RECURSIVE organigrama AS (
  SELECT employee_id, first_name || ' ' || last_name AS nombre, reports_to, 0 AS nivel
  FROM employees
  WHERE reports_to IS NULL

  UNION ALL

  SELECT e.employee_id, e.first_name || ' ' || e.last_name, e.reports_to, o.nivel + 1
  FROM employees e
  JOIN organigrama o ON e.reports_to = o.employee_id
)
SELECT * FROM organigrama ORDER BY nivel, nombre;
```

2) Pivot (fila a columna) con CASE — ventas por año por cliente (ejemplo resuelto)

```sql
SELECT c.company_name,
  SUM(CASE WHEN EXTRACT(YEAR FROM o.order_date) = 1996 THEN 1 ELSE 0 END) AS pedidos_1996,
  SUM(CASE WHEN EXTRACT(YEAR FROM o.order_date) = 1997 THEN 1 ELSE 0 END) AS pedidos_1997,
  SUM(CASE WHEN EXTRACT(YEAR FROM o.order_date) = 1998 THEN 1 ELSE 0 END) AS pedidos_1998
FROM customers c
JOIN orders o ON c.customer_id = o.customer_id
GROUP BY c.company_name
ORDER BY c.company_name
LIMIT 50;
```

3) Ventanas y marcos — promedio móvil de 3 días (ejemplo resuelto)

```sql
WITH facturacion_diaria AS (
  SELECT o.order_date::date AS dia,
         SUM(od.quantity * od.unit_price * (1 - od.discount)) AS total
  FROM orders o
  JOIN order_details od ON o.order_id = od.order_id
  GROUP BY o.order_date::date
)
SELECT dia, ROUND(total::numeric,2) AS venta_dia,
       ROUND(AVG(total) OVER (ORDER BY dia ROWS BETWEEN 2 PRECEDING AND CURRENT ROW)::numeric,2) AS promedio_movil_3dias
FROM facturacion_diaria
ORDER BY dia DESC
LIMIT 30;
```

4) LATERAL — 3 productos más caros por categoría

```sql
SELECT c.category_name, p.product_name, p.unit_price
FROM categories c
CROSS JOIN LATERAL (
  SELECT product_name, unit_price
  FROM products p
  WHERE p.category_id = c.category_id
  ORDER BY unit_price DESC
  LIMIT 3
) p;
```

5) Cohort analysis — matriz mensual (ejemplo resuelto)

```sql
WITH first_order AS (
  SELECT customer_id, MIN(order_date)::date AS join_date
  FROM orders
  GROUP BY customer_id
),
months AS (
  SELECT customer_id, DATE_TRUNC('month', order_date)::date AS month
  FROM orders
  GROUP BY customer_id, DATE_TRUNC('month', order_date)
),
cohort AS (
  SELECT f.join_date::date AS cohorte, m.month,
    EXTRACT(YEAR FROM AGE(m.month, f.join_date)) * 12 + EXTRACT(MONTH FROM AGE(m.month, f.join_date)) AS months_from_join
  FROM first_order f
  JOIN months m ON f.customer_id = m.customer_id
)
SELECT cohorte, months_from_join, COUNT(*) AS customers_active
FROM cohort
GROUP BY cohorte, months_from_join
ORDER BY cohorte, months_from_join;
```

6) PL/pgSQL: función simple que retorna ingresos por empleado (resuelto)

```sql
CREATE OR REPLACE FUNCTION ingresos_por_empleado(emp_id INT)
RETURNS NUMERIC LANGUAGE plpgsql AS $$
DECLARE
  total NUMERIC := 0;
BEGIN
  SELECT SUM(od.quantity * od.unit_price * (1 - od.discount)) INTO total
  FROM orders o
  JOIN order_details od ON o.order_id = od.order_id
  WHERE o.employee_id = emp_id;

  RETURN COALESCE(total, 0);
END; $$;

-- Uso:
SELECT employee_id, first_name || ' ' || last_name AS empleado, ingresos_por_empleado(employee_id) AS ingresos
FROM employees
ORDER BY ingresos DESC
LIMIT 5;
```
