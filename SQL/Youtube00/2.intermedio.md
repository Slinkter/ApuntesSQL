## 1. El Modelo Relacional y la Combinación de Conjuntos (`JOIN`)

La combinación de relaciones permite reconstruir tuplas del mundo real a partir de datos normalizados. En el motor físico de PostgreSQL, los JOINs se resuelven a través de tres algoritmos principales:
* **Nested Loop Join:** Cruce de bucle básico, óptimo para conjuntos muy pequeños con índices idóneos.
* **Hash Join:** Construye una tabla Hash en memoria (`work_mem`) con la relación más pequeña para escanear la segunda.
* **Merge Join:** Ordena ambas relaciones por la clave de unión y las funde linealmente; óptimo para grandes volúmenes preordenados.

### LEFT OUTER JOIN

**Teoría Relacional:** Preserva la totalidad de las tuplas de la relación izquierda ($R$), completando con valores nulos (`NULL`) en los atributos de la relación derecha ($S$) si no existe correspondencia física.

$$R \text{ } \mathchoice{\text{⟕}}{\text{⟕}}{\text{⟕}}{\text{⟕}} \text{ } S$$

> [!TIP]
> **Anotaciones del Príncipe de las Bases de Datos:**
> El orden de declaración de las tablas en un `LEFT JOIN` es crítico. La tabla que contiene la cardinalidad maestra obligatoria va a la **izquierda**. 
> *Truco para detectar exclusión:* Si deseas encontrar los elementos de la izquierda que **no** tienen correspondencia en la derecha (diferencia de conjuntos $R - S$), añade una cláusula `WHERE S.clave_primaria IS NULL`. Es infinitamente más rápido que hacer un `NOT IN` sobre tablas con nulos.

1. **Listar todos los clientes y los pedidos que han realizado, incluyendo aquellos clientes que jamás han comprado:**
   ```sql
   SELECT c.customer_id, c.company_name, o.order_id, o.order_date
   FROM customers c
   LEFT JOIN orders o ON c.customer_id = o.customer_id;
   ```

2. **Mostrar todos los productos y sus cantidades vendidas, incluyendo aquellos productos que nunca se han vendido:**
   ```sql
   SELECT p.product_id, p.product_name, od.quantity
   FROM products p
   LEFT JOIN order_details od ON p.product_id = od.product_id;
   ```

### RIGHT OUTER JOIN

**Teoría Relacional:** Preserva la totalidad de las tuplas de la relación derecha ($S$), completando con nulos en los atributos de la izquierda ($R$).

$$R \text{ } \mathchoice{\text{⟖}}{\text{⟖}}{\text{⟖}}{\text{⟖}} \text{ } S$$

> [!WARNING]
> **Anotaciones del Príncipe de las Bases de Datos:**
> En la práctica del diseño de software, el `RIGHT JOIN` se considera un "antipadrón" de legibilidad. Cualquier consulta estructurada con `RIGHT JOIN` puede reescribirse de forma idéntica e intuitiva como un `LEFT JOIN` simplemente invirtiendo el orden de las relaciones en las cláusulas. Úsalo con extrema moderación para mantener tu código mantenible.

1. **Listar todos los empleados y los pedidos que han gestionado, incluyendo aquellos empleados sin ventas registradas:**
   ```sql
   SELECT e.employee_id, e.first_name || ' ' || e.last_name AS empleado, o.order_id
   FROM orders o
   RIGHT JOIN employees e ON o.employee_id = e.employee_id;
   ```

### Combinaciones Multitabla

Permiten enlazar $N$ relaciones a través de predicados encadenados.

1. **Para el pedido 10250, proyectar el nombre del cliente, nombre del producto comprado y la cantidad:**
   ```sql
   SELECT c.company_name AS cliente, p.product_name AS producto, od.quantity
   FROM orders o
   INNER JOIN customers c ON o.customer_id = c.customer_id
   INNER JOIN order_details od ON o.order_id = od.order_id
   INNER JOIN products p ON od.product_id = p.product_id
   WHERE o.order_id = 10250;
   ```

2. **Mostrar el ID de cada pedido junto con la razón social del cliente, el nombre del empleado que lo atendió y la empresa transportista:**
   ```sql
   SELECT o.order_id, c.company_name AS cliente, e.first_name || ' ' || e.last_name AS empleado, s.company_name AS transportista
   FROM orders o
   INNER JOIN customers c ON o.customer_id = c.customer_id
   INNER JOIN employees e ON o.employee_id = e.employee_id
   INNER JOIN shippers s ON o.ship_via = s.shipper_id;
   ```

## 2. Subconsultas y Abstracción de Datos

Las subconsultas (consultas anidadas) permiten utilizar el resultado de una proyección interna para resolver un predicado lógico externo.

### Subconsultas en la Cláusula `WHERE`

1. **Identificar todos los productos suministrados por proveedores ubicados en Japón:**
   ```sql
   SELECT product_name, unit_price
   FROM products
   WHERE supplier_id IN (
       SELECT supplier_id 
       FROM suppliers 
       WHERE country = 'Japan'
   );
   ```

2. **Listar los nombres de clientes que realizaron transacciones después del 1 de marzo de 1998:**
   ```sql
   SELECT company_name
   FROM customers
   WHERE customer_id IN (
       SELECT customer_id 
       FROM orders 
       WHERE order_date > '1998-03-01'
   );
   ```

### Subconsultas en la Cláusula `FROM` (Tablas Virtuales)

> [!IMPORTANT]
> **Anotaciones del Príncipe de las Bases de Datos:**
> En SQL Estándar y en el motor de PostgreSQL, **toda subconsulta declarada dentro de la cláusula `FROM` actúa como una tabla física temporal en memoria y debe tener asignado obligatoriamente un alias** (`AS nombre_tabla`). De lo contrario, el compilador lanzará un error sintáctico inmediato.

1. **Calcular el promedio general de la cantidad de ítems distintos contenidos por pedido:**
   ```sql
   SELECT AVG(order_counts.total_productos) AS promedio_items_por_pedido
   FROM (
       SELECT order_id, COUNT(product_id) AS total_productos 
       FROM order_details 
       GROUP BY order_id
   ) AS order_counts;
   ```

### Subconsultas Correlacionadas

Una subconsulta está correlacionada cuando la consulta interna hace referencia explícita a un atributo de la relación externa, forzando una evaluación dependiente por cada tupla analizada.

> [!CAUTION]
> **Anotaciones del Príncipe de las Bases de Datos:**
> *¡Peligro de Rendimiento!* Las subconsultas correlacionadas en la cláusula `WHERE` pueden deteriorar drásticamente la velocidad si la tabla externa es masiva, ya que pueden forzar un comportamiento similar a una complejidad algorítmica cuadrática $O(N^2)$ en el motor si el Query Planner no logra reescribirla. Evalúa alternativas como `LATERAL JOIN` o expresiones de tabla común (CTEs) para optimizar.

1. **Encontrar aquellos productos cuyo precio unitario supere la media aritmética de su respectiva categoría:**
   ```sql
   SELECT p1.product_name, p1.unit_price, p1.category_id
   FROM products p1
   WHERE p1.unit_price > (
       SELECT AVG(p2.unit_price) 
       FROM products p2 
       WHERE p2.category_id = p1.category_id
   );
   ```

## 3. Filtrado de Agregaciones (`HAVING`)

La cláusula `HAVING` aplica predicados lógicos restrictivos **después** de que las tuplas se han agrupado en conjuntos lógicos mediante `GROUP BY`.

> [!IMPORTANT]
> **Anotaciones del Príncipe de las Bases de Datos:**
> **La Diferencia de Oro: `WHERE` vs `HAVING`:**
> * `WHERE` filtra tuplas individuales **antes** de que ocurra el agrupamiento de datos. No puede contener funciones de agregación.
> * `HAVING` filtra grupos consolidados **después** de realizar el agrupamiento. Opera estrictamente sobre funciones de agregación.
> *El ciclo de vida física de una consulta sigue estrictamente el siguiente orden de ejecución en el motor:*
> $$\text{FROM} \rightarrow \text{WHERE} \rightarrow \text{GROUP BY} \rightarrow \text{HAVING} \rightarrow \text{SELECT (Proyección)} \rightarrow \text{DISTINCT} \rightarrow \text{ORDER BY} \rightarrow \text{LIMIT}$$

1. **Identificar los países que albergan a más de 5 clientes en nuestro registro:**
   ```sql
   SELECT country, COUNT(*) AS total_clientes
   FROM customers
   GROUP BY country
   HAVING COUNT(*) > 5;
   ```

2. **Listar los IDs de empleados que han procesado de forma acumulada más de 80 pedidos:**
   ```sql
   SELECT employee_id, COUNT(order_id) AS total_pedidos
   FROM orders
   GROUP BY employee_id
   HAVING COUNT(order_id) > 80;
   ```

3. **Mostrar los nombres de los proveedores que nos surten más de 3 productos distintos:**
   ```sql
   SELECT s.company_name, COUNT(p.product_id) AS total_productos
   FROM suppliers s
   INNER JOIN products p ON s.supplier_id = p.supplier_id
   GROUP BY s.supplier_id, s.company_name
   HAVING COUNT(p.product_id) > 3;
   ```

## 4. Errores Comunes en Cláusulas de Agrupamiento y Filtrado

1. **Uso Erróneo de Agregaciones en el `WHERE` (Lanzará error de compilación):**
   ```sql
   -- INCORRECTO: No se pueden evaluar agregados en la etapa de filtrado de tuplas individuales.
   SELECT country, COUNT(*)
   FROM customers
   WHERE COUNT(*) > 5
   GROUP BY country;
   -- ERROR: aggregate function calls cannot be used in WHERE
   ```

2. **Declaración Fuera de Secuencia Lógica (Lanzará error de compilación):**
   ```sql
   -- INCORRECTO: La cláusula WHERE no puede posicionarse después del agrupamiento físico.
   SELECT country, COUNT(*)
   FROM customers
   GROUP BY country
   WHERE country != 'USA';
   -- ERROR: syntax error at or near "WHERE"
   ```

3. **Corrección de Secuencia y Filtrado Combinado (Formulación Óptima):**
   ```sql
   -- CORRECTO: Filtra primero las tuplas con WHERE y luego restringe los grupos resultantes con HAVING.
   SELECT country, COUNT(*) AS total
   FROM customers
   WHERE country != 'USA'
   GROUP BY country
   HAVING COUNT(*) > 3;
   ```

## 5. Lógica Condicional Proyectiva (`CASE`)

El constructor `CASE` implementa la bifurcación lógica condicional (IF-THEN-ELSE) directamente en la proyección de datos a nivel de tupla.

1. **Clasificar los productos bajo categorías semánticas de costo basadas en su precio:**
   ```sql
   SELECT product_name, unit_price,
          CASE
              WHEN unit_price < 20 THEN 'Económico'
              WHEN unit_price BETWEEN 20 AND 50 THEN 'Intermedio'
              ELSE 'Premium'
          END AS categoria_costo
   FROM products;
   ```

2. **Evaluar el origen geográfico de los clientes para operaciones logísticas:**
   ```sql
   SELECT company_name, country,
          CASE
              WHEN country IN ('USA', 'Canada', 'Mexico') THEN 'América del Norte'
              WHEN country IN ('UK', 'Germany', 'France', 'Spain', 'Italy') THEN 'Europa'
              ELSE 'Resto del Mundo'
          END AS zona_logistica
   FROM customers;
   ```

## 6. Expresiones de Tabla Comunes (`WITH` - CTEs)

Una CTE (Common Table Expression) actúa como una relación temporal nombrada con alcance limitado a la ejecución de la consulta principal. Funciona como una excelente abstracción para simplificar consultas multitabla complejas y optimizar los planes del Query Planner en PostgreSQL.

> [!TIP]
> **Anotaciones del Príncipe de las Bases de Datos:**
> *¡Gran Atajo de Legibilidad!* Las CTEs son el mejor sustituto para las subconsultas anidadas en el `FROM`. Hacen que el código se lea secuencialmente de arriba hacia abajo (como programación imperativa estructurada) en lugar de adentro hacia afuera, lo que facilita enormemente la depuración y mantenimiento por otros ingenieros.

1. **Aislar clientes ubicados en Londres mediante una CTE y cruzar sus datos con sus respectivos pedidos:**
   ```sql
   WITH london_customers AS (
       SELECT customer_id, company_name
       FROM customers
       WHERE city = 'London'
   )
   SELECT lc.company_name AS cliente, o.order_id, o.order_date
   FROM london_customers lc
   INNER JOIN orders o ON lc.customer_id = o.customer_id;
   ```

2. **Calcular las ventas totales acumuladas por categoría de producto utilizando una estructura modular:**
   ```sql
   WITH product_sales AS (
       SELECT p.category_id, (od.quantity * od.unit_price * (1 - od.discount)) AS importe_venta
       FROM order_details od
       INNER JOIN products p ON od.product_id = p.product_id
   )
   SELECT cat.category_name, ROUND(SUM(ps.importe_venta)::numeric, 2) AS ventas_totales
   FROM product_sales ps
   INNER JOIN categories cat ON ps.category_id = cat.category_id
   GROUP BY cat.category_name
   ORDER BY ventas_totales DESC;
   ```

## 7. Introducción a Funciones de Ventana (`WINDOW FUNCTIONS`)

Las funciones de ventana ejecutan cálculos analíticos avanzados a través de un conjunto de tuplas directamente relacionadas con la tupla actual (llamado "partición" o "ventana"), **sin colapsar ni agrupar las tuplas del resultado**.

> [!IMPORTANT]
> **Anotaciones del Príncipe de las Bases de Datos:**
> **La esencia de las Funciones de Ventana:**
> A diferencia de `GROUP BY`, que consolida múltiples tuplas en una única fila de salida, las funciones de ventana conservan la identidad individual de cada fila en el cursor final. 
> *Ubicación del Cálculo:* Las funciones de ventana se ejecutan en la fase final de proyección del `SELECT`. Por lo tanto, no puedes utilizarlas directamente dentro de la cláusula `WHERE` para filtrar registros. Si necesitas filtrar por el resultado de una función de ventana (por ejemplo, obtener solo el Top 1), debes envolver la consulta en una CTE o una subconsulta externa.

### Rangos Secuenciales (`ROW_NUMBER`, `RANK`, `DENSE_RANK`)

Diferencias analíticas en el comportamiento ante colisiones de valores idénticos:
* `ROW_NUMBER()`: Asigna una secuencia estrictamente secuencial ($1, 2, 3, 4, \dots$).
* `RANK()`: Asigna rangos idénticos ante empates, pero genera saltos en la numeración subsiguiente ($1, 1, 3, 4, \dots$).
* `DENSE_RANK()`: Asigna rangos idénticos ante empates sin generar huecos en la secuencia ($1, 1, 2, 3, \dots$).

1. **Enumerar de forma secuencial y única los productos según su precio unitario decreciente:**
   ```sql
   SELECT product_name, unit_price,
          ROW_NUMBER() OVER (ORDER BY unit_price DESC) AS ranking_secuencial
   FROM products;
   ```

2. **Clasificar y comparar los precios de productos dentro de sus respectivas categorías:**
   ```sql
   SELECT category_id, product_name, unit_price,
          RANK() OVER (PARTITION BY category_id ORDER BY unit_price DESC) AS rango_con_saltos,
          DENSE_RANK() OVER (PARTITION BY category_id ORDER BY unit_price DESC) AS rango_denso
   FROM products;
   ```

### Desplazamientos Temporales (`LEAD` y `LAG`)

Permiten acceder a atributos pertenecientes a tuplas ubicadas a una distancia física relativa por delante (`LEAD`) o por detrás (`LAG`) de la tupla activa.

$$\text{LAG}(Col, Offset, Default)$$

1. **Visualizar la fecha de compra de cada pedido junto con la fecha del siguiente pedido efectuado por el mismo cliente:**
   ```sql
   SELECT customer_id, order_id, order_date,
          LEAD(order_date, 1) OVER (
              PARTITION BY customer_id 
              ORDER BY order_date
          ) AS fecha_siguiente_pedido
   FROM orders;
   ```

2. **Analizar la tendencia de consumo mostrando el importe del pedido actual frente al importe del pedido inmediatamente anterior del mismo cliente:**
   ```sql
   WITH customer_order_totals AS (
       SELECT o.customer_id, o.order_id, o.order_date,
              SUM(od.quantity * od.unit_price * (1 - od.discount)) AS total_pedido
       FROM orders o
       INNER JOIN order_details od ON o.order_id = od.order_id
       GROUP BY o.customer_id, o.order_id, o.order_date
   )
   SELECT customer_id, order_id, order_date, ROUND(total_pedido::numeric, 2) AS total_actual,
          ROUND(LAG(total_pedido, 1, 0::double precision) OVER (
              PARTITION BY customer_id 
              ORDER BY order_date
          )::numeric, 2) AS total_anterior
   FROM customer_order_totals;
   ```

### Acumulados y Aritmética de Ventana (`SUM` / `AVG` OVER)

1. **Proyectar la información detallada de cada producto junto al precio promedio global de su categoría asociada (sin colapsar la relación):**
   ```sql
   SELECT product_name, category_id, unit_price,
          ROUND(AVG(unit_price) OVER (PARTITION BY category_id)::numeric, 2) AS promedio_categoria,
          ROUND((unit_price - AVG(unit_price) OVER (PARTITION BY category_id))::numeric, 2) AS desviacion_respecto_promedio
   FROM products
   ORDER BY category_id, unit_price DESC;
   ```

2. **Calcular la suma acumulada de ingresos por ventas mes a mes en Northwind (Series de Tiempo):**
   ```sql
   WITH monthly_revenue AS (
       SELECT TO_CHAR(o.order_date, 'YYYY-MM') AS periodo_mes,
              SUM(od.quantity * od.unit_price * (1 - od.discount)) AS ingresos_mes
       FROM orders o
       INNER JOIN order_details od ON o.order_id = od.order_id
       GROUP BY periodo_mes
   )
   SELECT periodo_mes, ROUND(ingresos_mes::numeric, 2) AS ingresos_mes,
          ROUND(SUM(ingresos_mes) OVER (
              ORDER BY periodo_mes
              ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
          )::numeric, 2) AS ingresos_acumulados_historico
   FROM monthly_revenue;
   ```

## 8. Operaciones de Conjuntos Puros (`UNION` y `UNION ALL`)

Unifican los resultados de múltiples proyecciones en una única relación compatible (mismos tipos de datos en el mismo orden).
* `UNION`: Sigue la teoría relacional estricta de conjuntos y **elimina duplicados lógicos**, realizando un ordenamiento implícito costoso en memoria.
* `UNION ALL`: Sigue la teoría de multiconjuntos (bags), **preserva la totalidad de las tuplas incluyendo duplicados**, lo cual es infinitamente más rápido al no requerir ordenación ni hash en disco.

$$\text{UNION (Elimina Duplicados): } A \cup B$$
$$\text{UNION ALL (Preserva Duplicados): } A \uplus B$$

> [!TIP]
> **Anotaciones del Príncipe de las Bases de Datos:**
> *¡Atajo de Rendimiento Vital!* En el 90% de los casos de desarrollo de software donde necesitas unificar resultados, sabes de antemano que las colecciones no se cruzan o no te importa la existencia de duplicados. **Usa siempre `UNION ALL` en lugar de `UNION`**. Te ahorrará una operación masiva de ordenamiento en memoria (`Sort`) que PostgreSQL ejecuta en disco cuando el volumen supera la variable `work_mem`.

1. **Generar un listado de ciudades únicas donde residen nuestros clientes o proveedores:**
   ```sql
   SELECT city FROM customers WHERE city IS NOT NULL
   UNION
   SELECT city FROM suppliers WHERE city IS NOT NULL
   ORDER BY city;
   ```

2. **Consolidar una nómina combinada de nombres de contacto de clientes y empleados de Northwind:**
   ```sql
   SELECT contact_name AS nombre_contacto, 'Cliente' AS tipo_entidad FROM customers
   UNION ALL
   SELECT first_name || ' ' || last_name AS nombre_contacto, 'Empleado' AS tipo_entidad FROM employees;
   ```

## 9. Operadores de Comprobación y Cuantificadores (`EXISTS`, `ANY`, `ALL`)

Operan sobre subconsultas para validar condiciones lógicas existenciales.

1. **Determinar la lista de clientes que han efectuado por lo menos una compra en la plataforma (Semi-Join Relacional $\ltimes$):**
   ```sql
   SELECT customer_id, company_name
   FROM customers c
   WHERE EXISTS (
       SELECT 1 
       FROM orders o 
       WHERE o.customer_id = c.customer_id
   );
   ```

2. **Identificar aquellos productos cuyo precio supere el precio unitario de por lo menos algún producto perteneciente a la categoría número 1:**
   ```sql
   SELECT product_name, unit_price
   FROM products
   WHERE unit_price > ANY (
       SELECT unit_price 
       FROM products 
       WHERE category_id = 1
   );
   ```
# Nivel 2 — Intermedio (Guía de estudio con ejercicios resueltos)

Objetivo: trabajar con agregaciones, HAVING, subconsultas, CTEs y funciones de ventana básicas.

Ejecutar un ejercicio único (psql):

```bash
psql -h localhost -U slinkter -d northwind -c "<TU_QUERY>"
```

Ejercicios resueltos

1) Clientes por país (conteo) y filtrar países con > 5 clientes.

Solución:

```sql
SELECT country, COUNT(*) AS clientes
FROM customers
GROUP BY country
HAVING COUNT(*) > 5
ORDER BY clientes DESC;
```

2) Promedio de unidades por pedido (order_details) por producto, mostrar top 10.

Solución:

```sql
SELECT p.product_id, p.product_name, ROUND(AVG(od.quantity)::numeric,2) AS avg_qty
FROM order_details od
JOIN products p ON od.product_id = p.product_id
GROUP BY p.product_id, p.product_name
ORDER BY avg_qty DESC
LIMIT 10;
```

3) Usar una CTE para calcular facturación por pedido y mostrar pedidos con facturación > 1000.

Solución:

```sql
WITH facturacion AS (
  SELECT o.order_id, SUM(od.quantity * od.unit_price * (1 - od.discount)) AS total
  FROM orders o
  JOIN order_details od ON o.order_id = od.order_id
  GROUP BY o.order_id
)
SELECT order_id, ROUND(total::numeric,2) AS total
FROM facturacion
WHERE total > 1000
ORDER BY total DESC;
```

4) Encontrar los 3 clientes con mayor número de pedidos únicos usando ROW_NUMBER() (ventana).

Solución:

```sql
WITH pedidos_por_cliente AS (
  SELECT c.customer_id, c.company_name, COUNT(o.order_id) AS pedidos
  FROM customers c
  LEFT JOIN orders o ON o.customer_id = c.customer_id
  GROUP BY c.customer_id, c.company_name
)
SELECT customer_id, company_name, pedidos
FROM (
  SELECT *, ROW_NUMBER() OVER (ORDER BY pedidos DESC) AS rn
  FROM pedidos_por_cliente
) t
WHERE rn <= 3;
```

5) Subconsulta correlacionada: obtener la última fecha de pedido por cliente.

Solución:

```sql
SELECT c.customer_id, c.company_name,
  (SELECT MAX(order_date) FROM orders o WHERE o.customer_id = c.customer_id) AS ultima_compra
FROM customers c
ORDER BY ultima_compra DESC
LIMIT 10;
```

6) EXPLAIN ANALYZE: comparar una versión con índice y sin índice. Crear índice de ejemplo y medir:

```sql
-- crear índice para la prueba
CREATE INDEX IF NOT EXISTS idx_orders_customer ON orders(customer_id);

EXPLAIN (ANALYZE, BUFFERS)
SELECT o.order_id FROM orders o WHERE o.customer_id = 'ALFKI';
```

Consejo: cuando uses ventanas en tablas grandes, aumentar `work_mem` temporalmente para evitar spills a disco.
