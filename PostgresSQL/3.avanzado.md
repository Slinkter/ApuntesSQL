## 1. Consultas Recursivas (Recursive CTEs)

> **PostgreSQL Tip:** Las CTEs recursivas usan UNION ALL para combinar el caso base con la recursive parte.
> 
> **Conexión con Datos:** La tabla employees tiene una relación jerárquica (reports_to) que es un árbol/nivel. Las CTEs recursivas son ideales para árboles.
> 
> **Cómo funciona:**
> 1. **Anchor/Base case:** El primer SELECT (empleados sin jefe/reports_to = NULL)
> 2. **Recursive case:** Une la CTE consigo misma, buscando empleados cuyo reports_to coincide con el employee_id del nivel anterior
> 3. **Termination:** Se detiene cuando no hay más filas que coincidan

**Objetivo Académico:** Resolver problemas de grafos y jerarquías dentro de una base de datos relacional (como árboles de mando en RRHH o dependencias de productos).

68. **Mostrar la jerarquía de empleados, indicando quién reporta a quién.**
    _Nota: La tabla `employees` tiene una columna `reports_to` que se refiere al `employee_id` del jefe._

    ```sql
    WITH RECURSIVE employee_hierarchy AS (
        -- Anchor member: el nivel más alto (empleados que no reportan a nadie)
        SELECT 
            employee_id, 
            first_name || ' ' || last_name AS employee_name, 
            reports_to, 
            0 AS level
        FROM employees
        WHERE reports_to IS NULL

        UNION ALL

        -- Recursive member: une la CTE consigo misma para encontrar subordinados
        SELECT 
            e.employee_id, 
            e.first_name || ' ' || e.last_name, 
            e.reports_to, 
            eh.level + 1
        FROM employees e
        INNER JOIN employee_hierarchy eh ON e.reports_to = eh.employee_id
    )
    SELECT * FROM employee_hierarchy 
    ORDER BY level, reports_to;
    ```

69. **(Explicación paso a paso) Cómo resuelve PostgreSQL:**
    ```
    Level 0: Empleados con reports_to = NULL (el jefe máximo)
            → Nancy, Andrew (CEO/President)
    Level 1: Empleados que.reportan aLevel 0
            → Janet, Margaret, Steven, Laura (reportan a Andrew)
    Level 2: Empleados quereportan aLevel 1
            → Michael, Robert, Anne (reportan a Steven)
    ```
    El motorrecursivamentebusca hasta que no encuentra más subordinados.

## 2. Modelado de Salida Crosstab (Simulación de PIVOT)

> **PostgreSQL Tip:** PostgreSQL no tiene PIVOT nativo. Usamos CASE + GROUP BY para simularlo.
> 
> **Concepto:** Convertir filas en columnas (ej: meses → columnas separadas)
> 
> **Cuándo usar:** Tableros de control donde necesités ver tendencias mensuales como columnas

**Objetivo Académico:** Transformar registros verticales en estructuras horizontales de visualización (Matriz de datos), lo cual es fundamental para tableros de control ejecutivos.

69. **Mostrar el total de pedidos por año, con cada año en una columna separada para cada cliente.**

    ```sql
    SELECT
        c.contact_name,
        SUM(CASE WHEN EXTRACT(YEAR FROM o.order_date) = 1996 THEN 1 ELSE 0 END) AS orders_1996,
        SUM(CASE WHEN EXTRACT(YEAR FROM o.order_date) = 1997 THEN 1 ELSE 0 END) AS orders_1997,
        SUM(CASE WHEN EXTRACT(YEAR FROM o.order_date) = 1998 THEN 1 ELSE 0 END) AS orders_1998
    FROM customers c
    JOIN orders o ON c.customer_id = o.customer_id
    GROUP BY c.contact_name
    ORDER BY c.contact_name;
    ```

70. **Explicación del CASE en GROUP BY:**
    ```
    Para cada cliente:
    - CASE verifica: EXTRACT(YEAR FROM order_date) = 1996?
      - Si TRUE: SUM suma 1
      - Si FALSE: SUM suma 0
    - Al final del GROUP BY, tenemos la suma total por año
    ```

71. **Mostrar la cantidad total de productos vendidos por categoría, con cada categoría como una columna.**
    ```sql
    SELECT
        p.product_name,
        SUM(CASE WHEN c.category_name = 'Beverages' THEN od.quantity ELSE 0 END) AS Beverages,
        SUM(CASE WHEN c.category_name = 'Condiments' THEN od.quantity ELSE 0 END) AS Condiments,
        SUM(CASE WHEN c.category_name = 'Confections' THEN od.quantity ELSE 0 END) AS Confections
        -- ... (se pueden añadir más categorías)
    FROM order_details od
    JOIN products p ON od.product_id = p.product_id
    JOIN categories c ON p.category_id = c.category_id
    GROUP BY p.product_name
    LIMIT 10;
    ```

## 3. Funciones de Ventana Avanzadas

**Objetivo:** Utilizar marcos de ventana (`ROWS BETWEEN`) para cálculos como promedios móviles.

71. **Calcular el promedio móvil de 3 días de las ventas totales.**
    ```sql
    WITH daily_sales AS (
        SELECT
            order_date,
            SUM(od.unit_price * od.quantity) AS total_sale
        FROM orders o
        JOIN order_details od ON o.order_id = od.order_id
        GROUP BY order_date
    )
    SELECT
        order_date,
        total_sale,
        AVG(total_sale) OVER (ORDER BY order_date ROWS BETWEEN 2 PRECEDING AND CURRENT ROW) AS moving_avg_3_days
    FROM daily_sales;
    ```

## 4. Consultas Temporales y `SELF JOIN`

**Objetivo:** Unir una tabla consigo misma para comparar filas dentro de la misma tabla.

72. **Encontrar clientes que viven en la misma ciudad.**

    ```sql
    SELECT A.contact_name, B.contact_name, A.city
    FROM customers A, customers B
    WHERE A.customer_id < B.customer_id -- Evita duplicados (A,B) y (B,A) y auto-comparaciones (A,A)
    AND A.city = B.city
    ORDER BY A.city;
    ```

73. **Encontrar productos que a menudo se compran juntos (en el mismo pedido).**
    ```sql
    SELECT
        p1.product_name AS Product1,
        p2.product_name AS Product2,
        COUNT(*) AS frequency
    FROM order_details od1
    JOIN order_details od2 ON od1.order_id = od2.order_id AND od1.product_id < od2.product_id
    JOIN products p1 ON od1.product_id = p1.product_id
    JOIN products p2 ON od2.product_id = p2.product_id
    GROUP BY p1.product_name, p2.product_name
    ORDER BY frequency DESC
    LIMIT 10;
    ```

## 5. Análisis de Cohortes (Behavioral Analytics)

**Objetivo Académico:** Agrupar sujetos por su fecha de adquisición (cohorte) para medir la retención y el ciclo de vida del cliente a largo plazo. Es una métrica vital para el éxito del producto.

74. **Análisis de cohorte: Calcular la retención mensual de clientes.**
    ```sql
    WITH customer_first_order AS (
        SELECT
            customer_id,
            TO_CHAR(MIN(order_date), 'YYYY-MM') AS cohort_month
        FROM orders
        GROUP BY customer_id
    ),
    monthly_orders AS (
        SELECT DISTINCT
            customer_id,
            TO_CHAR(order_date, 'YYYY-MM') AS order_month
        FROM orders
    ),
    cohort_retention AS (
      SELECT
        cfo.cohort_month,
        EXTRACT(YEAR FROM AGE(TO_DATE(mo.order_month, 'YYYY-MM'), TO_DATE(cfo.cohort_month, 'YYYY-MM'))) * 12 + EXTRACT(MONTH FROM AGE(TO_DATE(mo.order_month, 'YYYY-MM'), TO_DATE(cfo.cohort_month, 'YYYY-MM'))) as months_since_first_order,
        COUNT(DISTINCT mo.customer_id) as retained_customers
      FROM customer_first_order cfo
      JOIN monthly_orders mo ON cfo.customer_id = mo.customer_id
      GROUP BY 1, 2
    )
    SELECT * FROM cohort_retention ORDER BY 1, 2;
    ```

## 7. Optimización y Observabilidad de Consultas (`EXPLAIN`, `EXPLAIN ANALYZE`)

> **PostgreSQL Tips:**
> - Usa `EXPLAIN ANALYZE` para ver ejecución real (no solo estimado).
> - `EXPLAIN (BUFFERS, ANALYZE)` muestra cachés y tiempos reales.
> - Busca `Seq Scan` en tablas grandes - indica necesidad de índice.
> - **Conexión con Internals:** El Query Planner decide cuántas páginas leer basándose en estadísticas.
> - Evita `Bitmap Heap Scan` excesivo indica tabla con mucho "Bloat" (requiere VACUUM).

**Objetivo Académico:** Diseccionar el plan de ejecución generado por el Query Planner de PostgreSQL para identificar ineficiencias físicas, lecturas masivas a disco y oportunidades de indexación.

78. **Analizar el plan de ejecución de una consulta compleja.**
    ```sql
    EXPLAIN ANALYZE
    SELECT c.contact_name, p.product_name
    FROM customers c
    JOIN orders o ON c.customer_id = o.customer_id
    JOIN order_details od ON o.order_id = od.order_id
    JOIN products p ON od.product_id = p.product_id
    WHERE c.country = 'Germany';
    ```
    **Cómo leer `EXPLAIN ANALYZE`:**
    - `Seq Scan on table`: Escaneo completo - malo en tablas grandes.
    - `Index Scan` / `Index Only Scan`: Uso de índice - bueno.
    - `Bitmap Heap Scan`: Escaneo con bitmap - aceptable para grandes resultados.
    - `Execution time`: Tiempo real de ejecución.
    - `Buffers`: Cuántas páginas se leyeron de disco vs caché.

    ```sql
    WITH employee_sales AS (
        SELECT
            e.employee_id,
            e.first_name || ' ' || e.last_name AS employee_name,
            SUM(od.quantity * od.unit_price) AS total_sales,
            DENSE_RANK() OVER (ORDER BY SUM(od.quantity * od.unit_price) DESC) AS sales_rank
    ```

    ```sql
    FROM categories c
    CROSS JOIN LATERAL (
        SELECT p.product_name, p.unit_price
        FROM products p
        WHERE p.category_id = c.category_id
    ```

81. **Calcular el porcentaje del total de ventas que representa cada producto.**
    ```sql
    WITH product_sales AS (

            SUM(od.quantity * od.unit_price) AS product_total
        FROM order_details od
        JOIN products p ON od.product_id = p.product_id

    total_sales AS (
        SELECT SUM(product_total) AS grand_total FROM product_sales

        ps.product_name,
    ```

91. **Obtener el crecimiento de ventas mes a mes en porcentaje.**

96. **Análisis de "Market Basket": ¿Qué producto se vende más frecuentemente con 'Chai'?**

    ```sql
    WITH chai_orders AS (
        -- Encontrar todos los pedidos que contienen 'Chai'
        SELECT DISTINCT od.order_id
        FROM order_details od
        JOIN products p ON od.product_id = p.product_id
        WHERE p.product_name = 'Chai'

    GROUP BY p.product_name
    ORDER BY frequency DESC
    LIMIT 5;
    ```

98. **Completar datos faltantes con ceros ( completar ventas por día):**
    ```sql
    WITH dates AS (
        SELECT generate_series(
            '1997-01-01'::date,
            '1997-12-31'::date,
            '1 day'::interval
        ) AS day
    ),
    sales AS (
        SELECT order_date::date AS day, SUM(od.quantity * od.unit_price) AS total
        FROM orders o
        JOIN order_details od ON o.order_id = od.order_id
        WHERE order_date BETWEEN '1997-01-01' AND '1997-12-31'
        GROUP BY day
    )
    SELECT d.day, COALESCE(s.total, 0) AS daily_sales
    FROM dates d
    LEFT JOIN sales s ON d.day = s.day
    ORDER BY d.day;
    ```

    ```sql
    ON CONFLICT (product_id) 
    DO UPDATE SET unit_price = EXCLUDED.unit_price;
    ```

### Arrays en PostgreSQL

> **PostgreSQL Tip:** PostgreSQL soporta arrays nativos, útil para tags/categorías.

101. **Crear tabla con columna array:**
    ```sql
    CREATE TABLE employees_v2 (
        employee_id SERIAL PRIMARY KEY,
        first_name VARCHAR(50),
        skills TEXT[]  -- Array de strings
    );
    
    INSERT INTO employees_v2 (first_name, skills)
    VALUES ('Juan', ARRAY['SQL', 'Python', 'Docker']);
    
    -- Buscar en array
    SELECT * FROM employees_v2 WHERE 'SQL' = ANY(skills);
    ```

### JSON/JSONB en PostgreSQL

> **PostgreSQL Tip:** JSONB es binario e indexable - ideal para semi-estructurados.

102. **Consulta sobre columna JSON:**
    ```sql
    -- Suponiendo que tienes una columna metadata JSONB
    SELECT * FROM products 
    WHERE metadata->>'category' = 'Beverages';
    
    -- Indexar paraJSON
    CREATE INDEX idx_products_metadata ON products USING GIN(metadata);
    ```

### Particionamiento (Table Partitioning)

> **PostgreSQL Tip:** Desde v10, tablas particionadas mejoran rendimiento en tablas grandes.

103. **Crear tabla particionada por rango de fechas:**
    ```sql
    CREATE TABLE orders_partitioned (
        order_id SERIAL,
        order_date DATE NOT NULL,
        customer_id VARCHAR(5)
    ) PARTITION BY RANGE (order_date);
    
    CREATE TABLE orders_1996 PARTITION OF orders_partitioned
        FOR VALUES FROM ('1996-01-01') TO ('1997-01-01');
    
    CREATE TABLE orders_1997 PARTITION OF orders_partitioned
        FOR VALUES FROM ('1997-01-01') TO ('1998-01-01');
    ```
