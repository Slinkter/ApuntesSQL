1. Subconsultas (Queries dentro de Queries)

[Continuación temporal... ~02:50:00]

Una subconsulta permite procesar datos en un paso previo antes de entregárselos a la consulta principal.

Subconsultas en la cláusula SELECT (Escalares)

Se puede inyectar el resultado de una función de agregación atada al registro actual directamente en la proyección.

SELECT 
    name AS usuario,
    (SELECT SUM(total) FROM orders WHERE user_id = u.id) AS total_gastado
FROM users AS u;

Subconsultas en el FROM (Tablas Temporales o Derivadas)

Podemos crear toda una estructura en memoria (tabla virtual) y seleccionarla desde afuera.

SELECT usuario, total_gastado 
FROM (
    -- Esta subconsulta se convierte en una "tabla virtual" llamada "resumen"
    SELECT u.name AS usuario, SUM(o.total) AS total_gastado
    FROM users AS u
    JOIN orders AS o ON u.id = o.user_id
    GROUP BY u.id, u.name
) AS resumen
WHERE total_gastado > 100;

2. Agrupamiento y Prevención de Nulos (GROUP BY, HAVING, COALESCE)

[~03:00:00]

La diferencia estructural entre WHERE y HAVING

📌 ÉNFASIS DEL PROFESOR:
El WHERE actúa sobre la tabla base antes de agrupar, afectando al FROM. El HAVING es una condición que actúa después del agrupamiento (sobre el SELECT), permitiendo filtrar los resultados de las matemáticas (ej. SUM, AVG). 
SELECT u.name, SUM(o.total) AS gastado
FROM users u
JOIN orders o ON u.id = o.user_id
GROUP BY u.id, u.name
HAVING gastado > 100;

Función COALESCE()

Al hacer sumatorias cruzadas con LEFT JOIN, los usuarios sin compras devuelven NULL en sus montos matemáticos, lo que puede romper el frontend al sumar. COALESCE actúa como un Fallback de seguridad (un valor por defecto ante el nulo).

-- Si la sumatoria es NULL, proyectará un 0.00
SELECT COALESCE(SUM(total), 0.00) AS gastado_seguro FROM orders;

3. Buenas Prácticas Arquitectónicas de Base de Datos

[~03:15:00]

El profesor dedica una sección específica a estándares puros de la industria:

  • UNSIGNED: Siempre añadir a los IDs. Un INT UNSIGNED evita números negativos y duplica la capacidad de IDs positivos disponibles.
  • Manejo de Contraseñas: "PROHIBIDO guardar contraseñas en texto plano". Almacenar siempre el HASH (ej. bcrypt) por lo que la longitud del VARCHAR debe ser amplia (255).
  • Soft Delete (Borrado Lógico): No eliminar información real de producción. Reemplazar la ejecución de DELETE FROM por agregar una columna activo BOOLEAN DEFAULT TRUE y simplemente efectuar un UPDATE a FALSE.

5. Backend con Node.js: RAW Queries vs ORM (Sequelize)

[~03:35:00]

El curso finaliza con dos ejercicios prácticos que exponen un backend Node.js.

Opción A: Consultas Crudas (Raw Queries)

Se importan drivers nativos (ej. mysql2). Obliga a escribir SQL a mano y se deben proteger inyecciones mediante queries parametrizadas (los signos de interrogación ?):

const query = 'INSERT INTO profesores (nombre, email, materia) VALUES (?, ?, ?)';
await db.query(query, [nombre, email, materia]);

Opción B: Uso de ORM (Sequelize)

Un ORM (Object-Relational Mapping) actúa como un puente que mapea clases (POO) a las tablas. Al usar ORM como Sequelize, el backend nunca escribe SQL directamente.

// Definición (DDL dinámico):
const Profesor = sequelize.define('profesores', {
    nombre: { type: DataTypes.STRING, allowNull: false }
});

// Inserción (DML dinámico):
await Profesor.create({ nombre, email, materia });

// Extracción (DQL dinámico):
const profesores = await Profesor.findAll({ where: { activo: true } });

Beneficios del ORM: Agilidad, código más limpio, y protección intrínseca contra SQL Injection.

GLOSARIO TÉCNICO

Subconsulta (Subquery):
Consulta anidada que alimenta datos a la consulta externa (ya sea calculando un campo individual o formando una tabla temporal en el FROM).

Soft Delete (Borrado Lógico):
Arquitectura de base de datos que prefiere mantener la información inhabilitando registros mediante una columna (estado `0` / `false` o fecha `deleted_at`) en lugar de usar un comando DELETE.

SQL Injection:
[def. ampliada] Vulnerabilidad de ciberseguridad donde un usuario maligno inyecta comandos SQL destructivos en los inputs del frontend (ej. introduciendo '; DROP TABLE users; --). Se mitiga usando parametrización ? o librerías ORM.

ORM (Object-Relational Mapping):
Técnica que sincroniza el lenguaje de programación orientado a objetos (Node.js/JavaScript, Python, Java) con las bases de datos relacionales, ahorrando la escritura manual de Queries SQL.

PREGUNTAS DE REPASO (Estilo Examen)

NIVEL 1 — Comprensión directa

P1: ¿Cuál es el riesgo de subir contraseñas en archivos de Docker Compose a GitHub?

R1: Constituye un grave incidente de seguridad, dejando la base de datos abierta al acceso de terceros. Se soluciona obligando el uso de archivos .env inyectados mediante interpolación (${VARIABLE}) y agregando el archivo .env al .gitignore.

NIVEL 2 — Aplicación

P2: Un reporte de BI arroja que algunos promedios están corrompidos porque usuarios sin compras devolvieron valores Nulos. ¿Qué función matemática arregla esto?

R2: La función COALESCE(). Modificando el script a COALESCE(AVG(compras), 0) forzamos al motor a procesar los vacíos lógicos como ceros aritméticos.

NIVEL 3 — Análisis / Síntesis

P3: Desde un enfoque de mantenibilidad y seguridad de software, ¿cuáles son los beneficios frente a las desventajas de adoptar un ORM (como Sequelize) en lugar de utilizar RAW Queries desde Node.js?

R3: Los beneficios son significativos: aumento en la velocidad de desarrollo (agilidad), tipado estructurado de objetos, protección automática contra SQL Injection y el control simplificado sobre bases de datos cruzadas (MySQL o PostgreSQL). La principal desventaja es una sutil pérdida de performance frente a queries puras o la incapacidad del ORM de generar un Query Planner perfecto en subconsultas matemáticas exageradamente complejas (para lo cual se siguen permitiendo queries crudas).

EJERCICIOS PRÁCTICOS

Ejercicio 1 — Optimización de Backend (Borrado Lógico)
Objetivo: Traducir operaciones físicas a operaciones lógicas y seguras.
Enunciado: En el endpoint /eliminar_usuario de tu API Express, los desarrolladores Junior dejaron esta query bruta: DELETE FROM users WHERE email = ?. Tu misión como Senior es evitar la eliminación real de la data implementando borrado lógico sin romper el funcionamiento esperado de "ocultamiento".
Solución guiada:

  1. Paso 1: Se debe efectuar un ALTER TABLE users ADD COLUMN is_active BOOLEAN DEFAULT TRUE; directamente en la BD.
  2. Paso 2: En la API Node.js, reescribir la query: UPDATE users SET is_active = false WHERE email = ?.
  3. Paso 3: Alterar todos los endpoints GET en el backend, añadiendo la condicional WHERE is_active = true.
