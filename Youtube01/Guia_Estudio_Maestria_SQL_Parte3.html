1. DQL Avanzado: Ampliando Filtros Lógicos

[Continuación temporal... ~02:05:00]

El profesor decide inyectar productos con valores extremos (un "Automóvil" muy caro y "Chicles" extremadamente baratos con miles en stock) para someter a estrés las consultas.

Filtros Relacionales y Múltiples (AND / OR)

Podemos exigir que una fila cumpla varias reglas simultáneas para ser proyectada en el resultado.

SELECT name, price 
FROM products 
WHERE stock >= 10 AND price > 10;

Operadores Específicos: BETWEEN, IN e IS NULL

Para simplificar sentencias matemáticas largas (ej. precio >= 50 AND precio <= 150), se utilizan operadores semánticos:

  • Rango (BETWEEN): WHERE price BETWEEN 50 AND 150. Incluye ambos bordes numéricos.
  • Conjuntos explícitos (IN): WHERE id IN (1, 2, 3). Evalúa coincidencia exacta con cualquiera de los ítems de la lista generada.
  • Validación de Nulos (IS NULL): WHERE description IS NULL. Jamás usar el operador = para igualar a nulo. Opcionalmente: IS NOT NULL.

2. Control de Salida: Ordenamiento y Alias

[~02:15:00]

DISTINCT (Evitando redundancias visuales)

Si la base de datos tiene varios registros iguales a nivel de nombre (Ej. varias 'Notebook'), el comando DISTINCT los filtra para traer valores únicos, SIEMPRE Y CUANDO no incluyamos el id (ya que el ID hace que la fila entera sea mecánicamente distinta).

SELECT DISTINCT name, price FROM products;

Aliases de Tablas y Columnas (AS)

El profesor remarca que los alias (AS) se usan extensamente en el desarrollo Backend para mapear datos en inglés (motor) hacia el frontend en español.

SELECT name AS producto, price AS precio_usd FROM products;

3. Funciones Nativas de Agregación

[~02:25:00]

Las bases de datos no solo guardan datos, sino que poseen un poderoso motor matemático nativo. Extraer la data calculada directamente por el motor de base de datos es significativamente más veloz que extraer 100,000 registros al backend y sumarlos con un bucle for en Node.js.

  • COUNT(*): SELECT COUNT(*) AS total_productos FROM products; Cuenta la cantidad de filas que arroja la consulta. No suma valores internos, solo cuenta existencias.
  • SUM(columna): SELECT SUM(total) AS ganancia_total FROM orders; Suma matemática de la columna.
  • AVG(columna): "Average" / Promedio. Útil para segmentación de mercado o inteligencia de negocios. SELECT AVG(price) FROM products;
  • MIN() / MAX(): Extraen el techo y el piso de una columna respectivamente.
El profesor agrupó todas las funciones en una sola consulta para demostrar su poder analítico:
SELECT COUNT(*) as cant_prod, AVG(price) as precio_promedio, SUM(stock) as stock_total FROM products;

4. Introducción a la Teoría de Conjuntos (JOINs)

[~02:35:00]

El corazón de SQL reside en las uniones relacionales (JOIN). Se usa cuando la información requerida está diseminada o normalizada en múltiples tablas.

INNER JOIN

El INNER JOIN exige correspondencia estricta. Devuelve únicamente los registros cuyas llaves (Primary y Foreign) tengan coincidencias en ambas tablas cruzadas.

SELECT u.name AS comprador, o.total 
FROM users AS u
INNER JOIN orders AS o ON u.id = o.user_id;
📌 ÉNFASIS DEL PROFESOR:
En los JOINs es obligatorio dominar los ALIAS de tabla (users AS u, orders AS o) porque al llamar columnas que se llaman igual (ej. id), el motor lanzará el error de "Columna Ambigua". Debes ser explícito: u.name, u.id, o.id.

5. LEFT JOIN en la Práctica

[~02:45:00]

A diferencia del estricto INNER JOIN, el LEFT JOIN da prioridad absoluta a la tabla definida en el FROM (la izquierda de la operación).

  • Trae TODOS los registros de la tabla izquierda, existan o no en la tabla derecha.
  • Si el registro izquierdo no tiene coincidencias en la derecha (ej. Un usuario que nunca compró nada), el motor rellena las columnas de órdenes con el valor NULL.
-- Traer a TODOS los usuarios y sus compras, incluso si no han comprado nada:
SELECT u.name, o.total 
FROM users AS u
LEFT JOIN orders AS o ON u.id = o.user_id;

Cadenas de JOINs Múltiples:

Las consultas pueden anidar tantos JOIN como relaciones existan. Para saber "quién compró qué y cuánto pagó", debemos atravesar 3 tablas: Usuarios, Órdenes y Productos.

SELECT u.name AS comprador, p.name AS producto, o.total
FROM orders AS o
INNER JOIN users AS u ON o.user_id = u.id
INNER JOIN products AS p ON o.product_id = p.id;

GLOSARIO TÉCNICO

INNER JOIN:
Tipo de cruce relacional que devuelve únicamente las filas que comparten coincidencia directa entre las llaves designadas en la cláusula ON.

LEFT JOIN:
Tipo de cruce que fuerza a proyectar absolutamente todas las filas de la primera tabla declarada (izquierda), inyectando datos vacíos (NULL) en los espacios de las columnas cruzadas que no hallaron coincidencia.

Alias (AS):
Funcionalidad nativa de SQL que permite renombrar temporalmente a nivel de proyección (salida) el identificador de una tabla o columna. Evita ambigüedades técnicas.

Columna Ambigua:
[def. ampliada] Error reportado por el motor SQL cuando se hace un SELECT de una columna llamada (ej.) "id" habiendo unido dos tablas que poseen una columna con el mismo nombre. Se soluciona obligando el uso de prefijos de alias (u.id o o.id).

PREGUNTAS DE REPASO (Estilo Examen)

NIVEL 1 — Comprensión directa

P1: ¿Cuál es el propósito exacto de la cláusula ON dentro de un JOIN?

R1: La cláusula ON establece la condición o "puente" lógico bajo el cual el motor debe entrelazar las dos tablas. Generalmente, iguala la Llave Primaria de una tabla con la Llave Foránea de otra (ej. ON usuarios.id = ordenes.user_id).

P2: ¿Para qué sirve el modificador DISTINCT?

R2: Para limpiar el conjunto de resultados (la respuesta de la BBDD) eliminando filas que sean exactamente idénticas en todas las columnas proyectadas por el SELECT.

NIVEL 2 — Aplicación

P3: Eres encargado del reporte financiero. Requieres saber cuál fue el monto máximo pagado en una sola orden de compra por un usuario en toda la historia de la plataforma. ¿Qué función matemática usas?

R3: Utilizaría la función de agregación máxima: SELECT MAX(total) FROM orders;

NIVEL 3 — Análisis / Síntesis

P4: Imagina que en un sistema escolar solicitan un reporte de todos los alumnos registrados y las materias en las que están inscritos. El sistema utiliza un INNER JOIN y el departamento se queja de que "han desaparecido alumnos de la base de datos". ¿Qué falló arquitectónicamente en la query?

R4: El error fue utilizar un INNER JOIN. Al ser una intersección estricta, el motor ignoró y ocultó a todos los alumnos recién registrados que aún no se habían inscrito en ninguna materia. La query correcta debió utilizar un LEFT JOIN (FROM alumnos LEFT JOIN materias), lo cual mostraría a los alumnos sin materias indicando NULL en los campos académicos, sin "desaparecerlos" del reporte.

EJERCICIOS PRÁCTICOS (Basados en el Repositorio Oficial)

Ejercicio 1 — Análisis de Videojuegos (GameHub)
Objetivo: Calcular métricas de tiempo jugado uniendo 3 tablas.
Enunciado: Escribe una consulta que liste el título del videojuego y el promedio de horas jugadas por sesión. Ordena el resultado desde el juego con el promedio más alto al más bajo.
Solución guiada:

SELECT g.title, ROUND(AVG(s.hours_played), 2) AS promedio_horas
FROM games g
JOIN sessions s ON g.id = s.game_id
GROUP BY g.title
ORDER BY promedio_horas DESC;

Ejercicio 2 — Análisis de Restaurantes (FoodieTrack)
Objetivo: Encontrar los usuarios más activos del sistema combinando funciones de agregación.
Enunciado: Se requiere listar a todos los usuarios junto con la cantidad exacta de reseñas que han escrito en la plataforma. Ordena la lista empezando por los que más reseñas tienen.
Solución guiada:

SELECT u.username, COUNT(rv.id) AS cantidad_reseñas
FROM users u
JOIN reviews rv ON u.id = rv.user_id
GROUP BY u.username
ORDER BY cantidad_reseñas DESC;

Ejercicio 3 — Uso de HAVING vs WHERE
Enunciado: Siguiendo con la plataforma de restaurantes, ahora el CEO solo quiere ver la lista de restaurantes que tengan más de 1 reseña.
Solución guiada:

SELECT r.name AS restaurante, COUNT(rv.id) AS cantidad_reseñas
FROM restaurants r
JOIN reviews rv ON r.id = rv.restaurant_id
GROUP BY r.name
HAVING COUNT(rv.id) > 1;

Nota: Usamos HAVING porque COUNT(rv.id) es una función matemática que solo existe después del GROUP BY.
