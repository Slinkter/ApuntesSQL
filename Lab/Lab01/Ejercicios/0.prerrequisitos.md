## 1. Conceptos Base

### 1.1 ¿Qué es SQL?

**SQL (Structured Query Language)** es el lenguaje estándar universal para comunicarse con bases de datos relacionales. Nació en los años 70 en IBM bajo el nombre original de SEQUEL y hoy es soportado por prácticamente todos los sistemas de bases de datos del mundo.

> **Analogía:** Piensa en SQL como el **idioma inglés del mundo de las bases de datos**. Así como puedes usar inglés para comunicarte en un hotel en Japón, en un restaurante en Francia o en una conferencia en Alemania, puedes usar SQL para hablar con PostgreSQL en Perú, con MySQL en Brasil o con Oracle en Estados Unidos. Es el lenguaje común que todos entienden.

Con SQL puedes realizar 4 tipos de operaciones fundamentales:

| Operación | Subconjunto SQL | Comandos principales | Ejemplo en Northwind |
|-----------|----------------|---------------------|----------------------|
| **Definir estructuras** | DDL (Data Definition Language) | `CREATE`, `ALTER`, `DROP` | Crear la tabla `products` con sus columnas |
| **Manipular datos** | DML (Data Manipulation Language) | `INSERT`, `UPDATE`, `DELETE` | Insertar un nuevo producto en `products` |
| **Consultar datos** | DQL (Data Query Language) | `SELECT` | Consultar todos los productos con precio mayor a $30 |
| **Controlar acceso** | DCL (Data Control Language) | `GRANT`, `REVOKE` | Dar permisos de solo lectura a un usuario |

```sql
-- Ejemplo de cada subconjunto en acción:

-- DDL: Crear una tabla
CREATE TABLE audit_log (
    log_id SERIAL PRIMARY KEY,
    action TEXT NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- DML: Insertar un registro
INSERT INTO audit_log (action) VALUES ('Usuario conectado');

-- DQL: Consultar registros
SELECT * FROM audit_log WHERE action = 'Usuario conectado';

-- DCL: Dar permisos (requiere superusuario)
GRANT SELECT ON audit_log TO report_user;
```

#### SQL vs NoSQL: ¿Cuándo usar cada uno?

| Característica | SQL (Relacional) | NoSQL (No relacional) |
|----------------|------------------|----------------------|
| **Estructura** | Esquema fijo (tablas con columnas definidas) | Esquema flexible (documentos, grafos, clave-valor) |
| **Consistencia** | ACID — transacciones confiables | BASE — disponibilidad sobre consistencia |
| **Consultas** | Lenguaje estándar SQL | APIs propietarias por cada sistema |
| **Escalado** | Vertical (más CPU/RAM al servidor) | Horizontal (más servidores en red) |
| **Relaciones** | Excelentes (JOINs entre tablas) | Limitadas (duplicación de datos) |
| **Ejemplos** | PostgreSQL, MySQL, Oracle, SQL Server | MongoDB, Redis, Cassandra, Neo4j |
| **Cuándo usar** | Datos estructurados, integridad crítica, reportes complejos | Datos semi-estructurados, alta escritura, escalabilidad masiva |

> **Regla práctica en enterprise:** Si tus datos tienen relaciones claras (clientes → pedidos → productos) y necesitas integridad transaccional (banca, e-commerce, inventarios), usa SQL. Si necesitas escalabilidad masiva y tu esquema cambia frecuentemente (logs, IoT, redes sociales), considera NoSQL.

### 1.2 ¿Qué es PostgreSQL?

**PostgreSQL es un sistema de gestión de bases de datos relacional (RDBMS) de código abierto**, considerado el sucesor académico de la investigación original de Berkeley sobre bases de datos. Actualmente está en su versión 17 y es utilizado por empresas como Apple, Instagram, Spotify y el government de nhiều países.

> **Analogía:** Si los RDBMS fueran autos, PostgreSQL sería un **vehículo híbrido de última generación**: tiene la fiabilidad de un Toyota (cumplimiento ACID), la potencia de un BMW (funciones avanzadas como CTEs recursivas, JSONB, ventanas), y la personalización de un Tesla (extensibilidad total). MySQL, en comparación, sería un sedan confiable y rápido pero con menos features premium.

#### Características principales de PostgreSQL

| Característica | Qué significa | Por qué importa |
|----------------|---------------|-----------------|
| **ACID** | Atomicidad, Consistencia, Aislamiento, Durabilidad | Tus transacciones son confiables: si algo falla, todo se revierte. Sin pérdida de datos. |
| **SQL estándar avanzado** | Soporta CTEs, ventanas, JSONB, rangos, arrays | Puedes resolver problemas complejos en una sola consulta sin código adicional. |
| **MVCC** | Multi-Version Concurrency Control | Las lecturas nunca bloquean escrituras y viceversa. Ideal para sistemas multi-usuario. |
| **Extensibilidad** | Puedes crear tipos, operadores, funciones e índices personalizados | Si PostgreSQL no tiene lo que necesitas, lo creas tú mismo. |
| **JSONB nativo** | Almacena y consulta JSON de forma binaria y eficiente | Puedes usar PostgreSQL como base de datos relacional Y como NoSQL en uno solo. |
| **GPL/Open Source** | Licencia gratuita, sin costos de usuario | Cero costo de licenciamiento, ideal para startups y enterprise. |

#### MySQL vs PostgreSQL: Comparación para profesionales

| Aspecto | MySQL | PostgreSQL |
|---------|-------|------------|
| **Licencia** | GPL (Oracle) | PostgreSQL License (más permisiva) |
| **ACID completo** | Solo con InnoDB (motor por defecto) | Siempre, en todos los motores |
| **Tipos de dato** | Básicos (INT, VARCHAR, TEXT) | Avanzados (JSONB, arrays, rangos, hstore, geometric) |
| **CTEs** | Soporte básico (8.0+) | Soporte completo + recursivos + MATERIALIZED/NOT MATERIALIZED |
| **Ventanas (WINDOW)** | Soporte parcial (8.0+) | Soporte completo y optimizado |
| **FULL OUTER JOIN** | No soportado | Soportado |
| **UPSERT** | `INSERT ... ON DUPLICATE KEY UPDATE` | `INSERT ... ON CONFLICT DO UPDATE` (más expresivo) |
| **RETURNING** | No soportado (necesitas 2 queries) | Soportado — devuelve datos de la fila afectada |
| **Particionamiento** | Limitado (MySQL 8.0+) | Nativo y declarativo (desde PostgreSQL 10) |
| **MVCC** | Implementación parcial | Completo — lecturas nunca bloquean escrituras |
| **Extensibilidad** | Plugins limitados | Extensiones completas (PostGIS, pg_trgm, pgvector) |
| **Escritura** | Más rápido en carga simple | Más lento en carga masiva, más rápido en consultas complejas |
| **Comunidad** | Oracle (empresa privada) | Comunidad open source global |

> **Cuándo elegir PostgreSQL en un proyecto enterprise:**
> - Necesitas integridad transaccional completa (banca, pagos)
> - Trabajas con datos geoespaciales (PostGIS es el estándar de la industria)
> - Necesitas JSONB para datos semi-estructurados sin cambiar de base de datos
> - Tu equipo值a la estabilidad y estándares SQL sobre la velocidad de escritura bruta
> - Estás en cloud: AWS RDS, Google Cloud SQL y Azure Database soportan PostgreSQL nativamente

### 1.3 Modelo Relacional

El **modelo relacional** es la forma de organizar datos en tablas (llamadas "relaciones") donde cada fila representa una entidad y cada columna una propiedad de esa entidad. Fue inventado por Edgar F. Codd en 1970 y sigue siendo la base de casi todas las bases de datos del mundo empresarial.

> **Analogía:** Imagina una hoja de cálculo de Excel, pero con reglas estrictas:
> - Cada columna tiene un tipo de dato definido (no puedes poner texto en una columna de fechas)
> - Cada fila es única (no puedes tener dos clientes con el mismo ID)
> - Las celdas están relacionadas entre hojas (el pedido de un cliente apunta a ese cliente específico)
> - Las reglas se enforced automáticamente por la base de datos (si intentas borrar un cliente con pedidos, PostgreSQL te lo impide)

#### Componentes fundamentales

| Componente | Definición | Ejemplo en Northwind |
|------------|-----------|----------------------|
| **Tabla (Relación)** | Conjunto de filas con la misma estructura | `customers`, `orders`, `products` |
| **Fila (Tupla/Registro)** | Una instancia de la entidad | El cliente "ALFKI" de Alfreds Futterkiste |
| **Columna (Atributo)** | Una propiedad de la entidad | `company_name`, `order_date`, `unit_price` |
| **Clave Primaria (PK)** | Columna o combinación que identifica **de forma única** cada fila | `customer_id` en `customers`, `(order_id, product_id)` en `order_details` |
| **Clave Foránea (FK)** | Columna que referencia la PK de otra tabla, creando una relación | `orders.customer_id` → `customers.customer_id` |

#### Ejemplo con datos reales de Northwind

```
 customers                     orders                    order_details
┌──────────────┐          ┌─────────────────┐        ┌───────────────────────┐
│ customer_id  │          │ order_id (PK)   │        │ order_id (PK, FK)     │
│ company_name │          │ customer_id(FK) │        │ product_id (PK, FK)   │
│ contact_name │          │ order_date      │        │ unit_price            │
│ country      │          │ ship_country    │        │ quantity              │
└──────┬───────┘          └────────┬────────┘        └───────────┬───────────┘
       │                           │                             │
       │   ┌───────────────────────┘                             │
       │   │                                                     │
       └───┤ orders.customer_id = customers.customer_id          │
           │                                                     │
           └──────────────── order_details.order_id = orders.order_id
                                                       │
                                         order_details.product_id = products.product_id
```

```sql
-- Consulta que demuestra las 3 tablas relacionadas
SELECT
    c.company_name,          -- de customers
    o.order_date,            -- de orders
    p.product_name,          -- de products
    od.quantity,             -- de order_details
    od.unit_price            -- de order_details
FROM order_details od
JOIN orders o     ON od.order_id = o.order_id
JOIN customers c  ON o.customer_id = c.customer_id
JOIN products p   ON od.product_id = p.product_id
LIMIT 5;
```

#### Patrones de relación en bases de datos reales

| Patrones | Qué es | Ejemplo en Northwind | Diagrama |
|----------|--------|----------------------|----------|
| **1:N (Uno a Muchos)** | Una fila de A se relaciona con muchas filas de B | Un cliente tiene muchos pedidos | `customers (1) ──< (N) orders` |
| **N:N (Muchos a Muchos)** | Muchas filas de A se relacionan con muchas filas de B, vía tabla puente | Un pedido tiene muchos productos, un producto está en muchos pedidos | `orders ──< order_details >── products` |
| **Autorreferencial** | Una tabla se relaciona consigo misma | Un empleado reporta a otro empleado | `employees ──< employees` (via `reports_to`) |

```sql
-- Ejemplo de relación autorreferencial: jerarquía de empleados
SELECT
    e.first_name || ' ' || e.last_name AS empleado,
    COALESCE(m.first_name || ' ' || m.last_name, '(CEO)') AS reporta_a
FROM employees e
LEFT JOIN employees m ON e.reports_to = m.employee_id
ORDER BY e.reports_to;
```

### 1.4 Índices

**Un índice es una estructura de datos auxiliar que acelera significativamente la búsqueda de filas** en una tabla. Sin un índice, PostgreSQL debe leer toda la tabla fila por fila (escaneo secuencial). Con un índice, salta directamente a las filas que necesitas.

> **Analogía (mejorada):** Imagina que necesitas encontrar "el capítulo sobre transacciones" en un libro de 500 páginas:
> - **Sin índice:** Abres el libro en la página 1 y lees cada página hasta encontrar la palabra "transacciones". Si el libro tiene 500 páginas, podrías necesitar leer 400 páginas antes de encontrarlo. → **Esto es un Seq Scan.**
> - **Con índice:** Miras el índice alfabético al final del libro. Dice "transacciones — página 347". Abres directamente en la página 347. → **Esto es un Index Scan.**
> - **Con un índice compuesto (como un índice temático):** Si buscas "transacciones en PostgreSQL", el índice te dice primero "PostgreSQL (páginas 100-200)" y luego "transacciones (página 156)". → **Esto es un Index Scan con predicado de columna.**

#### Tipos de índices en PostgreSQL

| Tipo de índice | Cuándo usarlo | Ejemplo | Rendimiento |
|----------------|---------------|---------|-------------|
| **B-tree** (default) | Columnas con valores ordenables (=, <, >, BETWEEN, IS NULL) | `CREATE INDEX idx ON orders(order_date)` | Rápido para búsquedas de rango y igualdad |
| **Hash** | Solo igualdad exacta (=) | `CREATE INDEX idx ON customers USING hash(customer_id)` | Más rápido que B-tree para igualdad pura, pero no soporta rangos |
| **GiST** | Datos geoespaciales,全文, rangos, arrays | `CREATE INDEX idx ON locations USING gist(geom)` | Esencial para PostGIS y búsqueda textual |
| **GIN** | Arrays, JSONB,全文 (Full-Text Search) | `CREATE INDEX idx ON products USING gin(search_vector)` | Rápido para contiene (@>) y coincidencia textual |
| **BRIN** | Tablas非常大 con datos naturalmente ordenados (timestamps, secuencias) | `CREATE INDEX idx ON logs USING brin(created_at)` | Muy pequeño, ideal para tablas de billones de filas |

```sql
-- Ejemplo práctico: crear un índice y verificar su uso
-- Paso 1: Crear índice en la columna de búsqueda
CREATE INDEX idx_orders_customer ON orders(customer_id);

-- Paso 2: Verificar que PostgreSQL lo usa
EXPLAIN SELECT * FROM orders WHERE customer_id = 'ALFKI';

-- Output esperado:
-- Index Scan using idx_orders_customer on orders
--   Index Cond: (customer_id = 'ALFKI'::character(5))

-- Paso 3: Comparar con y sin índice
-- Sin índice: Seq Scan (lee las 830 filas)
-- Con índice: Index Scan (salta directamente a las filas de ALFKI)
```

#### ¿Cuándo crear un índice?

| Señal | Acción recomendada |
|-------|-------------------|
| `Seq Scan` en tabla > 10,000 filas en un `WHERE` frecuente | Crear índice en la columna del `WHERE` |
| `Seq Scan` en un `JOIN`频繁 | Crear índice en la columna de JOIN (FK) |
| Consultas `ORDER BY` lentas | Crear índice en la columna de ordenamiento |
| Tabla con muchos `NULL` y consultas que filtran por `IS NOT NULL` | Considerar índice parcial: `CREATE INDEX idx ON t(col) WHERE col IS NOT NULL` |
| Tabla pequeña (< 1,000 filas) | Generalmente NO necesita índice (el Seq Scan es suficiente) |

### 1.5 SET search_path

```sql
SET search_path TO public;
```

**`search_path` es el directorio de búsqueda de PostgreSQL.** Cuando escribes `SELECT * FROM customers`, PostgreSQL busca la tabla `customers` en cada schema listado en `search_path`, en orden. Por defecto busca en `public`.

> **Analogía:** Es como la variable `PATH` en tu terminal. Cuando escribes `python` en la terminal, el sistema busca en `/usr/bin`, `/usr/local/bin`, etc. Hasta encontrar el ejecutable. PostgreSQL hace lo mismo con los schemas.

```sql
-- Ver el search_path actual
SHOW search_path;

-- Resultado por defecto: "$user", public
-- PostgreSQL primero busca en un schema con tu nombre de usuario,
-- luego en "public"

-- En este laboratorio siempre estamos en "public"
SET search_path TO public;
```

> **En enterprise:** En producción, normalmente tienes múltiples schemas: `public` (datos de la aplicación), `audit` (logs de cambios), `reporting` (vistas materializadas). El `search_path` controla a cuál accedes por defecto.

## 2. Setup del Laboratorio (Docker)

> **Analogía del laboratorio virtual:** Docker es como un **laboratorio de química completamente equipado** que puedes encender y apagar con un botón. En lugar de instalar PostgreSQL directamente en tu computadora (como si construyeras un laboratorio propio), Docker te da un contenedor aislado con todo listo. Si algo sale mal, simplemente destruyes el contenedor y creas uno nuevo — sin dejar residuos en tu sistema operativo.

### 2.1 ¿Por qué Docker en vez de instalación nativa?

| Aspecto | Instalación nativa | Docker |
|---------|-------------------|--------|
| **Setup** | Descargar, instalar, configurar, crear BD | Un solo comando |
| **Aislamiento** | Se instala en tu sistema, puede conflictuar con otras versiones | Contenedor完全 aislado |
| **Cleanup** | Desinstalar manualmente, limpiar archivos residuales | `docker rm` — limpio en 1 segundo |
| **Portabilidad** | Funciona solo en tu máquina | Funciona igual en Windows, Mac, Linux, CI/CD |
| **Múltiples versiones** | Difícil (PostgreSQL 14 y 16 en la misma máquina) | Fácil (cada contenedor con su versión) |
| **Reproducibilidad** | "Funciona en mi máquina" | "Funciona en cualquier máquina con Docker" |

### 2.2 Requisitos de Software

- **Docker Desktop** (Windows/Mac) o **Docker Engine** (Linux) — v24+
- Al menos **4 GB de RAM libre** para el contenedor
- Opcional: `psql` cliente instalado localmente

### 2.3 Método Rápido: Docker Standalone

```powershell
# PowerShell (Windows)
docker run --name pg_northwind_lab ^
  -e POSTGRES_USER=postgres ^
  -e POSTGRES_PASSWORD=postgres ^
  -e POSTGRES_DB=northwind ^
  -v "${PWD}/db_northwind.sql:/docker-entrypoint-initdb.d/db_northwind.sql" ^
  -p 5432:5432 ^
  -d postgres:16-alpine
```

```bash
# Bash (Linux/Mac)
docker run --name pg_northwind_lab \
  -e POSTGRES_USER=postgres \
  -e POSTGRES_PASSWORD=postgres \
  -e POSTGRES_DB=northwind \
  -v "$(pwd)/../db_northwind.sql:/docker-entrypoint-initdb.d/db_northwind.sql" \
  -p 5432:5432 \
  -d postgres:16-alpine
```

### 2.4 ¿Qué hace `docker-entrypoint-initdb.d/`?

Es una carpeta especial dentro de la imagen oficial de PostgreSQL. Cuando el contenedor arranca **por primera vez** y la base de datos no existe, PostgreSQL ejecuta automáticamente todos los archivos `.sql` o `.sh` que encuentre ahí, en orden alfabético.

Esto significa que **no necesitas ejecutar nada manualmente** — el script `db_northwind.sql` se ejecuta solo.

### 2.5 Verificar que la Base Está Lista

```bash
# Ver logs
docker logs pg_northwind_lab

# Buscar la línea de éxito
docker logs pg_northwind_lab 2>&1 | grep "ready for start up"
```

## 3. Comandos Esenciales de psql

**`psql` es el cliente interactivo de PostgreSQL** — la herramienta de línea de comandos que te permite conectarte a una base de datos y ejecutar consultas directamente.

> **Analogía:** psql es como la **terminal SSH para bases de datos**. Así como usas SSH para conectarte a un servidor remoto y ejecutar comandos, usas psql para conectarte a PostgreSQL y ejecutar consultas SQL. Es la forma más directa y rápida de interactuar con la base de datos, sin necesidad de interfaces gráficas.

### 3.1 Conexión

```bash
# Conectar desde Docker
docker exec -it pg_northwind_lab psql -U postgres -d northwind

# Conectar desde local (si psql está instalado)
psql -h localhost -U postgres -d northwind
```

### 3.2 Meta-comandos (empiezan con `\`)

| Comando | Qué hace | Analogía |
|---------|----------|----------|
| `\l` | Lista bases de datos | `ls` en el directorio raíz |
| `\dt` | Lista tablas de la base actual | `ls` en el directorio actual |
| `\dt public.*` | Lista tablas del schema public | `ls -la public/` |
| `\d customers` | Describe la tabla (columnas, tipos, constraints) | `cat schema.sql` de esa tabla |
| `\di` | Lista índices | `ls *.idx` |
| `\dv` | Lista vistas | `ls *.view` |
| `\df` | Lista funciones | `ls *.func` |
| `\x` | Modo expandido (columnas como filas) | `grep -A 20 "resultado"` — útil para registros anchos |
| `\timing` | Mide tiempo de cada query | Cronómetro automático |
| `\i archivo.sql` | Ejecuta un archivo .sql | `bash script.sh` |
| `\o archivo.txt` | Redirige output a archivo | `>` o `tee` en bash |
| `\e` | Abre editor externo para escribir queries | `$EDITOR` para SQL |
| `\q` | Salir de psql | `exit` |
| `\?` | Ayuda de meta-comandos | `man psql` |
| `\h SELECT` | Ayuda sobre sintaxis de SELECT | `help select` |
| `\conninfo` | Información de la conexión actual | `whoami && hostname` |

## 4. Esquema Northwind

**Northwind es una base de datos de ejemplo que modela una empresa ficticia de distribución de alimentos.** Es el "Hello World" de las bases de datos relacionales: lo suficientemente simple para entender rápido, pero lo suficientemente compleja para practicar JOINs, agregaciones y subconsultas.

> **Por qué Northwind:** En lugar de crear tablas abstractas ("tabla_A", "tabla_B"), Northwind simula un negocio real: clientes que hacen pedidos, productos que se venden, proveedores que fabrican, transportistas que envían. Esto te permite practicar con escenarios que encontrarás en cualquier empresa de e-commerce, retail o distribución.

### 4.1 Diagrama ER (Entidad-Relación)


### 4.2 Patrones de Relación en Northwind

Northwind demuestra los 3 patrones fundamentales de diseño relacional:

#### Patrón 1: Uno a Muchos (1:N)

```
categories (1) ──< (N) products
Un categoría tiene muchos productos, un producto pertenece a una categoría.

customers (1) ──< (N) orders
Un cliente hace muchos pedidos, un pedido pertenece a un solo cliente.

employees (1) ──< (N) orders
Un empleado gestiona muchos pedidos, un pedido es gestionado por un empleado.
```

```sql
-- Ejemplo: ver cuántos productos tiene cada categoría
SELECT
    c.category_name,
    COUNT(p.product_id) AS total_productos
FROM categories c
LEFT JOIN products p ON c.category_id = p.category_id
GROUP BY c.category_name
ORDER BY total_productos DESC;
```

#### Patrón 2: Muchos a Muchos (N:N) vía tabla puente

```
orders (N) ──< order_details >── (N) products
Un pedido tiene muchos productos, un producto está en muchos pedidos.
La tabla "order_details" es la tabla puente (junction table) que resuelve la relación N:N.
```

> **Por qué no se puede hacer N:N directamente:** En el modelo relacional, una FK apunta a una sola fila de otra tabla. Si un pedido tiene 10 productos, ¿a cuál apuntaría `orders.product_id`? La solución es crear una tabla intermedia (`order_details`) donde cada fila es una combinación (pedido, producto) con su cantidad y precio.

```sql
-- Ejemplo: ver el detalle de un pedido específico
SELECT
    p.product_name,
    od.quantity,
    od.unit_price,
    od.discount,
    (od.quantity * od.unit_price * (1 - od.discount)) AS total_linea
FROM order_details od
JOIN products p ON od.product_id = p.product_id
WHERE od.order_id = 10248;
```

#### Patrón 3: Autorreferencial (Self-referencing)

```
employees (1) ──< (N) employees
Un empleado reporta a otro empleado (jerarquía organizacional).
La FK "reports_to" apunta a "employee_id" de la misma tabla.
```

```sql
-- Ejemplo: ver la jerarquía de empleados
SELECT
    e.first_name || ' ' || e.last_name AS empleado,
    e.title AS cargo,
    COALESCE(m.first_name || ' ' || m.last_name, '(CEO / Sin jefe)') AS reporta_a
FROM employees e
LEFT JOIN employees m ON e.reports_to = m.employee_id
ORDER BY e.reports_to NULLS FIRST;
```

### 4.3 Tipos de Datos PostgreSQL en Northwind

| Tipo PostgreSQL | Tamaño | Equivalente | Uso en Northwind | Cuándo usarlo |
|-----------------|--------|-------------|------------------|---------------|
| `smallint` | 2 bytes | INT2 | `category_id`, `supplier_id`, `quantity` | Enteros pequeños (IDs, cantidades, stocks) |
| `integer` | 4 bytes | INT4 | `discontinued` | Enteros estándar |
| `numeric` | Variable | DECIMAL | `unit_price`, `freight`, `discount` | Dinero, porcentajes — **nunca usa `float` para dinero** |
| `varchar(n)` | Variable | STRING | `company_name`, `contact_name` | Texto con longitud máxima conocida |
| `text` | Variable | TEXT | `description`, `notes` | Texto libre sin límite de longitud |
| `date` | 4 bytes | DATE | `order_date`, `birth_date` | Solo fecha (sin hora) |
| `bytea` | Variable | BLOB | `picture`, `photo` | Datos binarios (imágenes, archivos) |

> **Analogía `numeric` vs `float`:** ¿Preferirías que tu banco de alimentos usara `float` para calcular tus pagos? Un `float` es como un cálculo con regla de tres: rápido pero impreciso (`0.1 + 0.2 = 0.30000000000000004`). Un `numeric` es como una calculadora de contabilidad: exacto (`0.1 + 0.2 = 0.3`). Para dinero, SIEMPRE usa `numeric`.

### 4.4 Convención de Columnas

| Columna | Tipo | Descripción |
|---------|------|-------------|
| `unit_price` | `numeric` | Precio sin pérdida de precisión |
| `quantity` | `integer` | Cantidad de ítems |
| `discount` | `numeric` | Descuento en porcentaje (0.00 a 0.25) |
| `customer_id` | `character(5)` | Siempre 5 caracteres, PK de customers |
| `order_date` | `date` | Fecha del pedido (sin hora) |

### 4.5 Diseño de Constraints

| Constraint | Tipo | Ejemplo en Northwind | Propósito |
|------------|------|----------------------|-----------|
| **PRIMARY KEY** | `pk_orders` | `order_id` en `orders` | Garantiza unicidad — no puede haber dos pedidos con el mismo ID |
| **FOREIGN KEY** | `fk_orders_customers` | `customer_id` en `orders` → `customers` | Integridad referencial — no puede existir un pedido de un cliente inexistente |
| **NOT NULL** | Implícito en PKs | Todas las PKs | No permite valores nulos en la columna identificadora |
| **UNIQUE** | No definido en Northwind | — | Northwind no usa UNIQUE (en producción sería útil en `email`, `phone`) |
| **CHECK** | No definido en Northwind | — | Northwind no usa CHECK (en producción: `CHECK (unit_price > 0)`) |

> **Nota sobre Northwind:** El esquema original no incluye constraints `CHECK` ni `UNIQUE`. En un sistema de producción, agregarías validaciones como `CHECK (quantity > 0)`, `CHECK (discount >= 0 AND discount <= 1)`, o `UNIQUE (email)` en la tabla de usuarios.

## 5. Cómo Leer EXPLAIN (ANALYZE, BUFFERS)

**`EXPLAIN` es el GPS de tus consultas SQL.** Antes de ejecutar una ruta, un GPS te muestra el camino óptimo: "toma la autopista, luego gira a la derecha, evita el tráfico del centro". PostgreSQL hace exactamente lo mismo con tus consultas: analiza la tabla, estima costos, elige el mejor camino y te muestra el plan de ejecución.

> **Analogía completa:** Imagina que necesitas ir del aeropuerto de Lima al centro de la ciudad:
> - **Sin EXPLAIN:** Tomas el primer camino que ves y esperas que sea el más rápido (puedes terminar en El Rospo).
> - **Con EXPLAIN:** El GPS te dice: "Hay 3 rutas. La ruta A toma 25 min (autopista, 15km), la ruta B toma 40 min (vía expresa, 12km), la ruta C toma 60 min (callejera, 8km). Te recomiendo la ruta A."
> - **Con EXPLAIN ANALYZE:** El GPS te dice lo mismo, pero además te dice: "Tardé exactamente 23 minutos y recorrí 14.8km." (datos reales, no estimados).

### 5.1 Sintaxis

```sql
EXPLAIN (ANALYZE, BUFFERS) SELECT ... ;
```

| Flag | Qué hace | Cuándo usarlo |
|------|----------|---------------|
| `EXPLAIN` solo | Muestra el plan **estimado** (sin ejecutar) | Para ver qué hará PostgreSQL sin gastar recursos |
| `EXPLAIN ANALYZE` | **Ejecuta** la query y muestra tiempos reales | Para medir performance real |
| `EXPLAIN ANALYZE, BUFFERS` | Agrega información de **E/S de disco** | Para diagnosticar problemas de I/O |
| `EXPLAIN ANALYZE, BUFFERS, FORMAT JSON` | Salida en JSON para análisis programático | Para integrar con herramientas de monitoreo |

### 5.2 Ejemplo completo

```sql
EXPLAIN (ANALYZE, BUFFERS)
SELECT c.company_name, COUNT(o.order_id) AS pedidos
FROM customers c
LEFT JOIN orders o ON c.customer_id = o.customer_id
WHERE c.country = 'Mexico'
GROUP BY c.company_name;
```

### 5.3 Cómo pensar como el optimizador

Cuando PostgreSQL recibe una consulta, sigue estos pasos internos (que EXPLAIN te muestra):

| Paso | Qué hace PostgreSQL | Pregunta que se responde |
|------|---------------------|--------------------------|
| **1. Analiza tablas** | ¿Qué tablas necesito? ¿Cuántas filas tiene cada una? | `Seq Scan on customers c` |
| **2. Estima cardinalidad** | ¿Cuántas filas cree que coincidirán con el `WHERE`? | `rows=5` (estima 5 clientes en México) |
| **3. Elige algoritmo de JOIN** | ¿Hash Join, Nested Loop o Merge Join? | `Hash Right Join` |
| **4. Estima costos** | ¿Cuánto cuesta cada operación? (lectura de disco, CPU, memoria) | `cost=... rows=... width=...` |
| **5. Ejecuta el plan** | Sigue el árbol de nodos de abajo hacia arriba | `actual time=0.45..0.47` |

### 5.4 Interpretación de cada elemento

| Elemento | Significa | Qué mirar |
|----------|-----------|-----------|
| `Seq Scan` | Escaneo secuencial (lee toda la tabla) | Normal en tablas pequeñas,Problemático en tablas grandes |
| `Index Scan` | Escaneo por índice (salta a filas específicas) | Lo que quieres ver en consultas WHERE |
| `Hash Join` | Construye hash de una tabla y escanea la otra | Óptimo para JOINs sin índice en la columna de unión |
| `Nested Loop` | Para cada fila de A, busca en B | Óptimo para conjuntos pequeños o con índice |
| `Merge Join` | Ordena ambas tablas y las fusiona | Óptimo para datos preordenados o con índice |
| `cost=...` | Costo estimado (unidad arbitraria, comparativo) | No es tiempo real — es una estimación |
| `actual time=0.45..0.47` | Tiempo real en ms (inicio..fin del nodo) | El tiempo real de ejecución de ese nodo |
| `rows=5` | Filas reales producidas por ese nodo | Si difiere mucho del estimado → estadísticas desactualizadas |
| `loops=1` | Veces que se ejecutó este nodo | En Nested Loop, puede ser alto |
| `Buffers: shared hit=16` | 16 bloques leídos desde caché de memoria | Rápido — datos en RAM |
| `Buffers: shared read=10` | 10 bloques leídos desde disco | Lento — posible bottleneck de I/O |
| `Planning Time` | Tiempo que el optimizador tardó en elegir el plan | Si es alto, la query tiene muchas tablas o complejidad |
| `Execution Time` | Tiempo total de ejecución | El número que más te importa para performance |

### 5.5 Ejemplo práctico: antes/después de crear un índice

```sql
-- ANTES del índice: Seq Scan en 830 filas
EXPLAIN ANALYZE SELECT * FROM orders WHERE customer_id = 'ALFKI';
-- Seq Scan on orders  (cost=0.00..22.30 rows=1 width=...)
--   Filter: (customer_id = 'ALFKI'::character(5))
--   Rows Removed by Filter: 829
-- Execution Time: 0.15 ms  (pero lee TODA la tabla)

-- Crear el índice
CREATE INDEX idx_orders_customer ON orders(customer_id);

-- DESPUÉS del índice: Index Scan directo
EXPLAIN ANALYZE SELECT * FROM orders WHERE customer_id = 'ALFKI';
-- Index Scan using idx_orders_customer on orders
--   Index Cond: (customer_id = 'ALFKI'::character(5))
-- Execution Time: 0.05 ms  (salta directamente a la fila)
```

### 5.6 Señales de alarma en EXPLAIN

| Señal | Significado | Solución |
|-------|-------------|----------|
| `Seq Scan` en tabla > 10,000 filas | PostgreSQL lee toda la tabla | Crear índice en la columna del `WHERE` |
| `Buffers: shared read` alto | Muchas lecturas de disco | Aumentar `shared_buffers` o crear índices |
| Diferencia grande entre `rows` estimado y real | Estadísticas desactualizadas | Ejecutar `ANALYZE nombre_tabla` |
| `Sort Method: external merge Disk` | Sorting usingando disco (lento) | Aumentar `work_mem` o crear índice en la columna de `ORDER BY` |
| `Nested Loop` con `loops=1000` | Muchas iteraciones | Considerar un `Hash Join` o índice compuesto |
| `Planning Time` > `Execution Time` | El optimizador tarda más que la ejecución | Simplificar la consulta o reducir el número de tablas |

## 6. Transacciones: BEGIN / COMMIT / ROLLBACK

**Una transacción es una unidad atómica de trabajo: o se ejecutan TODAS las operaciones dentro de ella, o no se ejecuta NINGUNA.** Es el mecanismo fundamental que garantiza la integridad de datos en sistemas de producción.

> **Analogía del punto de restauración:** Imagina que estás jugando un videojuego con sistema de guardado:
> - **BEGIN** = Creas un punto de guardado nuevo
> - **INSERT/UPDATE/DELETE** = Avanzas en el juego (matas monstruos, recoges tesoros)
> - **COMMIT** = Guardas el juego permanentemente (los cambios son irreversibles)
> - **ROLLBACK** = Cargas el punto de guardado anterior (todo vuelve al estado anterior, como si no hubieras hecho nada)

### 6.1 Flujo Básico

```sql
-- Iniciar transacción (crea un punto de guardado)
BEGIN;

INSERT INTO orders (order_id, customer_id, employee_id, order_date)
VALUES (99999, 'ALFKI', 1, CURRENT_DATE);

UPDATE customers
SET contact_name = 'Nuevo Contacto'
WHERE customer_id = 'ALFKI';

-- Confirmar cambios (punto de guardado permanente)
COMMIT;
```

### 6.2 Rollback (deshacer)

```sql
BEGIN;
DELETE FROM orders WHERE order_id = 99999;
-- Ups, no quería borrar ese pedido
ROLLBACK;  -- todo vuelve al estado anterior, como si el DELETE nunca hubiera existido
```

### 6.3 Transacciones Anidadas con SAVEPOINT

```sql
BEGIN;
  INSERT INTO customers ...;
  SAVEPOINT sp1;               -- creamos un sub-punto de guardado
  INSERT INTO orders ...;      -- esto falla
  ROLLBACK TO SAVEPOINT sp1;   -- deshace solo el INSERT orders
  -- customers insertado se conserva
COMMIT;
```

> **En producción:** Los SAVEPOINT son útiles cuando una transacción grande puede fallar parcialmente. Por ejemplo, al importar 1000 registros, si el registro 500 falla, puedes hacer ROLLBACK TO SAVEPOINT antes del 500 y CONTINUAR con el resto.

### 6.4 Niveles de Aislamiento

| Nivel | Qué garantiza | Problema que resuelve | Performance | Uso en PostgreSQL |
|-------|--------------|----------------------|-------------|-------------------|
| **Read Uncommitted** | Puede leer datos no confirmados (dirty reads) | Nada | Más rápido | En PostgreSQL se comporta como Read Committed (MVCC) |
| **Read Committed** | Solo lee datos confirmados | Dirty reads | Bueno | **Por defecto en PostgreSQL** — el más usado |
| **Repeatable Read** | Las lecturas ven el mismo snapshot durante la transacción | Non-repeatable reads | Moderado | Útil para reportes que necesitan consistencia |
| **Serializable** | Transacciones se ejecutan como si fueran secuenciales | Phantom reads, serialization anomalies | Más lento | Para integridad crítica (banca, pagos) |

```sql
-- Ejemplo: cambiar nivel de aislamiento
BEGIN;
SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;
SELECT COUNT(*) FROM orders WHERE customer_id = 'ALFKI';  -- ve 5 pedidos
-- otro usuario inserta un pedido de ALFKI mientras tanto
SELECT COUNT(*) FROM orders WHERE customer_id = 'ALFKI';  -- AÚN ve 5 pedidos (mismo snapshot)
COMMIT;
```

### 6.5 Race Condition: el problema que las transacciones resuelven

```sql
-- SIN transacción (Peligroso):
-- Usuario A: Lee stock = 10
-- Usuario B: Lee stock = 10
-- Usuario A: UPDATE stock = 10 - 3 = 7
-- Usuario B: UPDATE stock = 10 - 5 = 5  ← ¡Sobrescribió el cambio de A!
-- Resultado final: stock = 5 (pero se vendieron 8 unidades, debería ser 2)

-- CON transacción (Seguro):
BEGIN;
SELECT units_in_stock FROM products WHERE product_id = 1;
-- stock = 10
UPDATE products SET units_in_stock = units_in_stock - 3 WHERE product_id = 1;
-- stock = 7 (otro usuario ve 7, no 10)
COMMIT;
```

### 6.6 Buenas Prácticas

| Práctica | Por qué |
|----------|---------|
| **Transacciones cortas** | Mantener el menor tiempo posible para reducir contención entre usuarios |
| **Siempre COMMIT o ROLLBACK explícito** | No confiar en el cierre de conexión — puede dejar transacciones abiertas |
| **DML destructivo con EXPLAIN** | `BEGIN; EXPLAIN ANALYZE <DML>; ROLLBACK;` — ver impacto sin ejecutar |
| **Usar SAVEPOINT en imports masivos** | Permitir recuperación parcial si falla un registro |

```sql
-- Patrón seguro para DML destructivo en producción
BEGIN;
EXPLAIN ANALYZE
DELETE FROM orders WHERE order_date < '1990-01-01';
-- Veo que borraría 0 filas (la fecha más antigua es 1996)
ROLLBACK;  -- no ejecuto el DELETE, solo quería ver el impacto
```

## 7. Convenciones PostgreSQL que Usaremos

PostgreSQL tiene features que MySQL no tiene. Estas convenciones son las que usaremos en todo el laboratorio porque son **las buenas prácticas de la industria**.

### 7.1 RETURNING

**`RETURNING` devuelve los valores de la fila afectada por un INSERT, UPDATE o DELETE**, evitando un SELECT posterior. En MySQL esto no existe — necesitarías ejecutar dos queries.

| PostgreSQL (con RETURNING) | MySQL (sin RETURNING) |
|---------------------------|----------------------|
| 1 query: `INSERT ... RETURNING id` | 1er query: `INSERT ...` |
| Obtienes el ID inmediatamente | 2do query: `SELECT LAST_INSERT_ID()` |
| **1 roundtrip a la BD** | **2 roundtrips a la BD** |

```sql
-- PostgreSQL: eficiente con RETURNING
INSERT INTO customers (customer_id, company_name, city, country)
VALUES ('NEWCO', 'Nueva Corp', 'Lima', 'Peru')
RETURNING customer_id, company_name, city;
-- Obtienes el resultado inmediatamente

-- MySQL: necesitas dos queries
-- INSERT INTO customers (customer_id, company_name, ...) VALUES (...);
-- SELECT LAST_INSERT_ID();
```

> **Por qué importa en enterprise:** En aplicaciones web con alta concurrencia, cada roundtrip a la base de datos cuesta ~0.1-1ms. Si tu app ejecuta 1000 inserts por segundo, RETURNING te ahorra 1000 roundtrips = 100-1000ms de latencia total.

### 7.2 ON CONFLICT (UPSERT)

**`ON CONFLICT` ejecuta un INSERT o un UPDATE automáticamente** si hay un conflicto de clave (duplicado). Es el equivalente a "inserta esto, pero si ya existe, actualízalo".

| PostgreSQL (`ON CONFLICT`) | MySQL (`ON DUPLICATE KEY UPDATE`) |
|---------------------------|----------------------------------|
| `INSERT ... ON CONFLICT (col) DO UPDATE SET ...` | `INSERT ... ON DUPLICATE KEY UPDATE col = ...` |
| Puede especificar la columna de conflicto | Usa la PK implícitamente |
| Puede hacer `DO NOTHING` (ignorar conflicto) | No tiene equivalente directo |
| Más explícito y legible | Más conciso pero menos claro |

```sql
-- PostgreSQL: UPSERT limpio
INSERT INTO products (product_id, product_name, unit_price)
VALUES (100, 'Producto Nuevo', 15.50)
ON CONFLICT (product_id)
DO UPDATE SET
  product_name = EXCLUDED.product_name,
  unit_price = EXCLUDED.unit_price;
-- Si product_id 100 ya existe: actualiza nombre y precio
-- Si no existe: inserta normalmente

-- Ignorar conflicto silenciosamente
INSERT INTO products (product_id, product_name, unit_price)
VALUES (100, 'Producto Nuevo', 15.50)
ON CONFLICT (product_id) DO NOTHING;
```

> **`EXCLUDED`** es una tabla especial que contiene los valores que **iban a ser insertados**. Es como un "snapshot" del INSERT que falló.

### 7.3 ILIKE (case-insensitive LIKE)

**`ILIKE` es LIKE pero ignora mayúsculas/minúsculas.** En MySQL, LIKE ya es case-insensitive por defecto. En PostgreSQL, LIKE es case-sensitive.

| PostgreSQL | MySQL |
|-----------|-------|
| `WHERE name ILIKE '%food%'` | `WHERE name LIKE '%food%'` (ya es case-insensitive) |
| `WHERE LOWER(name) LIKE '%food%'` (alternativa) | `WHERE name LIKE '%food%'` |

```sql
-- PostgreSQL: case-insensitive
SELECT * FROM customers WHERE company_name ILIKE '%food%';
-- Encuentra "Food", "FOOD", "food", "FoOd", etc.

-- Si no tienes ILIKE (ej: en MySQL), necesitas:
SELECT * FROM customers WHERE LOWER(company_name) LIKE '%food%';
-- Más verboso, y puede no usar índices eficientemente
```

### 7.4 FILTER (agregación condicional)

**`FILTER (WHERE ...)` permite contar, sumar o promediar solo las filas que cumplen una condición**, sin necesidad de subqueries. MySQL no lo tiene.

| PostgreSQL (con FILTER) | MySQL (sin FILTER) |
|------------------------|-------------------|
| `COUNT(*) FILTER (WHERE cond)` | `SUM(CASE WHEN cond THEN 1 ELSE 0 END)` |
| Claro y directo | Verboso y propenso a errores |

```sql
-- PostgreSQL: elegante con FILTER
SELECT
  COUNT(*) AS total_pedidos,
  COUNT(*) FILTER (WHERE ship_country = 'USA') AS usa,
  COUNT(*) FILTER (WHERE ship_country = 'Germany') AS germany,
  COUNT(*) FILTER (WHERE ship_country = 'Brazil') AS brazil
FROM orders;

-- MySQL: necesitas CASE WHEN
SELECT
  COUNT(*) AS total_pedidos,
  SUM(CASE WHEN ship_country = 'USA' THEN 1 ELSE 0 END) AS usa,
  SUM(CASE WHEN ship_country = 'Germany' THEN 1 ELSE 0 END) AS germany,
  SUM(CASE WHEN ship_country = 'Brazil' THEN 1 ELSE 0 END) AS brazil
FROM orders;
```

> **En enterprise:** FILTER es esencial para dashboards donde necesitas múltiples métricas condicionales en una sola query. Una query con 10 métricas condicionales usando FILTER es 10x más clara que 10 subqueries separadas.

### 7.5 SERIAL / BIGSERIAL

**`SERIAL` crea una columna auto-incremental automáticamente.** Es el atajo de PostgreSQL para columnas de secuencia.

| PostgreSQL | MySQL | SQL Estándar |
|-----------|-------|-------------|
| `id SERIAL PRIMARY KEY` | `id INT AUTO_INCREMENT PRIMARY KEY` | `id INT GENERATED ALWAYS AS IDENTITY PRIMARY KEY` |
| Crea secuencia automáticamente | Crea secuencia automáticamente | Estándar SQL (PostgreSQL 10+) |

```sql
-- SERIAL: el atajo de PostgreSQL
CREATE TABLE mi_tabla (
  id SERIAL PRIMARY KEY,  -- crea secuencia + default automáticamente
  nombre TEXT
);

-- Equivalente exacto (lo que SERIAL crea internamente)
CREATE TABLE mi_tabla (
  id INTEGER PRIMARY KEY,
  nombre TEXT
);
-- PostgreSQL crea automáticamente:
-- SEQUENCE mi_tabla_id_seq
-- DEFAULT nextval('mi_tabla_id_seq')

-- SQL Estándar (recomendado para nuevos proyectos)
CREATE TABLE mi_tabla (
  id INTEGER GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  nombre TEXT
);
```

> **En enterprise:** `GENERATED ALWAYS AS IDENTITY` es preferible a `SERIAL` porque es estándar SQL y evita la creación implícita de secuencias que pueden causar problemas en réplicas y migraciones.

### 7.6 Orden de Ejecución Lógico de SQL

**El orden en que PostgreSQL ejecuta las cláusulas de un SELECT NO es el orden en que las escribes.** Este orden conceptual determina qué alias están disponibles en cada parte de la consulta.

```
FROM       → JOIN      → WHERE     → GROUP BY  → HAVING    →
WINDOW     → SELECT    → DISTINCT  → UNION     → ORDER BY  → LIMIT/OFFSET
```

| Orden | Cláusula | Qué puede usar | Qué no puede usar |
|-------|----------|----------------|-------------------|
| 1 | `FROM` | Tablas, subqueries en FROM | Alias de columnas |
| 2 | `JOIN` | Tablas unidas | Alias de otras tablas en JOIN |
| 3 | `WHERE` | Columnas de tablas, funciones | Alias de SELECT, agregaciones |
| 4 | `GROUP BY` | Columnas de tablas | Alias de SELECT |
| 5 | `HAVING` | Agregaciones, columnas de GROUP BY | Alias de SELECT |
| 6 | `SELECT` | Todo lo anterior | — |
| 7 | `DISTINCT` | Resultados de SELECT | — |
| 8 | `ORDER BY` | Todo lo anterior | — |

```sql
-- Ejemplo: ¿por qué no puedes usar alias de SELECT en WHERE?
SELECT
  product_name,
  unit_price * quantity AS total
FROM order_details
WHERE total > 100;  -- ❌ ERROR: "total" no existe aún en el orden de ejecución

-- Solución correcta: repetir la expresión o usar subquery
SELECT * FROM (
  SELECT
    product_name,
    unit_price * quantity AS total
  FROM order_details
) sub
WHERE total > 100;  -- ✅ Funciona porque el alias está en el FROM
```

### 7.7 DML Destructivo con EXPLAIN

**Nunca ejecutes un DELETE o UPDATE masivo sin verificar primero cuántas filas afectará.** El patrón seguro es usar EXPLAIN dentro de una transacción que puedes deshacer.

```sql
-- Patrón seguro: ver qué harías SIN ejecutar
BEGIN;
EXPLAIN (ANALYZE, BUFFERS)
DELETE FROM orders WHERE order_date < '1990-01-01';
-- Output: "Seq Scan on orders ... Rows Removed by Filter: 830"
-- PostgreSQL estima 0 filas (la fecha más antigua es 1996)
ROLLBACK;  -- no ejecuto nada, solo quería ver el impacto

-- Si el EXPLAIN muestra muchas filas que SÍ quieres borrar:
BEGIN;
DELETE FROM orders WHERE order_date < '1997-01-01';
-- Verificar cuántas filas se borraron
SELECT COUNT(*) FROM orders WHERE order_date < '1997-01-01';
-- Si es correcto:
COMMIT;
-- Si no es correcto:
ROLLBACK;
```

### 7.8 CTEs con MATERIALIZED Hint

**Desde PostgreSQL 12, puedes controlar si una CTE (Common Table Expression) se materializa en memoria o se inlines como subquery.** Esto es un optimizador de performance que MySQL no tiene.

| Opción | Qué hace | Cuándo usarlo |
|--------|----------|---------------|
| `AS MATERIALIZED` | Ejecuta la CTE una vez, almacena resultado en memoria | CTE usada múltiples veces, o subquery costosa |
| `AS NOT MATERIALIZED` | Inlinia como subquery, optimizador la reescribe | CTE simple usada una sola vez |

```sql
-- MATERIALIZED: ejecuta la CTE una vez y reutiliza el resultado
WITH ventas_por_producto AS MATERIALIZED (
  SELECT
    product_id,
    SUM(quantity * unit_price) AS total_ventas
  FROM order_details
  GROUP BY product_id
)
SELECT
  p.product_name,
  v.total_ventas
FROM ventas_por_producto v
JOIN products p ON v.product_id = p.product_id
WHERE v.total_ventas > 1000;

-- NOT MATERIALIZED: el optimizador puede reescribir la CTE como subquery
WITH productos_activos AS NOT MATERIALIZED (
  SELECT product_id, product_name
  FROM products
  WHERE discontinued = 0
)
SELECT * FROM productos_activos WHERE product_id < 20;
-- PostgreSQL puede "colapsar" la CTE y optimizarla junto con el WHERE
```

### 7.9 generate_series

**`generate_series()` genera una secuencia de filas** — es extremadamente útil para pruebas, generación de datos y análisis de series temporales. MySQL no tiene nada equivalente.

```sql
-- Generar números del 1 al 10
SELECT generate_series(1, 10) AS numero;

-- Generar fechas diarias
SELECT generate_series('2024-01-01'::date, '2024-01-10'::date, '1 day'::interval) AS fecha;

-- Caso de uso real: reporte diario sin datos faltantes
-- (cuando quieres ver 0 ventas en días sin actividad)
SELECT
  d::date AS dia,
  COUNT(o.order_id) AS pedidos
FROM generate_series('2024-01-01'::date, '2024-01-31'::date, '1 day'::interval) d
LEFT JOIN orders o ON o.order_date = d::date
GROUP BY d::date
ORDER BY d::date;
```

## 8. Glosario Rápido

| Término | Definición | Contexto de uso |
|---------|-----------|-----------------|
| **ACID** | Atomicidad, Consistencia, Aislamiento, Durabilidad — garantías de transacción | Cada transacción en PostgreSQL es ACID por defecto |
| **BRIN** | Block Range Index — índice para tablas非常大 con datos ordenados | Tablas de logs con timestamps, tablas de billones de filas |
| **B-tree** | Índice balanceado (el por defecto en PostgreSQL) | La mayoría de consultas WHERE, JOIN, ORDER BY |
| **CTE** | Common Table Expression — consulta nombrada con `WITH` | Queries complejas, recursión, legibilidad |
| **DDL** | Data Definition Language — CREATE, ALTER, DROP | Definir estructura de tablas |
| **DML** | Data Manipulation Language — INSERT, UPDATE, DELETE | Manipular datos |
| **DQL** | Data Query Language — SELECT | Consultar datos |
| **FK** | Foreign Key — columna que referencia otra tabla | Integridad referencial entre tablas |
| **FULL OUTER JOIN** | JOIN que incluye filas de ambas tablas, incluso sin coincidencia | MySQL no lo tiene, PostgreSQL sí |
| **generate_series** | Función que genera secuencias de números o fechas | Pruebas, reportes diarios, series temporales |
| **GiST** | Generalized Search Tree — índice para datos complejos | PostGIS (geoespacial),全文, rangos |
| **GIN** | Generalized Inverted Index — índice invertido | Arrays, JSONB, Full-Text Search |
| **Hash Join** | Algoritmo de JOIN: construye hash de una tabla, escanea otra | JOINs sin índice en la columna de unión |
| **ILIKE** | LIKE case-insensitive (PostgreSQL) | Búsquedas de texto ignorando mayúsculas |
| **Index Scan** | Búsqueda por índice (rápida) | Lo que buscas en consultas WHERE |
| **INSERT ... RETURNING** | Devuelve datos de la fila insertada | Evitar SELECT posterior en PostgreSQL |
| **Junction Table** | Tabla que resuelve una relación N:N | `order_details` entre orders y products |
| **Materialized View** | Vista almacenada físicamente (caché pre-computada) | Dashboards, reportes pesados |
| **Merge Join** | Algoritmo de JOIN: ordena ambas tablas y las fusiona | Datos preordenados o con índice |
| **MVCC** | Multi-Version Concurrency Control — lecturas sin bloqueo | PostgreSQL nunca bloquea lecturas |
| **Nested Loop** | Algoritmo de JOIN: por cada fila A, buscar en B | Conjuntos pequeños o con índice |
| **numeric** | Tipo de dato exacto para decimales (dinero) | SIEMPRE usar para precios, cantidades monetarias |
| **ON CONFLICT** | UPSERT: inserta o actualiza según conflicto de clave | Evitar duplicados al insertar |
| **PK** | Primary Key — identificador único de fila | Cada tabla debe tener una PK |
| **pg_dump** | Herramienta para backup de bases de datos | `pg_dump -U postgres northwind > backup.sql` |
| **pg_restore** | Herramienta para restaurar backups de pg_dump | `pg_restore -U postgres northwind < backup.sql` |
| **Seq Scan** | Sequential Scan — lectura completa de tabla | Normal en tablas pequeñas,Problemático en grandes |
| **Serializable** | Nivel de aislamiento más estricto | Transacciones que requieren integridad total |
| **SERIAL** | Tipo de dato auto-incremental en PostgreSQL | IDs de tablas (preferir `GENERATED AS IDENTITY`) |
| **shared_buffers** | Memoria compartida para caché de datos | PostgreSQL lee datos de RAM, no de disco |
| **VACUUM** | Limpia espacio muerto de filas eliminadas | PostgreSQL ejecuta autovacuum automáticamente |
| **WAL** | Write-Ahead Logging — registro de cambios antes de aplicarlos | Replicación, point-in-time recovery |
| **Window Function** | Función que opera sobre un grupo de filas sin agrupar | Ranking, acumulados, diferencias entre filas |
| **work_mem** | Memoria para operaciones de sorting/hashing | Si es bajo, PostgreSQL usa disco (lento) |
