## Ejercicio 1: Proyeccion de la Fuerza de Ventas

### Analogia
Imagina que tienes un formulario de empleado con 20 campos, pero solo necesitas copiar el nombre, apellido y cargo para un reporte. No vas a copiar todo el formulario, solo lo que necesitas.

### Definicion
`SELECT` especifica que columnas quieres ver. Evita `SELECT *` en produccion: proyectar solo las columnas necesarias reduce la carga de memoria y el trafico de red.

### Codigo

```sql
SELECT
    employee_id,
    last_name,
    first_name,
    title
FROM employees;
```

### Guia del Profesor
- **Que evalua:** Comprension de proyeccion y nomenclatura de columnas.
- **Tip de performance:** En tablas con 100+ columnas, `SELECT *` puede duplicar el tiempo de respuesta.
- **Error comun:** Creer que `SELECT *` es mas rapido (es mas lento porque trae todo).

## Ejercicio 2: Segmentacion de Productos Premium

### Analogia
Es como ir al supermercado y solo mirar los productos que cuestan mas de $20. No te interesan los baratos.

### Definicion
`WHERE` filtra filas antes de mostrar resultados. El filtro se aplica despues de identificar la tabla en `FROM`.

### Codigo

```sql
SELECT
    product_name,
    unit_price
FROM products
WHERE unit_price > 30;
```

### Guia del Profesor
- **Que evalua:** Uso de comparadores numericos (>).
- **Tip de performance:** PostgreSQL puede usar un indice en `unit_price` si existe.
- **Error comun:** Olvidar que `WHERE` se ejecuta antes que `SELECT`.

## Ejercicio 3: Ordenamiento de Pedidos por Fecha

### Analogia
Es como ordenar una pila de facturas de la mas reciente a la mas antigua para encontrar el ultimo pedido.

### Definicion
`ORDER BY` ordena los resultados. `ASC` (ascendente) es el defecto; `DESC` invierte el orden.

### Codigo

```sql
SELECT
    order_id,
    customer_id,
    order_date
FROM orders
ORDER BY order_date DESC;
```

### Guia del Profesor
- **Que evalua:** Uso de ORDER BY con DESC.
- **Tip de performance:** Ordenar es costoso en `work_mem`. En tablas grandes, limita con `LIMIT`.
- **Error comun:** Poner `DESC` en la columna equivocada.

## Ejercicio 4: Ranking de Envios Costosos (Manejo de Nulos)

### Analogia
Es como ordenar notas de la mas alta a la mas baja, pero las notas sin calificar quieres que queden al final, no al principio.

### Definicion
En PostgreSQL, los nulos se tratan como los valores mas "grandes". `NULLS LAST` fuerza que aparezcan al final en orden DESC.

### Codigo

```sql
SELECT
    order_id,
    ship_name,
    freight
FROM orders
ORDER BY freight DESC NULLS LAST;
```

### Guia del Profesor
- **Que evalua:** Manejo correcto de nulos en ordenamiento.
- **Tip de performance:** `NULLS LAST` no afecta el plan de ejecucion; es solo presentacion.
- **Error comun:** Olvidar `NULLS LAST` y ver nulos primero.

## Ejercicio 5: Top 5 Productos Mas Caros

### Analogia
Es como hacer un podio de los 5 mas caros: primero los ordenas de mayor a menor, y solo te quedas con los primeros 5.

### Definicion
Combinar `ORDER BY DESC` con `LIMIT N` obtiene los N valores mas altos.

### Codigo

```sql
SELECT
    product_name,
    unit_price
FROM products
ORDER BY unit_price DESC
LIMIT 5;
```

### Guia del Profesor
- **Que evalua:** Paginacion basica con LIMIT.
- **Tip de performance:** `LIMIT` es muy eficiente porque el optimizador deja de leer filas despues de encontrar N.
- **Error comun:** Usar `TOP 5` (syntax de SQL Server, no PostgreSQL).

## Ejercicio 6: Paginacion de Pedidos

### Analogia
Es como ver la pagina 3 de un catalogo: te saltas las primeras 20 paginas y muestras las siguientes 10.

### Definicion
`LIMIT` define cuantas filas traer; `OFFSET` define cuantas saltar. Usado en dashboards y APIs REST.

### Codigo

```sql
SELECT
    order_id,
    order_date,
    customer_id
FROM orders
ORDER BY order_date ASC
LIMIT 10 OFFSET 20;
```

### Guia del Profesor
- **Que evalua:** Paginacion con LIMIT y OFFSET.
- **Tip de performance:** `OFFSET` alto es lento en tablas grandes (lee y descarta filas). Alternativa: keyset pagination.
- **Error comun:** Olvidar que OFFSET empieza en 0.

## Ejercicio 7: Filtrado Geografico de Clientes

### Analogia
Es como pedir al sistema que solo te muestre clientes de "Latinoamerica", listando los paises que incluyes.

### Definicion
`IN (valor1, valor2, ...)` es equivalente a multiples `OR` pero mas limpio y a veces mas rapido.

### Codigo

```sql
SELECT
    customer_id,
    company_name,
    country
FROM customers
WHERE country IN ('Germany', 'UK', 'USA')
ORDER BY country, company_name;
```

### Guia del Profesor
- **Que evalua:** Uso de IN para filtrar por lista de valores.
- **Tip de performance:** PostgreSQL expande `IN` a una serie de `OR` internamente, pero es mas legible.
- **Error comun:** Usar `IN` con una sola columna (innecesario, mejor usar `=`).

## Ejercicio 8: Rango de Precios de Productos

### Analogia
Es como buscar productos en una tienda online con filtro de precio: "entre $15 y $75".

### Definicion
`BETWEEN a AND b` es equivalente a `>= a AND <= b`. Es inclusivo: incluye los limites.

### Codigo

```sql
SELECT
    product_id,
    product_name,
    unit_price
FROM products
WHERE unit_price BETWEEN 15 AND 75
ORDER BY unit_price;
```

### Guia del Profesor
- **Que evalua:** Rangos inclusivos con BETWEEN.
- **Tip de performance:** `BETWEEN` en columnas indexadas usa Index Range Scan.
- **Error comun:** Creer que BETWEEN es exclusivo (no lo es).

## Ejercicio 9: Pedidos Antiguos (Filtro de Fecha)

### Analogia
Es como buscar documentos anteriores a una fecha especifica para un archivado.

### Definicion
Para filtrar fechas, usa formato ISO 8601 (`'YYYY-MM-DD'`). Esto evita errores de interpretacion segun la configuracion regional.

### Codigo

```sql
SELECT
    order_id,
    customer_id,
    order_date
FROM orders
WHERE order_date < '1997-01-01'
ORDER BY order_date;
```

### Guia del Profesor
- **Que evalua:** Comparacion de fechas con formato ISO.
- **Tip de performance:** Si tienes un indice en `order_date`, el filtro es muy rapido.
- **Error comun:** Usar formato de fecha regional (ej. `'01/01/1997'`) que puede fallar.

## Ejercicio 10: Empleados con Mayor Antiguedad

### Analogia
Es como buscar los empleados que entraron antes de cierto ano para un plan de beneficios.

### Definicion
Combinamos concatenacion de strings (`||`), filtros de fecha, ordenamiento y limite. La precedencia es: `FROM` -> `WHERE` -> `ORDER BY` -> `LIMIT`.

### Codigo

```sql
SELECT
    employee_id,
    first_name || ' ' || last_name AS empleado,
    title,
    hire_date
FROM employees
WHERE hire_date >= '1993-01-01'
ORDER BY hire_date ASC
LIMIT 10;
```

### Guia del Profesor
- **Que evalua:** Concatenacion con `||` y aliases con `AS`.
- **Tip de performance:** `||` es nativo de PostgreSQL; en MySQL usa `CONCAT()`.
- **Error comun:** Usar `+` para concatenar (syntax de SQL Server).

## Ejercicio 11: Busqueda de Contactos Comerciales

### Analogia
Es como buscar en una agenda todos los contactos que tengan "Ventas" en su cargo.

### Definicion
`LIKE` con `%` busca patrones de texto. `%` significa "cualquier secuencia de caracteres". Si el comodin va al inicio, no puede usar indices B-Tree.

### Codigo

```sql
SELECT
    company_name,
    contact_name,
    contact_title
FROM customers
WHERE contact_title LIKE '%Sales%'
ORDER BY company_name;
```

### Guia del Profesor
- **Que evalua:** Busqueda de texto con LIKE.
- **Tip de performance:** `LIKE '%texto%'` fuerza Seq Scan. Para busquedas frecuentes, usa `pg_trgm`.
- **Error comun:** Olvidar los `%` y buscar coincidencia exacta.

## Ejercicio 12: Productos por Letra Inicial

### Analogia
Es como buscar en un diccionario solo las palabras que empiezan con "C".

### Definicion
`'C%'` (comodin solo al final) permite que PostgreSQL use Index Range Scan, mucho mas rapido que `%C%`.

### Codigo

```sql
SELECT
    product_id,
    product_name,
    unit_price
FROM products
WHERE product_name LIKE 'C%'
ORDER BY product_name;
```

### Guia del Profesor
- **Que evalua:** LIKE con comodin al final (index-friendly).
- **Tip de performance:** `'C%'` puede usar indice; `'%C%'` no puede.
- **Error comun:** No saber que la posicion del comodin afecta el rendimiento.

## Ejercicio 13: Busqueda Insensible a Mayusculas (ILIKE)

### Analogia
Es como buscar "manager" sin importar si lo escribieron "Manager", "MANAGER" o "manager".

### Definicion
`ILIKE` (case-insensitive LIKE) es extension PostgreSQL. Es ideal para busquedas donde el usuario no sabe como se escribio el dato.

### Codigo

```sql
SELECT
    employee_id,
    first_name || ' ' || last_name AS empleado,
    title
FROM employees
WHERE title ILIKE '%manager%';
```

### Guia del Profesor
- **Que evalua:** ILIKE para busquedas case-insensitive.
- **Tip de performance:** En MySQL, `LIKE` ya es case-insensitive (depende del collation).
- **Error comun:** Usar `LOWER(title) LIKE '%manager%'` (funcional pero mas verboso).

## Ejercicio 14: Clientes sin Region Asignada

### Analogia
Es como buscar en una base de datos los registros que tienen un campo vacio, como un telefono sin registrar.

### Definicion
Para buscar valores nulos, **nunca** uses `= NULL`. Debes usar `IS NULL`. Esto es fundamental: NULL no es igual a nada, ni siquiera a si mismo.

### Codigo

```sql
SELECT
    customer_id,
    company_name,
    city,
    country
FROM customers
WHERE region IS NULL;
```

### Guia del Profesor
- **Que evalua:** Manejo correcto de NULLs.
- **Tip de performance:** Los indices B-Tree generalmente excluyen NULLs.
- **Error comun:** `WHERE region = NULL` siempre retorna vacio.

## Ejercicio 15: Productos con Stock Disponible

### Analogia
Es como buscar en un almacenes los productos que tienen stock (no estan vacios).

### Definicion
`IS NOT NULL` filtra registros que tienen contenido. Es la contraparte de `IS NULL`.

### Codigo

```sql
SELECT
    product_name,
    units_in_stock
FROM products
WHERE units_in_stock IS NOT NULL
ORDER BY units_in_stock DESC;
```

### Guia del Profesor
- **Que evalua:** IS NOT NULL para filtrar registros con contenido.
- **Tip de performance:** En Northwind, `units_in_stock` siempre tiene valor, pero es buena practica.
- **Error comun:** Usar `<> NULL` (siempre retorna vacio).

## Ejercicio 16: Campana USA y Canada

### Analogia
Es como pedir al sistema que te muestre clientes de dos paises especificos.

### Definicion
`OR` permite combinar condiciones donde al menos una debe cumplirse. Si tienes muchos valores, es mejor usar `IN`.

### Codigo

```sql
SELECT
    customer_id,
    company_name,
    country,
    phone
FROM customers
WHERE country = 'USA'
   OR country = 'Canada'
ORDER BY country, company_name;
```

### Guia del Profesor
- **Que evalua:** Uso de OR para combinar condiciones.
- **Tip de performance:** `IN` es preferible a multiples `OR` (mas legible, mismo rendimiento).
- **Error comun:** Escribir `country = 'USA' OR 'Canada'` (syntax invalida).

## Ejercicio 17: Productos Activos (No Descontinuados)

### Analogia
Es como filtrar un catalogo para mostrar solo los productos que todavia se venden.

### Definicion
`<>` o `!=` significa "no es igual a". En Northwind, `discontinued` es integer: 0 = activo, 1 = descontinuado.

### Codigo

```sql
SELECT
    product_id,
    product_name,
    unit_price,
    units_in_stock
FROM products
WHERE discontinued <> 1
ORDER BY product_name;
```

### Guia del Profesor
- **Que evalua:** Desigualdad con `<>`.
- **Tip de performance:** Si `discontinued` tiene un indice, el filtro es muy selectivo.
- **Error comun:** Usar `discontinued = 0` (funciona, pero `<> 1` es mas explicito para "no descontinuado").

## Ejercicio 18: Productos de Alta Rotacion y Precio

### Analogia
Es como buscar productos que sean caros Y tengan mucho stock: los dos condiciones a la vez.

### Definicion
`AND` requiere que **ambas** condiciones se cumplan simultaneamente.

### Codigo

```sql
SELECT
    product_name,
    unit_price,
    units_in_stock
FROM products
WHERE unit_price > 20
  AND units_in_stock > 50
ORDER BY unit_price DESC;
```

### Guia del Profesor
- **Que evalua:** Combinacion de filtros con AND.
- **Tip de performance:** El optimizador evalua la condicion mas restrictiva primero.
- **Error comun:** Usar `OR` cuando se necesita `AND` (resultado muy diferente).

## Ejercicio 20: Empleados de los 90s (Fuera de USA)

### Analogia
Es como buscar empleados de una decada especifica que no sean de un pais determinado.

### Definicion
Combinamos `BETWEEN` para el rango de fechas y `<>` para excluir el pais. Practica la precedencia de operadores logicos.

### Codigo

```sql
SELECT
    employee_id,
    first_name || ' ' || last_name AS empleado,
    title,
    hire_date,
    country
FROM employees
WHERE hire_date BETWEEN '1990-01-01' AND '1999-12-31'
  AND country <> 'USA'
ORDER BY hire_date;
```

### Guia del Profesor
- **Que evalua:** Combinacion de BETWEEN y <>.
- **Tip de performance:** En tablas grandes, el orden de filtros importa (selectividad).
- **Error comun:** Usar `BETWEEN '1990' AND '1999'` (formato incompleto).

## Ejercicio 21: Paises de Clientes (Sin Repeticiones)

### Analogia
Es como sacar una lista de todos los paises donde tenemos clientes, sin repetir ninguno.

### Definicion
`DISTINCT` elimina filas duplicadas del resultado. Internamente, PostgreSQL usa un HashAggregate.

### Codigo

```sql
SELECT DISTINCT
    country
FROM customers
ORDER BY country;
```

### Guia del Profesor
- **Que evalua:** Eliminacion de duplicados con DISTINCT.
- **Tip de performance:** `DISTINCT` fuerza ordenamiento o hash; en tablas grandes es costoso.
- **Error comun:** Usar `DISTINCT` sin necesidad (ya no hay duplicados).

```sql
SELECT
```

## Ejercicio 24: Precio Promedio del Catalogo

### Analogia
Es como calcular el precio promedio de todos los productos de una tienda.

### Definicion
`AVG` calcula la media aritmética. `ROUND` redondea. `AVG` ignora valores NULL automaticamente.

### Codigo

```sql
SELECT
    ROUND(AVG(unit_price)::numeric, 2) AS precio_promedio
```

### Guia del Profesor
- **Que evalua:** AVG con ROUND y casteo a numeric.
- **Tip de performance:** `::numeric` es necesario para que ROUND funcione correctamente.

## Ejercicio 25: Rango de Precios (Minimo y Maximo)

### Analogia
Es como saber cual es el producto mas barato y el mas caro de un catalogo.

### Definicion
`MIN` y `MAX` son funciones muy eficientes. Si la columna esta indexada, PostgreSQL los encuentra casi instantaneamente.

### Codigo

```sql
SELECT
    MIN(unit_price) AS precio_minimo,
    MAX(unit_price) AS precio_maximo,
    ROUND((MAX(unit_price) - MIN(unit_price))::numeric, 2) AS rango
FROM products;
```

### Guia del Profesor
- **Que evalua:** MIN, MAX y expresion aritmetica en SELECT.
- **Tip de performance:** En indices B-Tree, MIN/MAX son operaciones O(log n).
- **Error comun:** Calcular el rango sin redondear (resultado con muchos decimales).

## Ejercicio 26: Ingresos Netos por Producto

### Analogia
Es como calcular cuanto dinero genero cada producto vendido, considerando descuentos.

### Definicion
Formula: `Cantidad * Precio * (1 - Descuento)`. El descuento es un decimal (0.15 = 15%).

### Codigo

```sql
SELECT
    product_id,
    SUM(quantity * unit_price * (1 - discount)) AS ingreso_total
FROM order_details
GROUP BY product_id
ORDER BY ingreso_total DESC;
```

### Guia del Profesor
- **Que evalua:** SUM con expresion aritmetica y GROUP BY.
- **Tip de performance:** Para 2155 filas, HashAggregate es eficiente.
- **Error comun:** Usar `1 - discount/100` (discount ya es decimal).

## Ejercicio 27: Pedidos con Flete Elevado (Uso de FILTER)

### Analogia
Es como contar todos los pedidos, pero tambien quiero saber cuantos son "caros" de envio, en una sola consulta.

### Definicion
`FILTER (WHERE ...)` es una joya de PostgreSQL. Permite conteos condicionales sin `CASE WHEN` complejos.

### Codigo

```sql
SELECT
    COUNT(*) AS total_pedidos,
    COUNT(*) FILTER (WHERE freight > 200) AS pedidos_flete_elevado
FROM orders;
```

### Guia del Profesor
- **Que evalua:** FILTER para agregacion condicional.
- **Tip de performance:** `FILTER` es mas legible que `CASE WHEN` y tiene el mismo plan.
- **Error comun:** Usar `WHERE` en lugar de `FILTER` (no funciona en COUNT).

## Ejercicio 28: Promedio de Flete por Ano

### Analogia
Es como calcular el promedio de gastos de envio por ano, pero en una sola fila para comparar facilmente.

### Definicion
Combinamos `EXTRACT(YEAR FROM ...)` para obtener el ano y `FILTER` para separar promedios por ano.

### Codigo

```sql
SELECT
    ROUND(AVG(freight) FILTER (WHERE EXTRACT(YEAR FROM order_date) = 1996)::numeric, 2) AS flete_promedio_1996,
    ROUND(AVG(freight) FILTER (WHERE EXTRACT(YEAR FROM order_date) = 1997)::numeric, 2) AS flete_promedio_1997,
    ROUND(AVG(freight) FILTER (WHERE EXTRACT(YEAR FROM order_date) = 1998)::numeric, 2) AS flete_promedio_1998
FROM orders;
```

### Guia del Profesor
- **Que evalua:** FILTER con EXTRACT para pivote basico.
- **Tip de performance:** PostgreSQL evalua el filtro fila por fila en una sola pasada.
- **Error comun:** Usar subconsultas separadas para cada ano (3 scans en vez de 1).

## Ejercicio 29: Territorios por Empleado

### Analogia
Es como contar cuantos territorios tiene asignado cada vendedor, y destacar los que tienen territorios en la costa este.

### Definicion
`GROUP BY` y `FILTER` simultaneamente permiten obtener metricas generales y segmentadas en un solo paso.

```sql

GROUP BY employee_id
ORDER BY total_territorios DESC;
```

### Guia del Profesor
- **Que evalua:** GROUP BY con FILTER y LIKE.
- **Tip de performance:** `LIKE '0%'` puede usar indice si existe.
- **Error comun:** No saber que FILTER es exclusivo de PostgreSQL.

## Ejercicio 30: Metricas de Precios por Categoria

### Analogia
Es como generar un reporte estadistico de precios por tipo de producto: total, cuantos son caros, y promedio.

### Definicion
Ejercicio de consolidacion: multiples funciones de agregacion con FILTER sobre un GROUP BY.

### Codigo

```sql
SELECT
    category_id,
    COUNT(*) AS total_productos,
    COUNT(*) FILTER (WHERE unit_price > 50) AS productos_premium,
    ROUND(AVG(unit_price)::numeric, 2) AS precio_promedio,
    ROUND(AVG(unit_price) FILTER (WHERE units_in_stock > 0)::numeric, 2) AS precio_promedio_stock_disponible
FROM products
GROUP BY category_id
ORDER BY category_id;
```

### Guia del Profesor
- **Que evalua:** Consolidacion de funciones de agregacion con FILTER.
- **Tip de performance:** En PostgreSQL 15+, FILTER es igual de rapido que CASE WHEN.
- **Error comun:** Mezclar `FILTER` en `WHERE` (solo va en funciones de agregacion).

## Ejercicio 31: Clientes por Ciudad y Pais (COUNT DISTINCT)

### Analogia
Es como saber cuantos clientes hay en cada ciudad, sin contar dos veces al mismo cliente si aparece en varios registros.

### Definicion
`COUNT(DISTINCT columna)` cuenta valores unicos. Sin DISTINCT, contaria todas las filas del grupo.

### Codigo

```sql
SELECT
    country,
    city,
    COUNT(DISTINCT customer_id) AS total_clientes
FROM customers
GROUP BY country, city
HAVING COUNT(DISTINCT customer_id) > 1
ORDER BY total_clientes DESC;
```

### Guia del Profesor
- **Que evalua:** COUNT con DISTINCT y GROUP BY multicolumna.
- **Tip de performance:** `COUNT(DISTINCT ...)` es mas costoso que `COUNT(*)`.
- **Error comun:** Olvidar DISTINCT y contar filas duplicadas.

## Ejercicio 32: Expresiones en SELECT (Calculos Derivados)

### Analogia
Es como calcular automaticamente el valor de un pedido: cantidad por precio unitario, sin modificar los datos originales.

### Definicion
Puedes usar expresiones aritmeticas directamente en SELECT. No modifican los datos; son calculos en tiempo de consulta.

### Codigo

```sql
SELECT
    product_name,
    unit_price,
    units_in_stock,
    unit_price * units_in_stock AS valor_inventario,
    ROUND((unit_price * 1.10)::numeric, 2) AS precio_con_10_pct_aumento
FROM products
WHERE discontinued = 0
ORDER BY valor_inventario DESC;
```

### Guia del Profesor
- **Que evalua:** Expresiones aritmeticas en SELECT.
- **Tip de performance:** Las expresiones se evaluan en CPU, no en disco.
- **Error comun:** Creer que la columna calculada se almacena (no, es temporal).

## Ejercicio 33: Aliases Descriptivos

### Analogia
Es como ponerle nombre bonito a las columnas de un reporte para que el gerente entienda que significa cada numero.

### Definicion
`AS alias` renombra columnas temporalmente. Es obligatorio para columnas calculadas y recomendado siempre para legibilidad.

### Codigo

```sql
SELECT
    product_name AS "Nombre del Producto",
    unit_price AS "Precio Unitario ($)",
    units_in_stock AS "Unidades en Stock",
    category_id AS "ID Categoria"
```

## Ejercicio 34: Manejo de Nulos con COALESCE

## Ejercicio 41: Funcion AGE y Calculo de Antiguedad

### Analogia
Es como calcular cuantos anos lleva un empleado en la empresa automaticamente.

### Definicion
`AGE(fecha)` retorna la diferencia entre la fecha actual y la fecha proporcionada como intervalo.

### Codigo

```sql
SELECT
    first_name || ' ' || last_name AS empleado,
    hire_date,
    AGE(CURRENT_DATE, hire_date) AS antiguedad_intervalo,
    EXTRACT(YEAR FROM AGE(hire_date)) AS anos_en_empresa,
    CASE
        WHEN EXTRACT(YEAR FROM AGE(hire_date)) >= 10 THEN 'Veterano'
        WHEN EXTRACT(YEAR FROM AGE(hire_date)) >= 5 THEN 'Experimentado'
        ELSE 'Nuevo'
    END AS categoria
FROM employees
ORDER BY hire_date;
```

### Guia del Profesor
- **Que evalua:** AGE() y EXTRACT para calculo de antiguedad.
- **Tip de performance:** `AGE()` es immutable; PostgreSQL puede cachear resultados.
- **Error comun:** Calcular antiguedad como `EXTRACT(YEAR FROM CURRENT_DATE) - EXTRACT(YEAR FROM hire_date)` (ignora mes/dia).

## Ejercicio 45: CASE WHEN en GROUP BY - Pivote Basico

### Analogia
Es como separar productos en grupos por precio y contar cuantos hay en cada grupo, todo en una sola tabla.

### Definicion
`CASE` dentro de `SUM` permite crear columnas condicionales. Es el patron basico de pivoteo (filas a columnas).

### Codigo

```sql
SELECT
    category_id,
    COUNT(*) AS total_productos,
    SUM(CASE WHEN unit_price < 20 THEN 1 ELSE 0 END) AS economicos,
    SUM(CASE WHEN unit_price >= 20 AND unit_price < 50 THEN 1 ELSE 0 END) AS intermedios,
    SUM(CASE WHEN unit_price >= 50 THEN 1 ELSE 0 END) AS premium
FROM products
WHERE discontinued = 0
GROUP BY category_id
ORDER BY category_id;
```

### Guia del Profesor
- **Que evalua:** CASE con SUM para conteo condicional.
- **Tip de performance:** Es mas eficiente que hacer 3 consultas separadas.
- **Error comun:** Usar `COUNT(CASE ...)` que cuenta NULLs tambien (usar `SUM` con 1/0).

## Ejercicio 46: CASE WHEN con BETWEEN - Rangos de Fechas

### Analogia
Es como clasificar pedidos por trimestre automaticamente.

### Definicion
`CASE` con `BETWEEN` permite clasificar registros en rangos de fechas.

### Codigo

```sql
SELECT
    order_id,
    order_date,
    EXTRACT(QUARTER FROM order_date) AS trimestre,
    CASE
        WHEN EXTRACT(MONTH FROM order_date) BETWEEN 1 AND 3 THEN 'Enero-Marzo'
        WHEN EXTRACT(MONTH FROM order_date) BETWEEN 4 AND 6 THEN 'Abril-Junio'
        WHEN EXTRACT(MONTH FROM order_date) BETWEEN 7 AND 9 THEN 'Julio-Septiembre'
        ELSE 'Octubre-Diciembre'
    END AS periodo,
    freight
FROM orders
WHERE EXTRACT(YEAR FROM order_date) = 1997
ORDER BY order_date;
```

### Guia del Profesor
- **Que evalua:** CASE con EXTRACT y BETWEEN.
- **Tip de performance:** La expresion se evalua por cada fila; en 830 pedidos es instantaneo.
- **Error comun:** Usar `BETWEEN 1 AND 3` para meses sin saber que enero es 1.

## Ejercicio 47: CASE WHEN en ORDER BY - Ordenamiento Personalizado

### Analogia
Es como decir "primero muéstame los pedidos urgentes, luego los normales, y al final los lentos".

### Definicion
`CASE` dentro de `ORDER BY` crea un orden personalizado que no sigue el orden alfabetico ni numerico.

### Codigo

```sql
SELECT
    order_id,
    customer_id,
    order_date,
    shipped_date,
    CASE
        WHEN shipped_date IS NULL THEN 1
        WHEN shipped_date <= required_date THEN 2
        ELSE 3
    END AS prioridad_envio,
    CASE
        WHEN shipped_date IS NULL THEN 'Pendiente'
        WHEN shipped_date <= required_date THEN 'A tiempo'
        ELSE 'Con retraso'
    END AS estado_envio
FROM orders
WHERE EXTRACT(YEAR FROM order_date) = 1997
ORDER BY
    CASE
        WHEN shipped_date IS NULL THEN 1
        WHEN shipped_date <= required_date THEN 2
        ELSE 3
    END,
    order_date;
```

### Guia del Profesor
- **Que evalua:** CASE en ORDER BY para ordenamiento personalizado.
- **Tip de performance:** La expresion se evalua antes del sort y se almacena como clave.
- **Error comun:** Repetir el CASE en SELECT y ORDER BY (se evalua dos veces).

## Ejercicio 49: Ejercicio Integrador - Catalogo de Productos

### Analogia
Es como generar un reporte completo de productos con toda la informacion que un gerente necesita ver.

```sql
SELECT
    p.product_id,

    p.unit_price AS precio_actual,
```

## Ejercicio 50: Ejercicio Integrador - Reporte de Pedidos

### Analogia
Es como generar el reporte mensual de ventas para el gerente general: todo en una sola vista.

### Definicion
Combina: SELECT con alias, WHERE con multiples filtros, ORDER BY, LIMIT, expresiones aritmeticas, COALESCE y CASE.

### Codigo

```sql
SELECT
    o.order_id,
    o.order_date,
    c.company_name AS cliente,
    COALESCE(o.ship_country, 'No especificado') AS pais_destino,
    o.freight AS flete,
    ROUND(o.freight * 1.19, 2) AS flete_con_igv,
    CASE
        WHEN o.freight > 500 THEN 'Alto'
        WHEN o.freight > 100 THEN 'Medio'
        ELSE 'Bajo'
    END AS categoria_flete,
    CASE
        WHEN o.shipped_date IS NULL THEN 'Pendiente'
        WHEN o.shipped_date <= o.required_date THEN 'A tiempo'
        ELSE 'Retrasado'
    END AS estado_envio
FROM orders o
INNER JOIN customers c ON o.customer_id = c.customer_id
WHERE o.order_date >= '1997-01-01'
  AND o.order_date < '1998-01-01'
ORDER BY o.freight DESC
LIMIT 25;
```

### Guia del Profesor
- **Que evalua:** Integracion final de todos los conceptos del nivel basico.
- **Tip de performance:** Este ejercicio usa INNER JOIN (introductorio); en nivel intermedio se profundiza.
- **Error comun:** No manejar NULLs en `ship_country` con COALESCE.
