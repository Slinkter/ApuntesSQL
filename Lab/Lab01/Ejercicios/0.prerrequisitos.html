1. Conceptos Base

1.1 ¿Qué es SQL?

SQL (Structured Query Language) es el lenguaje estándar para comunicarse con bases de datos relacionales. Con SQL puedes:

  • Consultar datos (SELECT)
  • Crear y modificar estructuras (CREATE TABLE, ALTER TABLE)
  • Insertar, actualizar y eliminar datos (INSERT, UPDATE, DELETE)
  • Controlar acceso (GRANT, REVOKE)

1.2 ¿Qué es PostgreSQL?

PostgreSQL es un sistema de gestión de bases de datos relacional (RDBMS) de código abierto, conocido por:

  • Cumplimiento ACID (Atomicidad, Consistencia, Aislamiento, Durabilidad)
  • Soporte de estándares SQL avanzados (ventanas, CTEs recursivas, JSONB)
  • MVCC (Control de Concurrencia Multiversión) — las lecturas nunca bloquean escrituras
  • Extensibilidad — puedes crear tipos de datos, operadores, funciones e índices personalizados

1.3 Modelo Relacional

Los datos se organizan en tablas (relaciones). Cada tabla tiene:

  • Filas (tuplas o registros) — cada fila representa una entidad
  • Columnas (atributos) — cada columna representa una propiedad
  • Clave Primaria (PK) — columna o combinación que identifica de forma única cada fila
  • Clave Foránea (FK) — columna que referencia la PK de otra tabla, creando una relación
Ejemplo: orders (pedidos) → customer_id → customers (clientes)
  orders.customer_id = customers.customer_id
  ┌─────────────────┐         ┌──────────────────┐
  │    orders       │         │   customers       │
  ├─────────────────┤         ├──────────────────┤
  │ order_id (PK)   │         │ customer_id (PK) │
  │ customer_id (FK)│────────→│ company_name     │
  │ order_date      │         │ country          │
  │ ...             │         │ ...              │
  └─────────────────┘         └──────────────────┘

1.4 Índices

Un índice es una estructura auxiliar que acelera la búsqueda de filas. Es como el índice alfabético de un libro:

  • Sin índice: PostgreSQL lee toda la tabla secuencialmente (Seq Scan)
  • Con índice: salta directamente a las filas relevantes (Index Scan)
  • PostgreSQL soporta: B-tree (default), Hash, GiST, GIN, BRIN

1.5 SET search_path

SQL
SET search_path TO public;

Por defecto, PostgreSQL busca tablas en public. Siempre estamos en public en este laboratorio.

2. Setup del Laboratorio (Docker)

2.1 Requisitos de Software

  • Docker Desktop (Windows/Mac) o Docker Engine (Linux) — v24+
  • Al menos 4 GB de RAM libre para el contenedor
  • Opcional: psql cliente instalado localmente

2.2 Método Rápido: Docker Standalone

POWERSHELL
# PowerShell (Windows)
docker run --name pg_northwind_lab ^
  -e POSTGRES_USER=postgres ^
  -e POSTGRES_PASSWORD=postgres ^
  -e POSTGRES_DB=northwind ^
  -v "${PWD}/db_northwind.sql:/docker-entrypoint-initdb.d/db_northwind.sql" ^
  -p 5432:5432 ^
  -d postgres:16-alpine
BASH
# Bash (Linux/Mac)
docker run --name pg_northwind_lab \
  -e POSTGRES_USER=postgres \
  -e POSTGRES_PASSWORD=postgres \
  -e POSTGRES_DB=northwind \
  -v "$(pwd)/../db_northwind.sql:/docker-entrypoint-initdb.d/db_northwind.sql" \
  -p 5432:5432 \
  -d postgres:16-alpine

2.3 ¿Qué hace `docker-entrypoint-initdb.d/`?

Es una carpeta especial dentro de la imagen oficial de PostgreSQL. Cuando el contenedor arranca por primera vez y la base de datos no existe, PostgreSQL ejecuta automáticamente todos los archivos .sql o .sh que encuentre ahí, en orden alfabético.

Esto significa que no necesitas ejecutar nada manualmente — el script db_northwind.sql se ejecuta solo.

2.4 Verificar que la Base Está Lista

BASH
# Ver logs
docker logs pg_northwind_lab

# Buscar la línea de éxito
docker logs pg_northwind_lab 2>&1 | grep "ready for start up"

2.5 Docker Compose (Alternativa)

YAML
# docker-compose.yml
services:
  db:
    image: postgres:16-alpine
    container_name: pg_northwind_lab
    restart: always
    environment:
      POSTGRES_USER: postgres
      POSTGRES_PASSWORD: postgres
      POSTGRES_DB: northwind
    ports:
      - "5432:5432"
    volumes:
      - pg_data:/var/lib/postgresql/data
      - ./db_northwind.sql:/docker-entrypoint-initdb.d/db_northwind.sql

volumes:
  pg_data:
BASH
docker compose up -d

1. Instalar extensión PostgreSQL de Microsoft

2. Agregar conexión:

  • Server: localhost

2.7 Troubleshooting Rápido

| Problema | Solución |

|----------|----------|

| Puerto 5432 ocupado | Cambia -p 5432:5432 a -p 5433:5432 |

4. Esquema Northwind

Northwind es una base de datos clásica de una empresa de distribución de alimentos. Contiene 14 tablas que modelan:

  • Clientes (customers) — empresas que compran productos
  • Productos (products) — catálogo de alimentos
  • Categorías (categories) — clasificación de productos
  • Proveedores (suppliers) — quién fabrica los productos
  • Pedidos (orders) — transacciones de compra
  • Detalles de pedido (order_details) — ítems individuales de cada pedido
  • Empleados (employees) — fuerza de ventas
  • Territorios (territories, region) — zona geográfica de empleados
  • Transportistas (shippers) — empresas de envío
  • Us_states — lista de estados de EE.UU.

5. Cómo Leer EXPLAIN (ANALYZE, BUFFERS)

EXPLAIN muestra el plan de ejecución que PostgreSQL usará para resolver una consulta. Con ANALYZE lo ejecuta y muestra tiempos reales. Con BUFFERS muestra E/S.

Sintaxis

SQL
EXPLAIN (ANALYZE, BUFFERS) SELECT ... ;

Ejemplo

SQL
EXPLAIN (ANALYZE, BUFFERS)
SELECT c.company_name, COUNT(o.order_id) AS pedidos
FROM customers c
LEFT JOIN orders o ON c.customer_id = o.customer_id
WHERE c.country = 'Mexico'
GROUP BY c.company_name;

Output:

 HashAggregate  (cost=... rows=5 width=...) (actual time=0.45..0.47 rows=5 loops=1)
   Group Key: c.company_name
   Buffers: shared hit=16
   ->  Hash Right Join  (cost=... rows=... width=...)
         Hash Cond: (o.customer_id = c.customer_id)
         Buffers: shared hit=16
         ->  Seq Scan on orders o  (cost=... rows=... width=...) (actual time=0.01..0.18 rows=48 loops=1)
               Buffers: shared hit=10
         ->  Hash  (cost=... rows=... width=...)
               ->  Seq Scan on customers c  (cost=... rows=... width=...)
                     Filter: (country = 'Mexico'::text)
                     Buffers: shared hit=6
 Planning Time: 0.15 ms
 Execution Time: 0.55 ms

Interpretación

| Elemento | Significa |

|----------|-----------|

| Seq Scan | Escaneo secuencial (lee toda la tabla) — lento en tablas grandes |

| Index Scan | Escaneo por índice (salta a filas específicas) — rápido |

| Hash Join | Construye hash de una tabla y escanea la otra — óptimo para JOINs sin índice |

| Nested Loop | Para cada fila de A, busca en B — óptimo para conjuntos pequeños |

| Merge Join | Ordena ambas y las fusiona — óptimo para datos preordenados |

| cost=... | Costo estimado (unidad arbitraria, comparativo) |

| actual time=0.45..0.47 | Tiempo real en ms (inicio..fin) |

| rows=5 | Filas reales producidas |

| loops=1 | Veces que se ejecutó este nodo |

| Buffers: shared hit=16 | 16 bloques leídos desde caché (rápido) |

| Buffers: shared read=10 | 10 bloques leídos desde disco (lento) |

| Planning Time | Tiempo que el optimizador planificó |

| Execution Time | Tiempo total de ejecución |

Regla de Oro

  • Seq Scan en tabla pequeña (< 1000 filas) → normal
  • Seq Scan en tabla grande (> 10000 filas) → probablemente falta un índice
  • Buffers: shared read → lecturas de disco, posible bottleneck
  • Diferencia entre rows estimado y filas reales → estadísticas desactualizadas (correr ANALYZE)

6. Transacciones: BEGIN / COMMIT / ROLLBACK

Una transacción agrupa operaciones en una sola unidad atómica: o se ejecutan todas, o no se ejecuta ninguna.

Flujo Básico

SQL
-- Iniciar transacción
BEGIN;

INSERT INTO orders (order_id, customer_id, employee_id, order_date)
VALUES (99999, 'ALFKI', 1, CURRENT_DATE);

UPDATE customers
SET contact_name = 'Nuevo Contacto'
WHERE customer_id = 'ALFKI';

-- Confirmar cambios
COMMIT;

Rollback (deshacer)

SQL
BEGIN;
DELETE FROM orders WHERE order_id = 99999;
-- Ups, no quería borrar
ROLLBACK;  -- todo vuelve al estado anterior

Transacciones Anidadas con SAVEPOINT

SQL
BEGIN;
  INSERT INTO customers ...;
  SAVEPOINT sp1;
  INSERT INTO orders ...;     -- esto falla
  ROLLBACK TO SAVEPOINT sp1;  -- deshace solo el INSERT orders
  -- customers insertado se conserva
COMMIT;

Buenas Prácticas

  • Transacciones cortas — mantener el menor tiempo posible para reducir contención
  • Siempre hacer COMMIT o ROLLBACK explícito (no confiar en el cierre de conexión)
  • Para pruebas de DML destructivo: BEGIN; EXPLAIN ANALYZE <DML>; ROLLBACK;

7. Convenciones PostgreSQL que Usaremos

7.1 RETURNING

Devuelve valores de la fila afectada, evitando un SELECT posterior:

SQL
INSERT INTO customers (customer_id, company_name, city, country)
VALUES ('NEWCO', 'Nueva Corp', 'Lima', 'Peru')
RETURNING customer_id, company_name, city;

7.2 ON CONFLICT (UPSERT)

Inserta o actualiza si existe conflicto:

SQL
INSERT INTO products (product_id, product_name, unit_price)
VALUES (100, 'Producto Nuevo', 15.50)
ON CONFLICT (product_id)
DO UPDATE SET 
  product_name = EXCLUDED.product_name,
  unit_price = EXCLUDED.unit_price;

7.3 ILIKE (case-insensitive LIKE)

SQL
SELECT * FROM customers WHERE company_name ILIKE '%food%';
-- Encuentra "Food", "FOOD", "food", etc.

7.4 FILTER (agregación condicional)

SQL
SELECT
  COUNT(*) AS total_pedidos,
  COUNT(*) FILTER (WHERE ship_country = 'USA') AS usa,
  COUNT(*) FILTER (WHERE ship_country = 'Germany') AS germany
FROM orders;

7.5 SERIAL / BIGSERIAL

Auto-incrementales (equivalentes a IDENTITY en SQL estándar):

SQL
CREATE TABLE mi_tabla (
  id SERIAL PRIMARY KEY,  -- equivalente a INTEGER GENERATED BY DEFAULT AS IDENTITY
  nombre TEXT
);

7.6 Orden de Ejecución Lógica de SQL

Este orden conceptual (no físico) determina qué alias están disponibles dónde:

FROM       → JOIN      → WHERE     → GROUP BY  → HAVING    → 
WINDOW     → SELECT    → DISTINCT  → UNION     → ORDER BY  → LIMIT/OFFSET

7.7 DML Destructivo con EXPLAIN

SQL
BEGIN;
EXPLAIN (ANALYZE, BUFFERS)
DELETE FROM orders WHERE order_date < '1990-01-01';
ROLLBACK;

7.8 CTEs con MATERIALIZED Hint

Desde PostgreSQL 12, puedes controlar si una CTE se materializa o se inlining:

SQL
WITH cte AS MATERIALIZED (
  SELECT ...  -- se ejecuta una vez, se almacena en memoria
)
SELECT * FROM cte;

WITH cte AS NOT MATERIALIZED (
  SELECT ...  -- se inlined como subquery, puede ser más rápido
)
SELECT * FROM cte;

7.9 generate_series

Función de PostgreSQL para generar secuencias (útil para pruebas):

SQL
SELECT generate_series(1, 10);

SELECT generate_series('2024-01-01'::date, '2024-01-10'::date, '1 day'::interval);

8. Glosario Rápido

| Término | Definición |

|---------|------------|

| ACID | Atomicidad, Consistencia, Aislamiento, Durabilidad — garantías de transacción |

| CTE | Common Table Expression — consulta nombrada con WITH |

| DDL | Data Definition Language — CREATE, ALTER, DROP |

| DML | Data Manipulation Language — INSERT, UPDATE, DELETE |

| FK | Foreign Key — columna que referencia otra tabla |

| Index Scan | Búsqueda por índice (rápida) |

| MVCC | Multi-Version Concurrency Control — lecturas sin bloqueo |

| Nested Loop | Algoritmo de JOIN: por cada fila A, buscar en B |

| Hash Join | Algoritmo de JOIN: construir hash de una tabla, escanear otra |

| PK | Primary Key — identificador único de fila |

| Seq Scan | Sequential Scan — lectura completa de tabla |

| Window Function | Función que opera sobre un grupo de filas relacionadas |

| work_mem | Memoria para operaciones de sorting/hashing (por operación, no global) |

| shared_buffers | Memoria compartida para cache de datos en PostgreSQL |

| RETURNING | Devuelve datos de la fila afectada por DML |

| ON CONFLICT | UPSERT: inserta o actualiza según conflicto |

| ILIKE | LIKE case-insensitive |

| FILTER (WHERE ...) | Agregación condicional sin subquery |

| generate_series | Generador de filas secuenciales |
