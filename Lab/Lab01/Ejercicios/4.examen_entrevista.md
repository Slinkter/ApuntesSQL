## Seccion 1: Filtro y Proyeccion (4 preguntas)

### Pregunta 1 - Nivel: Facil

**Contexto:** El gerente de ventas necesita una lista de clientes activos en Europa para una campana de marketing. Solo le importan empresas con nombre, contacto y pais.

**Enunciado:** Obtener el nombre de la empresa, nombre del contacto, titulo del contacto y pais de todos los clientes que se encuentren en paises europeos (UK, Germany, France, Spain, Italy). Ordena por pais y luego por nombre de empresa.

<details>
<summary>Solucion</summary>

```sql
SELECT
    company_name,
    contact_name,
    contact_title,
    country
FROM customers
WHERE country IN ('UK', 'Germany', 'France', 'Spain', 'Italy')
ORDER BY country, company_name;
```

**Explicacion:**
- `WHERE country IN (...)` es mas limpio y eficiente que multiples `OR`.
- `ORDER BY country, company_name` permite agrupar visualmente por pais.
- **Error comun:** Usar `=` con cada pais separado por `OR` (mas verboso, mismo resultado).

</details>

### Pregunta 2 - Nivel: Facil

**Contexto:** Un analista necesita identificar productos premium que estan disponibles para venta. Los productos descontinuados no deben aparecer.

**Enunciado:** Listar el nombre del producto, precio unitario y unidades en stock de todos los productos con precio mayor a $20 que NO esten descontinuados (`discontinued = 0`). Ordena por precio descendente.

<details>
<summary>Solucion</summary>

```sql
SELECT
    product_name,
    unit_price,
    units_in_stock
FROM products
WHERE unit_price > 20
  AND discontinued = 0
ORDER BY unit_price DESC;
```

**Explicacion:**
- `AND` requiere que ambas condiciones se cumplan.
- `discontinued` es integer (no boolean) en Northwind: `0` = activo, `1` = descontinuado.
- **Error comun:** Usar `discontinued <> 1` en lugar de `discontinued = 0` (funciona igual, pero es menos explicito).

</details>

### Pregunta 3 - Nivel: Facil

**Contexto:** El equipo de logistica necesita revisar pedidos con flete alto para negociar mejores tarifas con transportistas.

**Enunciado:** Mostrar el ID del pedido, nombre del barco (ship_name), flete y pais de destino de los 10 pedidos con mayor flete. Los valores nulos de flete deben quedar al final.

<details>
<summary>Solucion</summary>

```sql
SELECT
    order_id,
    ship_name,
    freight,
    ship_country
FROM orders
ORDER BY freight DESC NULLS LAST
LIMIT 10;
```

**Explicacion:**
- `NULLS LAST` evita que nulos aparezcan primero en orden descendente.
- `LIMIT 10` recorta el resultado a las 10 filas mas costosas.
- **Error comun:** Olvidar `NULLS LAST` y obtener nulos primero en el ranking.

</details>

### Pregunta 4 - Nivel: Facil

**Contexto:** RRHH necesita un directorio de empleados con su ubicacion y jerarquia de reporte.

**Enunciado:** Listar el nombre completo del empleado (nombre y apellido concatenados), titulo, ciudad y el ID del empleado al que reporta. Solo mostrar empleados que reportan directamente a alguien (no incluir al presidente).

<details>
<summary>Solucion</summary>

```sql
SELECT
    first_name || ' ' || last_name AS nombre_completo,
    title,
    city,
    reports_to
FROM employees
WHERE reports_to IS NOT NULL
ORDER BY reports_to, last_name;
```

**Explicacion:**
- `IS NOT NULL` excluye al empleado sin jefe (Andrew Fuller, el presidente).
- Concatenacion con `||` es la forma estandar de PostgreSQL.
- **Error comun:** Usar `reports_to != NULL` (siempre retorna vacio; NULL no se compara con `=` o `!=`).

</details>

## Seccion 2: Agregacion (4 preguntas)

### Pregunta 5 - Nivel: Medio

**Contexto:** El equipo de compras necesita saber que categorias de productos generan mas ingresos para decidir que proveedores contactar.

**Enunciado:** Calcular el ingreso total por categoria (considerando cantidad * precio * (1 - descuento)). Mostrar solo las categorias con ingreso total mayor a $10,000. Ordenar de mayor a menor ingreso.

<details>
<summary>Solucion</summary>

```sql
SELECT
    p.category_id,
    SUM(od.quantity * od.unit_price * (1 - od.discount)) AS ingreso_total
FROM order_details od
INNER JOIN products p ON od.product_id = p.product_id
GROUP BY p.category_id
HAVING SUM(od.quantity * od.unit_price * (1 - od.discount)) > 10000
ORDER BY ingreso_total DESC;
```

**Explicacion:**
- `HAVING` filtra despues de la agregacion, `WHERE` filtra antes.
- `SUM(quantity * unit_price * (1 - discount))` calcula ingreso neto.
- **Error comun:** Usar `WHERE SUM(...) > 10000` (error de sintaxis: agregaciones no van en WHERE).

</details>

### Pregunta 6 - Nivel: Medio

**Contexto:** Un analista de negocio quiere saber como se distribuyen los pedidos por transportista, pero solo para transportistas que manejan pedidos con flete promedio alto.

**Enunciado:** Para cada transportista, mostrar el nombre de la compania, el total de pedidos y el flete promedio. Solo incluir transportistas cuyo flete promedio sea mayor a $50.

<details>
<summary>Solucion</summary>

```sql
SELECT
    s.company_name AS transportista,
    COUNT(o.order_id) AS total_pedidos,
    ROUND(AVG(o.freight)::numeric, 2) AS flete_promedio
FROM orders o
INNER JOIN shippers s ON o.ship_via = s.shipper_id
GROUP BY s.company_name
HAVING AVG(o.freight) > 50
ORDER BY flete_promedio DESC;
```

**Explicacion:**
- JOIN entre `orders` y `shippers` para obtener nombre del transportista.
- `HAVING AVG(freight) > 50` filtra grupos post-agregacion.
- `::numeric` es necesario para que `ROUND` funcione correctamente.
- **Error comun:** Usar alias `flete_promedio` en `HAVING` (en PostgreSQL 15+ funciona, pero no es portable).

</details>

### Pregunta 7 - Nivel: Medio

**Contexto:** El equipo de inventory necesita un reporte de productos que requieren reabastecimiento urgente.

**Enunciado:** Calcular por cada categoria: total de productos, productos con stock bajo (menos de 20 unidades), precio promedio y el producto mas caro de la categoria. Mostrar solo categorias con al menos 3 productos.

<details>
<summary>Solucion</summary>

```sql
SELECT
    c.category_name,
    COUNT(p.product_id) AS total_productos,
    COUNT(*) FILTER (WHERE p.units_in_stock < 20) AS stock_bajo,
    ROUND(AVG(p.unit_price)::numeric, 2) AS precio_promedio,
    MAX(p.unit_price) AS precio_maximo
FROM products p
INNER JOIN categories c ON p.category_id = c.category_id
GROUP BY c.category_name
HAVING COUNT(p.product_id) >= 3
ORDER BY total_productos DESC;
```

**Explicacion:**
- `COUNT(*) FILTER (WHERE ...)` es extension PostgreSQL mas eficiente que `CASE WHEN`.
- JOIN con `categories` para obtener nombre descriptivo.
- **Error comun:** Olvidar incluir `category_name` en `GROUP BY` cuando esta en `SELECT`.

</details>

### Pregunta 8 - Nivel: Medio

**Contexto:** Un gerente quiere entender la estacionalidad de ventas por trimestre para planificar inventario.

**Enunciado:** Calcular el ingreso total por trimestre del anio 1997. Mostrar el trimestre (Q1, Q2, Q3, Q4), el ingreso total y el porcentaje que representa cada trimestre sobre el total anual.

<details>
<summary>Solucion</summary>

```sql
WITH ingresos_trimestre AS (
    SELECT
        EXTRACT(QUARTER FROM o.order_date) AS trimestre,
        SUM(od.quantity * od.unit_price * (1 - od.discount)) AS ingreso
    FROM orders o
    INNER JOIN order_details od ON o.order_id = od.order_id
    WHERE EXTRACT(YEAR FROM o.order_date) = 1997
    GROUP BY EXTRACT(QUARTER FROM o.order_date)
)
SELECT
    'Q' || trimestre::text AS trimestre,
    ROUND(ingreso::numeric, 2) AS ingreso_total,
    ROUND((ingreso / SUM(ingreso) OVER () * 100)::numeric, 2) AS porcentaje
FROM ingresos_trimestre
ORDER BY trimestre;
```

**Explicacion:**
- `EXTRACT(QUARTER FROM ...)` obtiene el trimestre de la fecha.
- `SUM(...) OVER ()` calcula el total anual sin subconsulta adicional.
- `WITH` (CTE) mejora legibilidad separando calculos.
- **Error comun:** Dividir por el total anial sin usar window function (requiere subconsulta).

</details>

## Seccion 3: JOINs (5 preguntas)

### Pregunta 9 - Nivel: Medio

**Contexto:** El equipo de ventas necesita un reporte completo de pedidos con informacion del cliente y empleado asignado.

**Enunciado:** Mostrar el ID del pedido, fecha del pedido, nombre del cliente, nombre del empleado que lo gestiono y el transportista. Incluir solo pedidos de 1997. Ordenar por fecha descendente.

<details>
<summary>Solucion</summary>

```sql
SELECT
    o.order_id,
    o.order_date,
    c.company_name AS cliente,
    e.first_name || ' ' || e.last_name AS empleado,
    s.company_name AS transportista
FROM orders o
INNER JOIN customers c ON o.customer_id = c.customer_id
INNER JOIN employees e ON o.employee_id = e.employee_id
INNER JOIN shippers s ON o.ship_via = s.shipper_id
WHERE EXTRACT(YEAR FROM o.order_date) = 1997
ORDER BY o.order_date DESC;
```

**Explicacion:**
- Tres JOINs para enriquecer el pedido con cliente, empleado y transportista.
- `WHERE EXTRACT(YEAR FROM ...) = 1997` filtra antes de mostrar.
- **Error comun:** Usar `o.order_date BETWEEN '1997-01-01' AND '1997-12-31'` (puede fallar si hay hora).

</details>

### Pregunta 10 - Nivel: Medio

**Contexto:** Un analista necesita identificar clientes que nunca han hecho pedidos para una campana de reactivacion.

**Enunciado:** Listar el ID del cliente, nombre de la empresa, ciudad y pais de los clientes que NO tienen ningun pedido registrado. Usar el patron Anti-Join (LEFT JOIN + IS NULL).

<details>
<summary>Solucion</summary>

```sql
SELECT
    c.customer_id,
    c.company_name,
    c.city,
    c.country
FROM customers c
LEFT JOIN orders o ON c.customer_id = o.customer_id
WHERE o.order_id IS NULL;
```

**Explicacion:**
- `LEFT JOIN` preserva todos los clientes; `IS NULL` detecta los sin pedidos.
- Anti-Join es mas eficiente que `NOT IN` con subconsulta.
- **Error comun:** Usar `NOT IN (SELECT customer_id FROM orders)` (riesgo si hay NULLs).

</details>

### Pregunta 11 - Nivel: Medio

**Contexto:** El equipo de supply chain quiere saber que proveedores tienen productos que nunca se han vendido.

**Enunciado:** Listar el nombre del proveedor, pais, y la cantidad de productos que nunca han aparecido en un pedido. Solo proveedores con al menos 1 producto sin ventas.

<details>
<summary>Solucion</summary>

```sql
SELECT
    s.company_name AS proveedor,
    s.country,
    COUNT(p.product_id) AS productos_sin_ventas
FROM suppliers s
INNER JOIN products p ON s.supplier_id = p.supplier_id
LEFT JOIN order_details od ON p.product_id = od.product_id
WHERE od.product_id IS NULL
GROUP BY s.company_name, s.country
HAVING COUNT(p.product_id) >= 1
ORDER BY productos_sin_ventas DESC;
```

**Explicacion:**
- `LEFT JOIN order_details` + `WHERE IS NULL` identifica productos sin ventas.
- `GROUP BY` agrega por proveedor.
- **Error comun:** Usar `NOT EXISTS` sin GROUP por proveedor (requiere logica adicional).

</details>

### Pregunta 12 - Nivel: Medio

**Contexto:** Un gerente necesita ver la jerarquia de la organizacion: quien reporta a quien.

**Enunciado:** Mostrar el nombre del empleado, su cargo, el nombre de su jefe directo y el cargo del jefe. Incluir al presidente (sin jefe) con un valor "Sin jefe" en las columnas del jefe.

<details>
<summary>Solucion</summary>

```sql
SELECT
    e.first_name || ' ' || e.last_name AS empleado,
    e.title AS cargo_empleado,
    COALESCE(m.first_name || ' ' || m.last_name, 'Sin jefe') AS jefe,
    COALESCE(m.title, 'N/A') AS cargo_jefe
FROM employees e
LEFT JOIN employees m ON e.reports_to = m.employee_id
ORDER BY e.employee_id;
```

**Explicacion:**
- `LEFT JOIN` preserva al presidente (reports_to IS NULL).
- `COALESCE` reemplaza NULLs con valores legibles.
- **Error comun:** Usar `INNER JOIN` que excluiria al presidente.

</details>

### Pregunta 13 - Nivel: Dificil

**Contexto:** El equipo de finanzas necesita un reporte completo que cruce clientes, pedidos y productos para conciliacion mensual.

**Enunciado:** Para cada cliente, mostrar: nombre de la empresa, total de pedidos realizados, fecha del primer pedido, fecha del ultimo pedido, y el producto mas caro que han comprado. Solo clientes con mas de 5 pedidos.

<details>
<summary>Solucion</summary>

```sql
WITH metricas_cliente AS (
    SELECT
        o.customer_id,
        COUNT(DISTINCT o.order_id) AS total_pedidos,
        MIN(o.order_date) AS primer_pedido,
        MAX(o.order_date) AS ultimo_pedido
    FROM orders o
    GROUP BY o.customer_id
    HAVING COUNT(DISTINCT o.order_id) > 5
),
producto_mas_caro AS (
    SELECT DISTINCT ON (o.customer_id)
        o.customer_id,
        p.product_name,
        od.unit_price
    FROM orders o
    INNER JOIN order_details od ON o.order_id = od.order_id
    INNER JOIN products p ON od.product_id = p.product_id
    ORDER BY o.customer_id, od.unit_price DESC
)
SELECT
    c.company_name,
    mc.total_pedidos,
    mc.primer_pedido,
    mc.ultimo_pedido,
    pcm.product_name AS producto_mas_caro,
    pcm.unit_price AS precio
FROM metricas_cliente mc
INNER JOIN customers c ON mc.customer_id = c.customer_id
LEFT JOIN producto_mas_caro pcm ON mc.customer_id = pcm.customer_id
ORDER BY mc.total_pedidos DESC;
```

**Explicacion:**
- Dos CTEs separan metricas de pedidos y producto mas caro.
- `DISTINCT ON` (PostgreSQL) obtiene el producto mas caro por cliente.
- **Error comun:** Usar `LIMIT 1` sin `DISTINCT ON` (solo retorna 1 fila total).

</details>

## Seccion 4: Subconsultas (4 preguntas)

### Pregunta 14 - Nivel: Medio

**Contexto:** Un analista quiere comparar cada producto contra el promedio de su categoria.

**Enunciado:** Listar el nombre del producto, precio unitario, nombre de la categoria y el precio promedio de la categoria. Solo productos con precio MAYOR al promedio de su categoria.

<details>
<summary>Solucion</summary>

```sql
SELECT
    p.product_name,
    p.unit_price,
    c.category_name,
    sub.precio_promedio_categoria
FROM products p
INNER JOIN categories c ON p.category_id = c.category_id
INNER JOIN (
    SELECT category_id, AVG(unit_price) AS precio_promedio_categoria
    FROM products
    GROUP BY category_id
) sub ON p.category_id = sub.category_id
WHERE p.unit_price > sub.precio_promedio_categoria
ORDER BY c.category_name, p.unit_price DESC;
```

**Explicacion:**
- Subconsulta Derived Table calcula promedio por categoria.
- JOIN con la derived table para comparar cada producto contra su promedio.
- **Error comun:** Usar subconsulta correlacionada en WHERE (menos eficiente).

</details>

### Pregunta 15 - Nivel: Medio

**Contexto:** Un gerente quiere saber que clientes han comprado productos de la categoria "Beverages".

**Enunciado:** Listar el ID del cliente, nombre de la empresa y el total gastado en productos de la categoria "Beverages". Ordenar por total gastado descendente.

<details>
<summary>Solucion</summary>


**Explicacion:**

### Pregunta 16 - Nivel: Dificil

**Contexto:** Un analista quiere encontrar productos cuyo precio es mayor que TODOS los productos de la categoria 1 (Beverages).

**Enunciado:** Listar el nombre del producto, precio unitario y categoria de los productos que sean mas caros que el producto mas caro de la categoria "Beverages". Usar subconsulta con ALL.

<details>
<summary>Solucion</summary>

```sql
SELECT
    p.product_name,
    p.unit_price,
    c.category_name
FROM products p
INNER JOIN categories c ON p.category_id = c.category_id
WHERE p.unit_price > ALL (
    SELECT p2.unit_price
    FROM products p2
    INNER JOIN categories c2 ON p2.category_id = c2.category_id
    WHERE c2.category_name = 'Beverages'
)
ORDER BY p.unit_price DESC;
```

**Explicacion:**
- `> ALL` retorna TRUE si el valor supera TODOS los de la subconsulta.
- Equivale a `> (SELECT MAX(unit_price) FROM ... WHERE category = 'Beverages')`.
- **Error comun:** Confundir `> ANY` (mayor que al menos uno) con `> ALL` (mayor que todos).

</details>

### Pregunta 17 - Nivel: Medio

**Contexto:** Un gerente quiere saber el ultimo pedido de cada cliente para una encuesta de satisfaccion.

**Enunciado:** Para cada cliente, mostrar el nombre de la empresa, el ID del ultimo pedido, la fecha y el flete. Usar subconsulta correlacionada en SELECT.

## Seccion 5: Funciones de Ventana (4 preguntas)

### Pregunta 18 - Nivel: Dificil

**Contexto:** Un analista quiere rankear a los productos mas vendidos por categoria.

**Enunciado:** Rankear los productos por cantidad total vendida dentro de cada categoria. Mostrar: nombre de la categoria, nombre del producto, unidades vendidas y el ranking (1 = mas vendido). Usar ROW_NUMBER.

<details>
<summary>Solucion</summary>

```sql
WITH ventas_producto AS (
    SELECT
        c.category_name,
        p.product_name,
        SUM(od.quantity) AS unidades_vendidas
    FROM order_details od
    INNER JOIN products p ON od.product_id = p.product_id
    INNER JOIN categories c ON p.category_id = c.category_id
    GROUP BY c.category_name, p.product_name
)
SELECT
    category_name,
    product_name,
    unidades_vendidas,
    ROW_NUMBER() OVER (
        PARTITION BY category_name
        ORDER BY unidades_vendidas DESC
    ) AS ranking
FROM ventas_producto
ORDER BY category_name, ranking;
```

**Explicacion:**
- `ROW_NUMBER()` asigna ranking unico por particion.
- `PARTITION BY` reinicia el ranking en cada categoria.
- **Error comun:** Usar `RANK()` en lugar de `ROW_NUMBER()` (RANK puede tener empates).

</details>

### Pregunta 19 - Nivel: Dificil

**Contexto:** Un gerente quiere ver como evolucionaron las ventas mes a mes de cada empleado.

**Enunciado:** Para cada empleado, mostrar el mes, las ventas del mes, las ventas del mes anterior y la variacion porcentual. Usar LAG. Solo empleados con al menos 3 meses de actividad.

<details>
<summary>Solucion</summary>

```sql
WITH ventas_mensuales AS (
    SELECT
        o.employee_id,
        DATE_TRUNC('month', o.order_date)::date AS mes,
        SUM(od.quantity * od.unit_price * (1 - od.discount)) AS ventas
    FROM orders o
    INNER JOIN order_details od ON o.order_id = od.order_id
    GROUP BY o.employee_id, DATE_TRUNC('month', o.order_date)
),
con_lag AS (
    SELECT
        employee_id,
        mes,
        ventas,
        LAG(ventas) OVER (PARTITION BY employee_id ORDER BY mes) AS ventas_anterior
    FROM ventas_mensuales
)
SELECT
    e.first_name || ' ' || e.last_name AS empleado,
    cl.mes,
    ROUND(cl.ventas::numeric, 2) AS ventas_mes,
    ROUND(cl.ventas_anterior::numeric, 2) AS ventas_anterior,
    ROUND(((cl.ventas - cl.ventas_anterior) / NULLIF(cl.ventas_anterior, 0) * 100)::numeric, 2) AS variacion_pct
FROM con_lag cl
INNER JOIN employees e ON cl.employee_id = e.employee_id
WHERE cl.ventas_anterior IS NOT NULL
ORDER BY empleado, cl.mes;
```

**Explicacion:**
- `LAG(ventas)` obtiene el valor de la fila anterior en la particion.
- `NULLIF(..., 0)` evita division por cero.
- **Error comun:** No particionar por empleado (compararia meses de diferentes empleados).

</details>

### Pregunta 20 - Nivel: Dificil

**Contexto:** Un analista quiere calcular el promedio movil de 3 meses de ventas totales para detectar tendencias.

**Enunciado:** Calcular las ventas totales por mes y su promedio movil de 3 meses. Mostrar: mes, ventas totales, promedio movil y si la venta esta por encima o por debajo del promedio. Usar ROWS BETWEEN.

<details>
<summary>Solucion</summary>

```sql
WITH ventas_mensuales AS (
    SELECT
        DATE_TRUNC('month', o.order_date)::date AS mes,
        SUM(od.quantity * od.unit_price * (1 - od.discount)) AS ventas
    FROM orders o
    INNER JOIN order_details od ON o.order_id = od.order_id
    GROUP BY DATE_TRUNC('month', o.order_date)
)
SELECT
    mes,
    ROUND(ventas::numeric, 2) AS ventas_totales,
    ROUND(AVG(ventas) OVER (
        ORDER BY mes
        ROWS BETWEEN 2 PRECEDING AND CURRENT ROW
    )::numeric, 2) AS promedio_movil_3m,
    CASE
        WHEN ventas > AVG(ventas) OVER (
            ORDER BY mes
            ROWS BETWEEN 2 PRECEDING AND CURRENT ROW
        ) THEN 'Por encima'
        ELSE 'Por debajo'
    END AS tendencia
FROM ventas_mensuales
ORDER BY mes;
```

**Explicacion:**
- `ROWS BETWEEN 2 PRECEDING AND CURRENT ROW` define ventana de 3 filas.
- `CASE` clasifica cada mes contra su promedio movil.
- **Error comun:** Usar `RANGE BETWEEN` (agrupa valores iguales, no filas).

</details>

### Pregunta 21 - Nivel: Dificil

**Contexto:** Un gerente quiere ver el acumulado de ventas por cliente mes a mes para analizar comportamiento de compra.

**Enunciado:** Para cada cliente, mostrar el mes, las ventas del mes y las ventas acumuladas desde su primer pedido. Usar RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW.

<details>
<summary>Solucion</summary>

```sql
WITH ventas_cliente_mes AS (
    SELECT
        o.customer_id,
        DATE_TRUNC('month', o.order_date)::date AS mes,
        SUM(od.quantity * od.unit_price * (1 - od.discount)) AS ventas
    FROM orders o
    INNER JOIN order_details od ON o.order_id = od.order_id
    GROUP BY o.customer_id, DATE_TRUNC('month', o.order_date)
)
SELECT
    c.company_name,
    vcm.mes,
    ROUND(vcm.ventas::numeric, 2) AS ventas_mes,
    ROUND(SUM(vcm.ventas) OVER (
        PARTITION BY vcm.customer_id
        ORDER BY vcm.mes
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    )::numeric, 2) AS ventas_acumuladas
FROM ventas_cliente_mes vcm
INNER JOIN customers c ON vcm.customer_id = c.customer_id
ORDER BY c.company_name, vcm.mes;
```

**Explicacion:**
- `SUM(...) OVER (PARTITION BY ... ORDER BY ... ROWS BETWEEN UNBOUNDED PRECEDING)` acumula desde el inicio.
- `PARTITION BY customer_id` reinicia la acumulacion por cliente.
- **Error comun:** Olvidar `PARTITION BY` (acumularia todos los clientes juntos).

</details>

## Seccion 6: CTEs (3 preguntas)

### Pregunta 22 - Nivel: Dificil

**Contexto:** Un data architect necesita un reporte ejecutivo que resuma la salud del negocio.

**Enunciado:** Crear un CTE que calcule: total de clientes, total de pedidos, ingreso total, ticket promedio por pedido, y el cliente con mayor facturacion. Mostrar todo en una sola fila.

<details>
<summary>Solucion</summary>

```sql
WITH metricas_generales AS (
    SELECT
        COUNT(DISTINCT o.customer_id) AS total_clientes,
        COUNT(DISTINCT o.order_id) AS total_pedidos,
        ROUND(SUM(od.quantity * od.unit_price * (1 - od.discount))::numeric, 2) AS ingreso_total,
        ROUND((SUM(od.quantity * od.unit_price * (1 - od.discount)) / COUNT(DISTINCT o.order_id))::numeric, 2) AS ticket_promedio
    FROM orders o
    INNER JOIN order_details od ON o.order_id = od.order_id
),
top_cliente AS (
    SELECT
        o.customer_id,
        c.company_name,
        ROUND(SUM(od.quantity * od.unit_price * (1 - od.discount))::numeric, 2) AS facturacion
    FROM orders o
    INNER JOIN order_details od ON o.order_id = od.order_id
    INNER JOIN customers c ON o.customer_id = c.customer_id
    GROUP BY o.customer_id, c.company_name
    ORDER BY facturacion DESC
    LIMIT 1
)
SELECT
    mg.total_clientes,
    mg.total_pedidos,
    mg.ingreso_total,
    mg.ticket_promedio,
    tc.company_name AS cliente_top,
    tc.facturacion AS facturacion_top
FROM metricas_generales mg, top_cliente tc;
```

**Explicacion:**
- Dos CTEs independientes: metricas generales y top cliente.
- CROSS JOIN implicito (ambas CTEs tienen 1 fila).
- **Error comun:** intentar hacer ambos calculos en un solo CTE (mezcla agregacion con TOP 1).

</details>

### Pregunta 23 - Nivel: Dificil

**Contexto:** Un analista quiere crear una tabla temporal de dias habiles para analysis de lead time de pedidos.

**Enunciado:** Generar una serie de fechas desde '1997-01-01' hasta '1997-01-31' y marcar cada dia como laboral o fin de semana. Usar CTE con generate_series.

<details>
<summary>Solucion</summary>

```sql
WITH dias AS (
    SELECT
        generate_series(
            '1997-01-01'::date,
            '1997-01-31'::date,
            '1 day'::interval
        )::date AS fecha
)
SELECT
    fecha,
    TO_CHAR(fecha, 'Day') AS dia_semana,
    CASE
        WHEN EXTRACT(ISODOW FROM fecha) BETWEEN 1 AND 5 THEN 'Laboral'
        ELSE 'Fin de semana'
    END AS tipo_dia
FROM dias
ORDER BY fecha;
```

**Explicacion:**
- `generate_series` crea una secuencia de fechas automaticamente.
- `EXTRACT(ISODOW FROM fecha)` retorna 1=Lunes ... 7=Domingo.
- **Error comun:** Usar `EXTRACT(DOW FROM ...)` que retorna 0=Domingo (confuso).

</details>

### Pregunta 24 - Nivel: Dificil

**Contexto:** Un gerente quiere simular que pasaria si aumentamos los precios de productos un 10%.

**Enunciado:** Crear un CTE con los productos actuales y otro CTE con precios simulados (+10%). Mostrar comparacion: nombre, precio actual, precio simulado y diferencia. Solo productos activos.

<details>
<summary>Solucion</summary>

```sql
WITH precios_actuales AS (
    SELECT
        product_id,
        product_name,
        unit_price AS precio_actual
    FROM products
    WHERE discontinued = 0
),
precios_simulados AS (
    SELECT
        product_id,
        ROUND((precio_actual * 1.10)::numeric, 2) AS precio_simulado
    FROM precios_actuales
)
SELECT
    pa.product_name,
    pa.precio_actual,
    ps.precio_simulado,
    ROUND((ps.precio_simulado - pa.precio_actual)::numeric, 2) AS diferencia
FROM precios_actuales pa
INNER JOIN precios_simulados ps ON pa.product_id = ps.product_id
ORDER BY diferencia DESC;
```

**Explicacion:**
- Dos CTEs encadenados: el segundo referencia al primero.
- Simulacion simple de incremento de precios.
- **Error comun:** Calcular el incremento en la consulta externa sin CTE (menos legible).

</details>

## Seccion 7: Optimizacion (2 preguntas)

### Pregunta 25 - Nivel: Dificil

**Contexto:** Un DBA necesita identificar queries lentas y proponer mejoras.

**Enunciado:** Explica que index crearias para optimizar esta consulta y por que:

```sql
SELECT c.company_name, COUNT(o.order_id) AS total_pedidos
FROM customers c
LEFT JOIN orders o ON c.customer_id = o.customer_id
WHERE c.country = 'Germany'
GROUP BY c.company_name
ORDER BY total_pedidos DESC;
```

<details>
<summary>Respuesta</summary>

**Indices recomendados:**

1. **`CREATE INDEX idx_customers_country ON customers(country);`**
   - Permite Index Scan en vez de Seq Scan para el filtro `WHERE country = 'Germany'`.
   - Reduce el conjunto de clientes de 91 a ~10 antes del JOIN.

2. **`CREATE INDEX idx_orders_customer_id ON orders(customer_id);`**
   - Acelera el LEFT JOIN (Hash Join o Nested Loop con Index Scan).
   - Sin este indice, PostgreSQL haria Seq Scan sobre 830 pedidos.

3. **`CREATE INDEX idx_customers_country_covering ON customers(country, company_name);`**
   - Index-Only Scan: PostgreSQL responde la consulta solo desde el indice sin tocar el Heap.
   - Ahorra E/S de disco.

**Explicacion:**
- El filtro `WHERE country = 'Germany'` es el predicado mas selectivo.
- El JOIN es la operacion mas costosa; un indice en la FK acelera enormemente.
- **Error comun:** Crear indices en columnas que ya tienen PK (innecesario).

</details>

### Pregunta 26 - Nivel: Dificil

**Contexto:** Un analista ejecuta esta consulta frecuentemente y es lenta:

```sql
SELECT product_name, unit_price
FROM products
WHERE LOWER(product_name) LIKE '%chocolate%';
```

**Enunciado:** Explica por que es lenta y como la reescribirias para mejorar el rendimiento.

<details>
<summary>Respuesta</summary>

**Problema:**
- `LOWER(product_name)` impide usar indices B-Tree estandar (funcion no indexable).
- `LIKE '%chocolate%'` con comodin al inicio no permite Index Range Scan.
- Resultado: Seq Scan completo sobre 77 filas (escaneo secuencial).

**Soluciones:**

**Opcion 1: Usar pg_trgm + GIN (recomendado)**
```sql
CREATE EXTENSION IF NOT EXISTS pg_trgm;
CREATE INDEX idx_products_name_trgm ON products USING GIN (product_name gin_trgm_ops);

SELECT product_name, unit_price
FROM products
WHERE product_name ILIKE '%chocolate%';
```
- `pg_trgm` indexa trigramas (secuencias de 3 caracteres).
- `ILIKE` con GIN index es extremadamente rapido.

**Opcion 2: Usar un indice funcional**
```sql
CREATE INDEX idx_products_lower_name ON products(LOWER(product_name));

SELECT product_name, unit_price
FROM products
WHERE LOWER(product_name) LIKE '%chocolate%';
```
- Indexa el resultado de `LOWER()`.
- Permite Index Scan en vez de Seq Scan.

**Explicacion:**
- Funciones en columnas destruyen la usabilidad de indices.
- `pg_trgm` es la solucion estandar para busquedas de texto parcial.
- **Error comun:** Agregar mas filtros sin atacar la causa raiz (funcion no indexable).

</details>

## Seccion 8: Caso Practico (2 preguntas)

### Pregunta 27 - Nivel: Dificil

**Contexto:** La gerencia quiere un dashboard de ventas que muestre KPIs clave del ultimo trimestre con datos.

**Enunciado:** Construir una consulta que retorne en UNA SOLA FILA:
- Total de pedidos del ultimo trimestre
- Total de clientes unicos que compraron
- Ingreso total del trimestre
- Ticket promedio por pedido
- Producto mas vendido (nombre)
- Pais con mas pedidos (ship_country)

<details>
<summary>Solucion</summary>

```sql
WITH pedidos_trimestre AS (
    SELECT o.order_id, o.customer_id, o.ship_country, o.order_date
    FROM orders o
    WHERE o.order_date >= (
        SELECT MAX(order_date) - INTERVAL '3 months' FROM orders
    )
    AND o.order_date <= (SELECT MAX(order_date) FROM orders)
),
detalle_ventas AS (
    SELECT
        pt.order_id,
        pt.customer_id,
        pt.ship_country,
        od.product_id,
        od.quantity * od.unit_price * (1 - od.discount) AS importe
    FROM pedidos_trimestre pt
    INNER JOIN order_details od ON pt.order_id = od.order_id
),
producto_top AS (
    SELECT product_id, SUM(quantity) AS total_unidades
    FROM order_details
    WHERE order_id IN (SELECT order_id FROM pedidos_trimestre)
    GROUP BY product_id
    ORDER BY total_unidades DESC
    LIMIT 1
),
pais_top AS (
    SELECT ship_country, COUNT(*) AS total
    FROM pedidos_trimestre
    GROUP BY ship_country
    ORDER BY total DESC
    LIMIT 1
)
SELECT
    (SELECT COUNT(DISTINCT order_id) FROM pedidos_trimestre) AS total_pedidos,
    (SELECT COUNT(DISTINCT customer_id) FROM pedidos_trimestre) AS total_clientes,
    (SELECT ROUND(SUM(importe)::numeric, 2) FROM detalle_ventas) AS ingreso_total,
    (SELECT ROUND((SUM(importe) / COUNT(DISTINCT order_id))::numeric, 2) FROM detalle_ventas) AS ticket_promedio,
    (SELECT product_name FROM products WHERE product_id = (SELECT product_id FROM producto_top)) AS producto_top,
    (SELECT ship_country FROM pais_top) AS pais_top;
```

**Explicacion:**
- Multiples CTEs separan logica: pedidos del trimestre, detalle de ventas, producto top, pais top.
- Subconsultas escalares en SELECT para cada KPI.
- **Error comun:** intentar calcular todo en un solo nivel de agregacion (mezcla metricas que requieren diferentes granularidades).

</details>

### Pregunta 28 - Nivel: Dificil

**Contexto:** Un data engineer necesita diseñar una tabla de hechos de ventas para un Data Warehouse simple.

**Enunciado:** Diseña y crea una tabla de hechos `fact_ventas` usando los datos de Northwind. La tabla debe tener:
- `order_id` (INT, FK)
- `product_id` (INT, FK)
- `customer_id` (VARCHAR, FK)
- `employee_id` (INT, FK)
- `order_date` (DATE)
- `quantity` (INT)
- `unit_price` (NUMERIC)
- `discount` (NUMERIC)
- `ingreso_neto` (NUMERIC, calculado)
- `trimestre` (INT, calculado)
- `anio` (INT, calculado)

Luego inserta los datos usando una consulta INSERT...SELECT.

<details>
<summary>Solucion</summary>

```sql
-- 1. Crear la tabla de hechos
CREATE TABLE fact_ventas (
    order_id INT NOT NULL,
    product_id INT NOT NULL,
    customer_id VARCHAR(5),
    employee_id INT,
    order_date DATE,
    quantity INT,
    unit_price NUMERIC,
    discount NUMERIC,
    ingreso_neto NUMERIC GENERATED ALWAYS AS (
        ROUND((quantity * unit_price * (1 - discount))::numeric, 2)
    ) STORED,
    trimestre INT GENERATED ALWAYS AS (
        EXTRACT(QUARTER FROM order_date)::int
    ) STORED,
    anio INT GENERATED ALWAYS AS (
        EXTRACT(YEAR FROM order_date)::int
    ) STORED,
    PRIMARY KEY (order_id, product_id)
);

-- 2. Insertar datos
INSERT INTO fact_ventas (
    order_id, product_id, customer_id, employee_id,
    order_date, quantity, unit_price, discount
)
SELECT
    od.order_id,
    od.product_id,
    o.customer_id,
    o.employee_id,
    o.order_date,
    od.quantity,
    od.unit_price,
    od.discount
FROM order_details od
INNER JOIN orders o ON od.order_id = o.order_id;

-- 3. Verificar
SELECT
    anio,
    trimestre,
    COUNT(*) AS total_lineas,
    ROUND(SUM(ingreso_neto)::numeric, 2) AS ingreso_total
FROM fact_ventas
GROUP BY anio, trimestre
ORDER BY anio, trimestre;
```

**Explicacion:**
- `GENERATED ALWAYS AS ... STORED` calcula columnas derivadas automaticamente.
- PK compuesta `(order_id, product_id)` es la natural de la tabla de hechos.
- El INSERT...SELECT trae datos de las tablas transaccionales.
- **Error comun:** No crear PK compuesta (duplicaria filas en cargas incrementales).

</details>
