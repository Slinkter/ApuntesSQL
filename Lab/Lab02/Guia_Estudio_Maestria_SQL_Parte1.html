1. Introducción a SQL

[00:00:00 → 00:08:00]

El curso inicia destacando la importancia de SQL para interactuar con bases de datos relacionales. Su objetivo es almacenar y manejar grandes cantidades de datos.

SQL (Structured Query Language) es el lenguaje estándar diseñado específicamente para crear, gestionar y consultar bases de datos relacionales. Permite almacenar, modificar y acceder a información de forma estructurada.

📌 ÉNFASIS DEL PROFESOR:
SQL NO es un lenguaje de propósito general (como Python, Java o JavaScript que se pueden usar para hacer desarrollo web o automatizar cosas en general). Es un lenguaje de tipo "descriptivo" (declarativo), enfocado exclusivamente en consultas a bases de datos.

Sin embargo, cumple con las características formales de un lenguaje de programación:

  • Tiene sintaxis y reglas propias.
  • Permite operaciones lógicas y condicionales (ej. WHERE, CASE, IF).
  • Permite crear scripts y procedimientos almacenados (Store Procedures), que ejecutan secuencias complejas de comandos directamente en la base de datos.

2. Bases de Datos: Relacionales vs No Relacionales

[00:08:00 → 00:15:00]

Una Base de Datos (BBDD) es un sistema organizado para almacenar, gestionar y acceder a información de forma eficiente. La base de datos NO es lo mismo que el lenguaje (SQL) utilizado para consultarla.

2.1 Base de Datos Relacional (SQL)

  • Organiza la información en tablas con filas y columnas.
  • Usa relaciones entre las tablas mediante claves primarias (Primary Keys) y foráneas (Foreign Keys).
  • Es ideal para datos estructurados y consistentes.
  • Requiere un esquema fijo (la estructura estricta que un dato debe cumplir para ser válido).

2.2 Base de Datos No Relacional (NoSQL)

  • Almacena datos en estructuras flexibles como documentos, pares clave-valor o grafos.
  • No requiere un esquema fijo, lo que le da muchísima flexibilidad. Un registro puede tener campos totalmente distintos a otro.
  • Ideal para datos grandes, distribuidos y sin estructura definida.
El profesor compara los pares clave-valor de NoSQL con los "diccionarios" en Python o los "objetos" en JavaScript.

3. ¿Para qué se usa SQL? (Casos Prácticos en la Industria)

[00:15:00 → 00:22:00]

El profesor detalla los principales roles y escenarios empresariales donde SQL es la herramienta principal:

  • Desarrollo Web: SQL permite la interacción directa de cara al usuario.
    • Ejemplo 1: Listar todos los comentarios que ha hecho un usuario específico en una red social.
    • Ejemplo 2: Mostrar en pantalla únicamente los productos que tienen inventario disponible (en stock) dentro de una tienda online.
  • Análisis de Datos: Obtención de métricas masivas estructuradas.
    • Ejemplo 1: Obtener la cantidad exacta de ventas realizadas durante un mes en particular.
    • Ejemplo 2: Listar y agrupar a los clientes que más compran para aplicarles descuentos.
  • Ingeniería de Software: Operaciones de modificación y trazabilidad del sistema.
    • Ejemplo 1: Máquinas de estado: Actualizar el estado de un pedido modificando su columna de "pendiente" a "enviado".
    • Ejemplo 2: Logs y Trazabilidad (Bancos): Registrar el historial de cambios de un documento, asegurando que cada mutación en el sistema deje una huella auditable en una tabla exclusiva de logs.
  • Ciencia de Datos: SQL es extremadamente poderoso para procesar datos de muchísimo volumen de forma veloz.
    • Ejemplo: Obtener los valores promedios matemáticos de una métrica específica divididos y agrupados por región geográfica.

4. Diferencia entre Lenguaje SQL y Motores DBMS

[00:22:00 → 00:28:00]
📌 ÉNFASIS DEL PROFESOR:
"¡No confundan SQL con MySQL! SQL es el IDIOMA que le dice a la base de datos qué hacer. MySQL, PostgreSQL, SQL Server y Oracle son los MOTORES (DBMS), es decir, el programa de software físico que almacena los datos y que ENTIENDE el idioma SQL."

Aunque todos los motores relacionales entienden SQL, cada motor tiene diferencias que los hacen únicos para distintos proyectos:

  • Diferencias de Rendimiento: Qué tan rápido y eficiente funciona cada motor en escenarios específicos.
  • Diferencias de Licencias: Algunos son de código abierto (gratuitos) y otros son empresariales y de pago (ej. Oracle, Microsoft SQL Server).
  • Funciones Adicionales (Dialectos): SQL es el estándar general, pero cada motor añade pequeñas funcionalidades específicas (ej. manejo avanzado de JSON en PostgreSQL) provocando ligeras diferencias en la forma de escribir las consultas según la marca del DBMS.

5. Jerarquía Estructural de las Bases de Datos

[00:28:00 → 00:35:00]

Para entender dónde se alojan los datos, el profesor explica la jerarquía de mayor a menor:

graph TD; A[Servidor / Hardware] -->|Aloja| B[Motor DBMS] B -->|Contiene| C[Base de Datos] C -->|Agrupa| D[Tablas] D -->|Estructura| E[Columnas / Campos] D -->|Almacena| F[Filas / Registros] style A fill:#f9e79f,stroke:#f39c12,stroke-width:2px style B fill:#aed6f1,stroke:#2980b9,stroke-width:2px style C fill:#abebc6,stroke:#27ae60,stroke-width:2px style D fill:#f5b041,stroke:#d35400,stroke-width:2px
  1. Servidor: La máquina física o software local que aloja uno o varios motores de base de datos.
  2. Base de Datos (Database): Un conjunto organizado que puede contener múltiples tablas relacionadas. Un servidor puede tener múltiples BBDD.
  3. Tablas (Tables): Estructuras bidimensionales de datos compuestas por filas y columnas (ej. estudiantes, cursos).
  4. Registros (Filas): Cada entrada horizontal en una tabla. Un dato de entidad completo (ej. el estudiante "Juan").
  5. Campos (Columnas): Los atributos verticales de un registro (ej. edad, nombre, email).

7. DDLData Definition Language: Define la estructura inmutable (CREATE, DROP, ALTER): Lenguaje de Definición de Datos

[00:50:00 → 01:25:00]

El DDLData Definition Language: Define la estructura inmutable (CREATE, DROP, ALTER) define la estructura inmutable de las tablas.

Creación Básica

CREATE DATABASE ecommerce;
USE ecommerce; /* Selecciona la BD (específico de MySQL) */
DROP DATABASE ecommerce; /* Peligroso: borra la base de datos entera */

Tipos de Datos Tácticos

  • INT: Números enteros (IDs, stock).
    • Para tablas con transacciones masivas, usar SERIAL o BIGSERIAL (en PostgreSQL) es más eficiente para el B-Tree de los índices que generar UUIDs v4 aleatorios. Si la seguridad requiere ocultar secuencias en APIs públicas, los arquitectos prefieren UUID v7 (ordenables por tiempo) para evitar la fragmentación severa del disco.
  • VARCHAR(n): Texto corto con límite de caracteres (ej. VARCHAR(255) para emails o nombres).
  • TEXT: Texto extremadamente largo (ej. descripciones de productos).
  • DECIMAL(precision, scale): Números fraccionales. DECIMAL(10,2) asegura 10 dígitos y 2 decimales exactos.
  • BOOLEAN: Verdadero (1) o Falso (0).
  • DATE / DATETIME: Fechas puras o con reloj de tiempo.

Restricciones Críticas (Constraints)

  • PRIMARY KEYClave Primaria: Identificador único e irrepetible (el DNI de la fila): El identificador único e irrepetible de la fila (el "DNI" del registro).
  • AUTO_INCREMENT: Generación numérica secuencial automática para no insertar el ID manualmente.
  • NOT NULL: Prohíbe dejar el campo en blanco.
  • UNIQUE: Evita duplicados en columnas específicas (ideal para correos electrónicos).
  • DEFAULT: Valor automático si se omite en la inserción (ej. DEFAULT 0 para stock).
  • CHECK: Valida lógica booleana antes de permitir la entrada (ej. CHECK (price > 0)).
Ejemplo Final del profesor (Creando tabla Products): 
CREATE TABLE products (
    id INT AUTO_INCREMENT PRIMARY KEYClave Primaria: Identificador único e irrepetible (el DNI de la fila),
    name VARCHAR(100) NOT NULL,
    description TEXT,
    price DECIMAL(10,2) CHECK (price > 0),
    stock INT DEFAULT 0,
    created_at DATE DEFAULT CURRENT_DATE
);

GLOSARIO TÉCNICO

SQL (Structured Query Language):
Es el lenguaje declarativo estándar para interactuar con bases de datos relacionales.

DBMS (Database Management System / Motor):
El software (programa) como MySQL o PostgreSQL que almacena los archivos en el disco y entiende/ejecuta las instrucciones SQL enviadas.

Constraint:
Regla estricta aplicada a una columna que el motor forzará para asegurar que el dato sea válido (ej. NOT NULL, UNIQUE).

EJERCICIOS PRÁCTICOS

Ejercicio 1 — Análisis de Casos Reales
Enunciado: Identifica si los siguientes casos se resuelven de forma óptima con una Base Relacional (SQL) o No Relacional (NoSQL).
1. Almacenar el catálogo de inventario de facturación gubernamental con campos inflexibles.
2. Almacenar logs JSON generados aleatoriamente por 10,000 servidores web con distintas propiedades.
Solución guiada:
1. Relacional (SQL): Al requerir campos inflexibles y estructurados contablemente, un esquema fijo (tablas/columnas) es el estándar obligatorio.
2. No Relacional (NoSQL): Los archivos JSON aleatorios y dinámicos que no comparten los mismos campos requieren una estructura de "documentos" flexible que una base relacional rechazaría por no coincidir con un esquema pre-configurado.
