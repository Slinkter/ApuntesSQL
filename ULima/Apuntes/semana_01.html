Insight del Arquitecto: El Valor del Dato

Los datos son un activo estratégico: En 2013 las empresas comenzaban a entender que los datos son un recurso competitivo. C.J. Date nos recuerda que una base de datos no es solo un lugar donde guardar cosas — es la representación formal de la realidad del negocio.

La clave: Datos → Información → Conocimiento → Decisión. Sin el primer paso, no hay nada.

1. Del Dato al Conocimiento

Conceptos Fundamentales

Los negocios generan datos permanentemente. El dato es un recurso organizacional para competir. Existe necesidad de información en todos los niveles de la organización.

  • Dato: Representación simbólica de un hecho. Objetivo, sin contexto. Ej: 3.50
  • Información: Dato procesado con contexto. Reduce incertidumbre. Ej: "El precio del producto es 3.50"
  • Conocimiento: Información analizada para tomar decisiones. Ej: "El precio subió 75% en 3 años, debemos ajustar estrategia"

Concepto clave: Los datos son el insumo básico. La información surge al procesarlos. El conocimiento emerge al analizar la información para la toma de decisiones.

2. El Problema: Sistema Tradicional de Archivos

Antes del enfoque de BD

Cada aplicación tenía sus propios archivos, generando los siguientes problemas:

Dependencia programa/datos: Cada sistema debe actualizarse con sus propios datos, incluir código para la metadata de cada archivo y tener sus propias rutinas de CRUD. Esto produce débil coordinación y control centralizado.

  • Redundancia de datos: Duplicación de la misma información en múltiples archivos. Espacio desperdiciado y mantenimiento costoso.
  • Inconsistencia: Cuando los datos cambian en un archivo pero no en otro, se compromete la integridad de los datos.
  • Limitaciones para compartir datos: No hay forma estándar de acceso entre distintas aplicaciones.
  • Mayor tiempo de desarrollo: Cada sistema reinventa las rutinas de acceso a datos.
  • Sin estándares: No hay coordinación ni control centralizado sobre los datos.

3. La Solución: Enfoque de Base de Datos

Base de Datos: Colección de datos almacenados en un formato estándar, diseñado para que pueda ser compartido por múltiples usuarios.

Analogía: El DBMS actúa como un bibliotecario experto que conoce la ubicación exacta de cada libro (dato) y puede recuperarlo rápidamente sin que el usuario necesite saber dónde está físicamente almacenado.

Ventajas del enfoque de Base de Datos:

  • Independencia programa/datos: Los metadatos se almacenan en el DBMS. Las aplicaciones no necesitan preocuparse por formatos de datos.
  • Redundancia mínima: Se reduce la duplicación, mejorando consistencia e integridad.
  • Mejor compartición de datos: Diferentes usuarios obtienen diferentes vistas de los datos.
  • Reforzamiento de estándares: Todos los accesos a datos se realizan de la misma manera.
  • Mejor calidad de datos: Restricciones, reglas de validación.
  • Accesibilidad: Uso de un lenguaje estándar de consultas (SQL).
  • Seguridad, Backup/Recovery, Concurrencia: La recuperación ante desastres es más sencilla.

4. Costos y Riesgos

Tipo Ejemplos
Costos iniciales Software, hardware, comunicaciones, conversión de datos, estudios, diseño, capacitación
Costos permanentes Personal especializado (DBA), upgrades de software, mantenimiento, soporte
Riesgos Complejidad de instalación, conflictos organizacionales, determinación de procesos de backup

5. Componentes del Entorno de BD

Componente Función
Herramientas CASE Usadas para diseñar BD y aplicaciones
Aplicación Programa usado para crear, modificar y leer en la BD
Repositorio Base de conocimientos donde se guardan definiciones de datos, relaciones y formatos
Base de Datos Colección organizada de datos lógicamente relacionados
DBMS Software empleado para crear, mantener y proveer información con acceso controlado

6. Ciclo de Vida SDLC y Desarrollo de BD

System Development Life Cycle

El desarrollo de bases de datos sigue las fases del SDLC, mapeando cada etapa a actividades específicas de BD:

Fase SDLC Actividad de BD
Identificación y Planeamiento Modelamiento de la empresa, identificación del negocio
Análisis Modelamiento conceptual de datos (DER)
Diseño Lógico Diseño lógico de la base de datos (modelo relacional)
Diseño Físico Diseño y definición física de la base de datos
Implementación Implementación de la base de datos (DDL, carga de datos)
Mantenimiento Mantenimiento de la base de datos

8. Terminología Clave

  • Transacción: Operación completa de lectura o escritura sobre la base de datos.
  • Database Administrator (DBA): Administrador responsable de la gestión centralizada.
  • Metadata: Describe las propiedades y características de los datos y de su contexto.
