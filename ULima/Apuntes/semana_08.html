La Evaluación Parcial como Hito de Arquitectura

El examen parcial valida el dominio del modelado Entidad-Relación y el diseño lógico en el modelo relacional. Diseñar un esquema limpio y normalizado en esta etapa previene fallas catastróficas de rendimiento e inconsistencia en las fases avanzadas del proyecto de software.

Persistencia Cloud e Integridad Transaccional

Los sistemas empresariales modernos delegan la alta disponibilidad a la nube, pero la consistencia de los datos depende enteramente del diseño del esquema lógico. Una base de datos mal diseñada fallará bajo presión transaccional masiva (Deadlocks), sin importar qué tan grande sea la instancia de AWS asignada.

Instrucciones: El examen parcial cubre las semanas 01-07. Consta de 4 secciones con un total de 100 puntos. Dispone de 90 minutos para completarlo. No se permite material de consulta. Escriba clara y legiblemente. Puntaje minimo aprobatorio: 60.

Tips para el Examen: (1) Lea la pregunta completa antes de responder. (2) En modelamiento E-R, identifique primero las entidades y luego las relaciones. (3) En normalizacion, marque las dependencias funcionales paso a paso. (4) En SQL, escriba las consultas con indentacion clara.

Seccion 4: SQL - Soluciones

Ejercicio 1 - DDL:

CREATE TABLE departamento ( cod_depto CHAR(3) PRIMARY KEY, nombre VARCHAR(50) NOT NULL, presupuesto DECIMAL(10,2) CHECK (presupuesto > 0) ); CREATE TABLE empleado ( cod_emp INT PRIMARY KEY, nombre VARCHAR(100) NOT NULL, salario DECIMAL(8,2), cod_depto CHAR(3), FOREIGN KEY (cod_depto) REFERENCES departamento(cod_depto) ON DELETE CASCADE );

Ejercicio 2 - INSERT:

BEGIN; INSERT INTO departamento VALUES ('TI', 'Tecnologia Informatica', 50000.00), ('VEN', 'Ventas Corporativas', 35000.00); INSERT INTO empleado (cod_emp, nombre, salario, cod_depto) VALUES (1, 'Ana Torres', 4500.00, 'TI'), (2, 'Luis Gomez', 3200.00, 'TI'), (3, 'Pedro Diaz', 1800.00, 'VEN'); COMMIT;

Ejercicio 3 - UPDATE:

UPDATE empleado SET salario = salario * 1.15 WHERE cod_depto = (SELECT cod_depto FROM departamento WHERE nombre = 'TI');

Ejercicio 4 - DELETE:

DELETE FROM empleado WHERE salario < 1500 AND cod_depto = (SELECT cod_depto FROM departamento WHERE nombre = 'VENTAS');

Ejercicio 5 - SELECT con JOIN:

SELECT e.nombre AS empleado, d.nombre AS departamento, p.nombre AS proyecto FROM empleado e JOIN departamento d ON e.cod_depto = d.cod_depto LEFT JOIN proyecto p ON d.cod_depto = p.cod_depto WHERE e.salario > (SELECT AVG(salario) FROM empleado) ORDER BY e.salario DESC;

💡 Tips del Profesor

•

Administre el tiempo en evaluaciones: Al diseñar bases de datos bajo presión, resuelva primero las restricciones de integridad y luego las consultas de lectura.

•

Valide sintaxis SQL de forma estricta: Un solo error de coma o paréntesis en DDL invalida la creación de tablas completas.

•

Pruebe siempre con datos mock representativos: Nunca dé por aprobado un diseño de base de datos sin antes poblarlo con miles de registros aleatorios para verificar el comportamiento de los índices.

✍️ Autoevaluación y Ejercicios Prácticos

1 ¿Cómo se valida que un esquema cumple con la 3FN durante un examen?

Verificando que no existan grupos repetitivos (1FN), que no haya dependencias parciales si hay claves compuestas (2FN) y que no existan dependencias transitivas entre columnas no clave (3FN).

2 ¿Qué importancia tiene el control de concurrencia en la evaluación?

Asegura que las operaciones DML planteadas en las preguntas no generen condiciones de carrera ni lecturas sucias en transacciones simultáneas.

3 ¿Cómo influye el tipo de datos elegido en la optimización de almacenamiento en AWS?

Elegir tipos precisos (ej. SMALLINT para rangos cortos en lugar de INTEGER o VARCHAR) reduce la huella física en memoria y disco, mejorando el rendimiento de los índices.
