Modelado de Datos Dimensional: Kimball vs. Inmon

El almacenamiento analítico para Business Intelligence (Data Warehouses) requiere una estructura optimizada para consultas agregadas y reportes rápidos:

  • Modelo Estrella (Star Schema): Una tabla central de Hechos (Facts, con métricas cuantitativas) rodeada de tablas de Dimensiones (Dimensions, con atributos descriptivos) vinculadas mediante claves PK/FK directas. Sin redundancia analítica pero desnormalizado.
  • Modelo Copo de Nieve (Snowflake Schema): Variante del modelo estrella donde algunas dimensiones se normalizan en subtablas. Ahorra espacio pero incrementa JOINs de lectura.
  • Tabla de Hechos vs. Dimensión: Hechos registra eventos temporales de transacciones; Dimensiones registra los contextos de negocio (Quién, Dónde, Cuándo, Qué).

Arquitectura DW en la Nube y Procesos ETL

En entornos cloud empresariales, el Data Warehouse migra hacia motores columnares dedicados (como AWS Redshift o Google BigQuery), los cuales almacenan los datos por columnas en lugar de filas para acelerar las agregaciones masivas.

Proceso ETL/ELT: Los datos operativos se extraen de bases OLTP, se transforman y se cargan al DW. En nubes modernas, el patrón ELT (Extract, Load, Transform) se prefiere aprovechando la enorme capacidad de cómputo del Data Warehouse de destino para transformar los datos usando SQL.

Objetivos: Comprender los fundamentos de Data Warehousing y Business Intelligence. Diferenciar sistemas OLTP de OLAP. Disenar esquemas estrella y copo de nieve. Conocer el proceso ETL y los tipos de implementacion de cubos OLAP.

Insight del Arquitecto: El DW como Centro de Decisiones

Bill Inmon, padre del Data Warehousing, define el Data Warehouse como "una coleccion de datos orientada a temas, integrada, no volatil y variante en el tiempo, disenada para soportar la toma de decisiones". A diferencia de una base de datos operacional (OLTP) que optimiza transacciones, el DW optimiza consultas analiticas. El 80% del esfuerzo en un proyecto DW esta en el proceso ETL, no en el modelado.

1. Business Intelligence (BI)

Inteligencia de Negocios

Conjunto de productos y servicios que permiten a los usuarios finales acceder y analizar de manera rapida y sencilla la informacion para la toma de decisiones de negocio a nivel operativo, tactico y estrategico.

Componentes clave del BI:

  • Data Warehouse: Repositorio centralizado de datos.
  • Data Mart: Subconjunto departamental (ventas, finanzas, RRHH).
  • Cubos OLAP: Estructura multidimensional para consultas rapidas.
  • ETL: Proceso de extraccion, transformacion y carga.
  • Metadatos: Datos sobre los datos del warehouse.

3. Arquitectura del Data Warehouse

La arquitectura de referencia incluye las siguientes capas:

  • Fuentes de datos: Sistemas operacionales (ERP, CRM), archivos planos, fuentes externas.
  • Extraccion e Integracion: Proceso ETL: extraer, limpiar, transformar, integrar.
  • Almacenamiento: Data Warehouse corporativo + Data Marts departamentales.
  • Aplicaciones: OLAP, reporting, Data Mining, EIS.
  • Metadatos: Repositorio central de definiciones y linaje de datos.

Nota: Existen dos enfoques de implementacion:

  • Top-Down (Inmon): Primero el DW corporativo, luego los Data Marts.
  • Bottom-Up (Kimball): Primero los Data Marts, luego se integran en el DW.

4. Modelo Estrella (Star Schema)

Fact Table + Dimension Tables

El modelo estrella es el esquema más utilizado en Data Warehousing. Consta de una tabla central de hechos (fact table) rodeada de tablas de dimensión (dimension tables) relacionadas en relaciones 1:N directas (tipo estrella).

Conceptos Clave de Diseño:

  • Tabla de Hechos (Fact Table): Almacena eventos de negocio (ventas, transacciones). Contiene las métricas operativas y las claves foráneas (FK) a las dimensiones.
  • Hecho (Fact): Cada fila de la tabla de hechos, que representa un evento específico.
  • Medidas (Measures): Valores cuantitativos (numéricos) acumulables que miden el negocio (ej: ventas total, cantidad vendida).
  • Tablas de Dimensión: Entidades que describen el contexto del negocio (tiempo, clientes, productos, ubicación).

Jerarquías, Niveles y Miembros (Members)

Una dimensión contiene una o más jerarquías compuestas por niveles ordenados de agregación. Por ejemplo, en la dimensión Ubigeo:

  • Niveles de Agregación: País → Departamento → Provincia → Distrito.
  • Miembros (Members): Las ocurrencias reales de cada nivel. Por ejemplo, "Lima" es un miembro del nivel Departamento.

Ejemplo: Estructura del Star Schema clásico (FoodMart)

El esquema de FoodMart representa la estructura real de ventas de una cadena minorista en 1997:

-- Dimensión: store CREATE TABLE store ( store_id INTEGER PRIMARY KEY, store_type VARCHAR(30), region_id INTEGER, store_name VARCHAR(40), store_number INTEGER, store_street_address VARCHAR(50), store_city VARCHAR(30), store_state VARCHAR(30), store_postal_code VARCHAR(15), store_country VARCHAR(30), store_manager VARCHAR(40), store_phone VARCHAR(20), store_fax VARCHAR(20), first_opened_date DATE, last_remodel_date DATE, store_sqft INTEGER, grocery_sqft INTEGER, frozen_sqft INTEGER ); -- Dimensión: promotion CREATE TABLE promotion ( promotion_id INTEGER PRIMARY KEY, promotion_district_id INTEGER, promotion_name VARCHAR(50), media_type VARCHAR(30), cost NUMERIC(10,2), start_date DATE ); -- Dimensión: time_by_day CREATE TABLE time_by_day ( time_id INTEGER PRIMARY KEY, the_date DATE, the_day VARCHAR(15), the_month VARCHAR(15), the_year INTEGER, day_of_month INTEGER, week_of_year INTEGER, month_of_year INTEGER, quarter VARCHAR(10), fiscal_period VARCHAR(10) ); -- Dimensión: product CREATE TABLE product ( product_id INTEGER PRIMARY KEY, product_class_id INTEGER, brand_name VARCHAR(50), product_name VARCHAR(100), SKU VARCHAR(20), SRP NUMERIC(10,2), gross_weight NUMERIC(10,2), net_weight NUMERIC(10,2), recyclable_package BOOLEAN, low_fat BOOLEAN, units_per_case INTEGER, cases_per_pallet INTEGER, shelf_width NUMERIC(5,2), shelf_height NUMERIC(5,2), shelf_depth NUMERIC(5,2) ); -- Dimensión: customer CREATE TABLE customer ( customer_id INTEGER PRIMARY KEY, account_num VARCHAR(30), lname VARCHAR(50), fname VARCHAR(50), mi CHAR(1), address1 VARCHAR(100), address2 VARCHAR(100), address3 VARCHAR(100), address4 VARCHAR(100), city VARCHAR(30), state_province VARCHAR(30), postal_code VARCHAR(15), country VARCHAR(30), customer_region_id INTEGER, phone1 VARCHAR(20) ); 5. Modelo Copo de Nieve (Snowflake Schema)

En el modelo Snowflake, las dimensiones se normalizan en múltiples tablas, una por cada nivel jerárquico. Esto reduce la redundancia al mínimo pero aumenta significativamente la complejidad de las consultas (más JOINs) y ralentiza el procesamiento.

Ejemplo: Estructura Normalizada de Dimensión Producto (Copo de Nieve)

-- Tabla: FAMILIA_DIM CREATE TABLE FAMILIA_DIM ( Familia_Key INTEGER PRIMARY KEY, IdFamilia INTEGER, Familia VARCHAR(50) ); -- Tabla: SUBFAMILIA_DIM CREATE TABLE SUBFAMILIA_DIM ( SubFamilia_Key INTEGER PRIMARY KEY, Familia_Key INTEGER REFERENCES FAMILIA_DIM(Familia_Key), IdSubFamilia INTEGER, SubFamilia VARCHAR(50) ); -- Tabla: MARCA_DIM CREATE TABLE MARCA_DIM ( Marca_Key INTEGER PRIMARY KEY, SubFamilia_Key INTEGER REFERENCES SUBFAMILIA_DIM(SubFamilia_Key), IdMarca INTEGER, Marca VARCHAR(50) ); -- Tabla: PRESENTACION_DIM (Esta es la tabla que se relaciona con la Fact Table) CREATE TABLE PRESENTACION_DIM ( Presentacion_Key INTEGER PRIMARY KEY, Marca_Key INTEGER REFERENCES MARCA_DIM(Marca_Key), IdPresentacion INTEGER, Presentacion VARCHAR(50) );

Comparación: STAR vs SNOWFLAKE

Caracteristica STAR (Estrella) SNOWFLAKE (Copo de Nieve)
Entendimiento del modelo Sencillo Mayor dificultad
Numero de tablas Menor Mayor
Complejidad de consulta Baja Alta
Rendimiento Rapida Lenta

6. Proceso ETL

Extraccion, Transformacion, Carga

El proceso ETL consume aproximadamente el 80% del tiempo y recursos en un proyecto Data Warehouse (segun Bill Inmon).

Fases del ETL

  1. Extraccion: Obtener datos de fuentes heterogeneas (Bases de datos, archivos planos, sistemas legacy, fuentes externas).
  2. Transformacion: Limpiar, filtrar, validar, calcular, agregar, convertir tipos, aplicar reglas de negocio.
  3. Carga: Insertar los datos transformados en el Data Warehouse (full refresh, carga incremental, historico).

Herramientas ETL comunes: Warehouse Manager, SQL Server Integration Services (SSIS), Oracle Warehouse Builder, Pentaho, Talend, Informatica.

7. Tipos de Cubos OLAP

ROLAP vs MOLAP vs HOLAP
Tipo Descripcion Ventajas Desventajas
ROLAP Almacena datos en tablas relacionales. Las agregaciones se calculan sobre la marcha. Escalable, datos en tiempo real, usa SQL estandar. Rendimiento mas lento en consultas complejas.
MOLAP Almacena datos en una estructura multidimensional propietaria (cubo precalculado). Rendimiento excelente, consultas rapidas, indices optimizados. Menos escalable, datos no en tiempo real, espacio adicional.
HOLAP Combina ROLAP y MOLAP: datos detallados en BD relacional, agregaciones en cubo. Balance entre rendimiento y escalabilidad. Mayor complejidad de administracion.

8. Creacion de Dimensiones en Oracle

CREATE DIMENSION

Oracle permite definir dimensiones y jerarquias directamente en el diccionario de datos.

Ejemplo: Dimension Tiempo con Jerarquias

CREATE DIMENSION time_dim LEVEL time_key IS time.time_key LEVEL month IS time.month LEVEL quarter IS time.quarter LEVEL year IS time.year LEVEL week IS time.week LEVEL fiscal_qtr IS time.fiscal_qtr HIERARCHY calendar_rollup ( time_key CHILD OF month CHILD OF quarter CHILD OF year ) HIERARCHY fiscal_rollup ( time_key CHILD OF week ) ATTRIBUTE time_key DETERMINES (day_number_in_month, day_number_in_year) ATTRIBUTE week DETERMINES week_number_of_year ATTRIBUTE month DETERMINES full_month_name;

Ejemplo: Dimension Empleado (desde script de ejemplo)

CREATE DIMENSION DIM_EMP LEVEL employee IS (DIM_EMPLEADO.employee_id) LEVEL job IS (DIM_EMPLEADO.job_id) HIERARCHY job_id ( employee CHILD OF job ) ATTRIBUTE employee DETERMINES (DIM_EMPLEADO.nombreCompleto, DIM_EMPLEADO.salary) ATTRIBUTE job DETERMINES (DIM_EMPLEADO.job_title);

9. Fact Table: Ejemplo RRHH

Creacion de tablas dimension y hecho para un Data Warehouse de RRHH

-- Dimension Empleado CREATE TABLE DIM_EMPLEADO ( employee_id NUMBER(6) PRIMARY KEY, nombreCompleto VARCHAR(50), salary NUMBER(8,2), job_id VARCHAR(10), job_title VARCHAR(35) ); -- Dimension Departamento CREATE TABLE DIM_DEPARTAMENTO ( department_id NUMBER(4) PRIMARY KEY, department_name VARCHAR(30), city_id NUMBER(4), city VARCHAR(30), country_id CHAR(2), country_name VARCHAR(40) ); -- Dimension Tiempo CREATE TABLE DIM_TIEMPO ( tiempo_id NUMBER PRIMARY KEY, numMes NUMBER, numAnio NUMBER ); -- Tabla de Hechos RRHH CREATE TABLE FACT_RRHH ( employee_id NUMBER(6) REFERENCES DIM_EMPLEADO(employee_id), department_id NUMBER(4) REFERENCES DIM_DEPARTAMENTO(department_id), tiempo_id NUMBER REFERENCES DIM_TIEMPO(tiempo_id), CostoTotal NUMBER, TotalEmpleado NUMBER );

10. Índices en Data Warehouse

  • Índices Bitmap: Ideales para columnas con baja cardinalidad (ej. género, estado, región). Reducen sustancialmente el uso de espacio y su mantenimiento es sumamente eficiente en consultas y cargas DML paralelas.
  • Índices B-tree: Ordenan los datos como un árbol inverso, ideales para accesos a consultas típicas de alta cardinalidad.

11. Metodología de Calidad de Datos

Antes de cargar los datos en el Data Warehouse, es vital asegurar su calidad y consistencia mediante una metodología por niveles:

Nivel Tipo de Análisis Descripción
Nivel 0 Análisis de valores de dominio Verificar que los valores de los datos estén en los rangos esperados.
Nivel 1 Integridad & Análisis de validez Comprobar datos faltantes, valores nulos y formatos de datos inválidos.
Nivel 2 Análisis de integridad estructural Garantizar que las relaciones de clave primaria/clave foránea e integridad referencial sean correctas.
Nivel 3 Análisis de reglas de negocios Verificar el cumplimiento de restricciones complejas basadas en lógica de negocio empresarial.
Nivel 4 Reglas de conversión de datos Comprobar y unificar codificaciones heterogéneas (ej. unificar género: F/M, 1/0, femenino/masculino en una sola norma).

12. Explotación de Datos y Minería de Datos (Data Mining)

Una vez estructurado el Data Warehouse, existen tres formas principales para explotar y analizar la información:

  • Consultas Ad-hoc (Query Ad-hoc): Proceso iterativo de preguntas y respuestas formuladas manualmente por expertos para encontrar información valiosa.
  • OLAP (Análisis Multidimensional): Preguntas y respuestas multidimensionales complejas para asistir a los tomadores de decisiones.
  • Data Mining (Minería de Datos): Proceso automático que busca tendencias y patrones en el contenido de la información, permitiendo al usuario descubrir la esencia y relaciones entre sus datos.

Tipos de Minería de Datos

Minería de Descubrimiento

Encuentra patrones en los datos sin una dirección preconcebida. Analiza todo el almacén de datos para descubrir relaciones valiosas de forma simple.

Minería Predictiva

Usa resultados históricos conocidos para construir modelos que permitan predecir comportamientos o valores futuros (requiere control exhaustivo del analista).

Técnicas de Data Mining

Las herramientas modernas aplican diversas técnicas estadísticas y de inteligencia artificial:

Enfoque Analítico Técnicas Algorítmicas
Detección de Patrones y Agrupación Análisis de la Cesta de la Compra (Market Basket Analysis), Segmentación de Clientes, Análisis de Enlaces (Link Analysis), Agrupamiento (Clustering), Clasificación y Predicción.
Modelos de IA y Modelado Numérico Árboles de Decisión (Decision Trees), Redes Neuronales (Neural Networks), Funciones de Base Radial (Radial Basis), Modelado de Ingresos, Series Temporales (Time Series) y Optimización de Búsqueda.

13. Beneficios del Data Warehouse

Desde la perspectiva de TI:

  • Arquitectura flexible de data warehousing que permite cambiar ante nuevas necesidades.
  • Procesos automáticos que actualizan y limpian la data.
  • Menor desgaste del área de IT ante requerimientos ad-hoc de los usuarios.

Desde la perspectiva de usuarios finales:

  • Tener una vista global de los datos de la organización, independientemente de la fuente.
  • Contar con herramientas intuitivas y fáciles de usar.
  • Capacidad de generar reportes analíticos de manera independiente (Self-service BI).
  • Acceso flexible vía intranet o internet.

Importante (Estudio de IDC): Las organizaciones que implementan soluciones analíticas logran retornos de inversión (ROI) que van del 17% al 2000%, con una media de 112% (139% en el área financiera). El 49% logra recuperar el costo en menos de un año, el costo de reportes se reduce un 60% y el tiempo total de generación baja de 10 a 1.

💡 Tips del Profesor

•

Prefiera el esquema Estrella para consultas directas: La desnormalización controlada del modelo Estrella simplifica enormemente la estructura de las consultas para las herramientas de BI (como Tableau o PowerBI) y acelera drásticamente la latencia.

•

Evite actualizaciones frecuentes en el DW: Las bases de datos analíticas están optimizadas para cargas masivas por lotes (batch) y lecturas rápidas. Actualizaciones fila por fila degradan el sistema.

•

Use claves sustitutas (Surrogate Keys): En lugar de usar claves de negocio naturales en el DW, use identificadores secuenciales enteros internos para aislar el almacén de cambios en los sistemas transaccionales de origen.

✍️ Autoevaluación y Ejercicios Prácticos

1 ¿Cuál es la diferencia fundamental entre el esquema Estrella y el Copo de Nieve?

El esquema Estrella mantiene las tablas de dimensiones completamente desnormalizadas en un solo nivel de JOIN con la tabla de hechos. El Copo de Nieve normaliza las dimensiones en múltiples subtablas, reduciendo la redundancia de datos pero agregando JOINs.

2 ¿Qué tipo de datos se registra típicamente en una tabla de Hechos?

Se registran medidas numéricas, cuantitativas e históricas del negocio (ej. monto total vendido, cantidad de productos, tiempo transcurrido) junto con claves externas que apuntan a las dimensiones correspondientes.

3 ¿Por qué las bases de datos analíticas cloud (OLAP) suelen ser de almacenamiento columnar?

Porque las consultas analíticas típicamente necesitan calcular agregados (ej. promedios o sumatorias) sobre un grupo reducido de columnas a lo largo de millones de filas. El formato columnar permite leer del disco solo las columnas necesarias.
