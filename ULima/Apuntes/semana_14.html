Insight del Arquitecto: El DW como Centro de Decisiones

Bill Inmon, padre del Data Warehousing, define el Data Warehouse como "una coleccion de datos orientada a temas, integrada, no volatil y variante en el tiempo, disenada para soportar la toma de decisiones". A diferencia de una base de datos operacional (OLTP) que optimiza transacciones, el DW optimiza consultas analiticas. El 80% del esfuerzo en un proyecto DW esta en el proceso ETL, no en el modelado.

1. Business Intelligence (BI)

Inteligencia de Negocios

Conjunto de productos y servicios que permiten a los usuarios finales acceder y analizar de manera rapida y sencilla la informacion para la toma de decisiones de negocio a nivel operativo, tactico y estrategico.

Componentes clave del BI:

  • Data Warehouse: Repositorio centralizado de datos.
  • Data Mart: Subconjunto departamental (ventas, finanzas, RRHH).
  • Cubos OLAP: Estructura multidimensional para consultas rapidas.
  • ETL: Proceso de extraccion, transformacion y carga.
  • Metadatos: Datos sobre los datos del warehouse.

2. Sistemas OLTP vs OLAP

Caracteristica Sistemas Operacionales (OLTP) Sistemas Analiticos (OLAP)
Proposito Ejecutar el negocio Administrar el negocio
Usuarios Oficinistas, administrativos Decision makers, analistas
Actualizacion Al segundo Periodica (diaria, semanal)
Volumen Altos volumenes de transacciones simples Consultas complejas sobre grandes volumenes
Datos Detallados, actuales Detallados, sumarizados e historicos
Integracion Por aplicacion Integrado y consistente

3. Arquitectura del Data Warehouse

La arquitectura de referencia incluye las siguientes capas:

  • Fuentes de datos: Sistemas operacionales (ERP, CRM), archivos planos, fuentes externas.
  • Extraccion e Integracion: Proceso ETL: extraer, limpiar, transformar, integrar.
  • Almacenamiento: Data Warehouse corporativo + Data Marts departamentales.
  • Aplicaciones: OLAP, reporting, Data Mining, EIS.
  • Metadatos: Repositorio central de definiciones y linaje de datos.

Nota: Existen dos enfoques de implementacion:

  • Top-Down (Inmon): Primero el DW corporativo, luego los Data Marts.
  • Bottom-Up (Kimball): Primero los Data Marts, luego se integran en el DW.

4. Modelo Estrella (Star Schema)

Fact Table + Dimension Tables

El modelo estrella es el esquema mas utilizado en Data Warehousing. Consta de una tabla central de hechos (fact table) rodeada de tablas de dimension (dimension tables).

Componentes:

  • Tabla de Hechos (Fact Table): Almacena eventos de negocio (ventas, transacciones). Contiene medidas numericas (cantidad, monto) y foreign keys a las dimensiones.
  • Tablas de Dimension: Entidades de negocio (clientes, productos, tiempo, geografia). Contienen atributos descriptivos.
  • Medidas (Measures): Valores cuantitativos que miden la efectividad del negocio.

Ejemplo: Esquema Estrella de Ventas

-- Tabla de Dimension: Producto
CREATE TABLE DIM_PRODUCTO (
    producto_key INTEGER PRIMARY KEY,
    id_producto INTEGER,
    nombre VARCHAR(100),
    categoria VARCHAR(50),
    subcategoria VARCHAR(50),
    marca VARCHAR(50),
    precio_actual NUMERIC(10,2)
);

-- Tabla de Dimension: Tiempo
CREATE TABLE DIM_TIEMPO (
    tiempo_key INTEGER PRIMARY KEY,
    fecha DATE,
    dia_semana VARCHAR(20),
    mes INTEGER,
    nombre_mes VARCHAR(20),
    trimestre INTEGER,
    anio INTEGER
);

-- Tabla de Dimension: Cliente
CREATE TABLE DIM_CLIENTE (
    cliente_key INTEGER PRIMARY KEY,
    id_cliente INTEGER,
    nombre VARCHAR(100),
    ciudad VARCHAR(50),
    region VARCHAR(50),
    pais VARCHAR(50),
    segmento VARCHAR(30)
);

-- Tabla de Hechos: Ventas
CREATE TABLE FACT_VENTAS (
    producto_key INTEGER REFERENCES DIM_PRODUCTO,
    cliente_key INTEGER REFERENCES DIM_CLIENTE,
    tiempo_key INTEGER REFERENCES DIM_TIEMPO,
    vendedor_key INTEGER REFERENCES DIM_VENDEDOR,
    cantidad_vendida INTEGER,
    monto_total NUMERIC(12,2),
    costo_total NUMERIC(12,2),
    ganancia NUMERIC(12,2)
);

5. Modelo Copo de Nieve (Snowflake Schema)

En el modelo Snowflake, las dimensiones se normalizan en multiples tablas, una por cada nivel jerarquico. Esto reduce redundancia pero aumenta la complejidad de las consultas.

Caracteristica STAR (Estrella) SNOWFLAKE (Copo de Nieve)
Entendimiento del modelo Sencillo Mayor dificultad
Numero de tablas Menor Mayor
Complejidad de consulta Baja Alta
Rendimiento Rapida Lenta
Redundancia Alta (desnormalizado) Baja (normalizado)
Espacio de almacenamiento Mayor Menor

6. Proceso ETL

Extraccion, Transformacion, Carga

El proceso ETL consume aproximadamente el 80% del tiempo y recursos en un proyecto Data Warehouse (segun Bill Inmon).

Fases del ETL

  1. Extraccion: Obtener datos de fuentes heterogeneas (Bases de datos, archivos planos, sistemas legacy, fuentes externas).
  2. Transformacion: Limpiar, filtrar, validar, calcular, agregar, convertir tipos, aplicar reglas de negocio.
  3. Carga: Insertar los datos transformados en el Data Warehouse (full refresh, carga incremental, historico).

Herramientas ETL comunes: Warehouse Manager, SQL Server Integration Services (SSIS), Oracle Warehouse Builder, Pentaho, Talend, Informatica.

7. Tipos de Cubos OLAP

ROLAP vs MOLAP vs HOLAP
Tipo Descripcion Ventajas Desventajas
ROLAP Almacena datos en tablas relacionales. Las agregaciones se calculan sobre la marcha. Escalable, datos en tiempo real, usa SQL estandar. Rendimiento mas lento en consultas complejas.
MOLAP Almacena datos en una estructura multidimensional propietaria (cubo precalculado). Rendimiento excelente, consultas rapidas, indices optimizados. Menos escalable, datos no en tiempo real, espacio adicional.
HOLAP Combina ROLAP y MOLAP: datos detallados en BD relacional, agregaciones en cubo. Balance entre rendimiento y escalabilidad. Mayor complejidad de administracion.

8. Creacion de Dimensiones en Oracle

CREATE DIMENSION

Oracle permite definir dimensiones y jerarquias directamente en el diccionario de datos.

Ejemplo: Dimension Tiempo con Jerarquias

CREATE DIMENSION time_dim
    LEVEL time_key IS time.time_key
    LEVEL month IS time.month
    LEVEL quarter IS time.quarter
    LEVEL year IS time.year
    LEVEL week IS time.week
    LEVEL fiscal_qtr IS time.fiscal_qtr

    HIERARCHY calendar_rollup (
        time_key CHILD OF
        month CHILD OF
        quarter CHILD OF year
    )

    HIERARCHY fiscal_rollup (
        time_key CHILD OF week
    )

    ATTRIBUTE time_key DETERMINES
        (day_number_in_month, day_number_in_year)
    ATTRIBUTE week DETERMINES week_number_of_year
    ATTRIBUTE month DETERMINES full_month_name;

Ejemplo: Dimension Empleado (desde script de ejemplo)

CREATE DIMENSION DIM_EMP
    LEVEL employee IS (DIM_EMPLEADO.employee_id)
    LEVEL job IS (DIM_EMPLEADO.job_id)

    HIERARCHY job_id (
        employee CHILD OF job
    )

    ATTRIBUTE employee DETERMINES
        (DIM_EMPLEADO.nombreCompleto, DIM_EMPLEADO.salary)
    ATTRIBUTE job DETERMINES
        (DIM_EMPLEADO.job_title);

9. Fact Table: Ejemplo RRHH

Creacion de tablas dimension y hecho para un Data Warehouse de RRHH

-- Dimension Empleado
CREATE TABLE DIM_EMPLEADO (
    employee_id NUMBER(6) PRIMARY KEY,
    nombreCompleto VARCHAR(50),
    salary NUMBER(8,2),
    job_id VARCHAR(10),
    job_title VARCHAR(35)
);

-- Dimension Departamento
CREATE TABLE DIM_DEPARTAMENTO (
    department_id NUMBER(4) PRIMARY KEY,
    department_name VARCHAR(30),
    city_id NUMBER(4),
    city VARCHAR(30),
    country_id CHAR(2),
    country_name VARCHAR(40)
);

-- Dimension Tiempo
CREATE TABLE DIM_TIEMPO (
    tiempo_id NUMBER PRIMARY KEY,
    numMes NUMBER,
    numAnio NUMBER
);

-- Tabla de Hechos RRHH
CREATE TABLE FACT_RRHH (
    employee_id NUMBER(6) REFERENCES DIM_EMPLEADO(employee_id),
    department_id NUMBER(4) REFERENCES DIM_DEPARTAMENTO(department_id),
    tiempo_id NUMBER REFERENCES DIM_TIEMPO(tiempo_id),
    CostoTotal NUMBER,
    TotalEmpleado NUMBER
);

10. Indices en Data Warehouse

  • Indices Bitmap: Ideales para columnas con baja cardinalidad (genero, estado, categoria). Muy eficientes en consultas ad-hoc sobre grandes volumenes. Reducen uso de espacio comparado con B-tree.
  • Indices B-tree: Arbol invertido, optimo para busquedas de alta selectividad. Utilizado para acceso a consultas tipicas con alta cardinalidad.

11. Beneficios del Data Warehouse

Desde la perspectiva de TI:

  • Arquitectura flexible que permite cambios.
  • Actualizacion automatica de datos.
  • Reduccion de carga sobre sistemas operacionales.

Desde la perspectiva de usuarios de negocio:

  • Vista global de los datos de la organizacion.
  • Herramientas faciles de usar.
  • Capacidad de generar reportes independientemente.
  • Acceso via intranet/internet.

Importante: Segun estudios de IDC, las organizaciones que implementan soluciones analiticas logran un ROI promedio de 112%, con periodos de recuperacion de menos de un ano en el 49% de los casos. El costo de generacion de reportes se reduce aproximadamente en un 60%.
