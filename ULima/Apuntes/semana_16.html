Seccion 1: Teoria Avanzada

1. BCNF vs 3FN: 3FN permite dependencias transitivas donde el determinante no es clave candidata. BCNF exige que TODO determinante sea clave candidata. Ejemplo: Tabla Profesor(curso, profesor, oficina) donde un profesor dicta un curso y cada profesor tiene una oficina. Esta en 3FN (atributo no clave depende de otro no clave), pero no en BCNF porque curso -> profesor pero profesor -> oficina y profesor no es clave candidata.

2. Dependencia multivaluada (MVD): Ocurre cuando la presencia de una fila implica la presencia de otras filas independientes. Ejemplo: Empleado(nombre, proyecto, hobby). Un empleado puede tener varios proyectos y varios hobbies independientes. Se genera producto cartesiano. Solucion 4FN: separar en Emp_Proyecto(nombre, proyecto) y Emp_Hobby(nombre, hobby).

3. Desnormalizacion: Agregar redundancia intencional para mejorar rendimiento de lecturas (evitar JOINs). Contexto: Data Warehouses, sistemas OLAP, reportes. Riesgos: inconsistencias, mayor espacio, mantenimiento complejo.

4. OLTP vs OLAP: OLTP = muchas transacciones simples, datos actualizados, usuarios operativos. OLAP = consultas complejas, datos historicos y agregados, usuarios analiticos/toma de decisiones.

5. Star vs Snowflake: Star: desnormalizado, menos tablas, consultas rapidas, mas redundancia. Snowflake: normalizado, mas tablas, consultas lentas, menos redundancia. Star es preferido en DW por rendimiento.

6. ROLAP vs MOLAP vs HOLAP: ROLAP = datos en BD relacional, escalable, lento. MOLAP = datos en cubo propietario, rapido, menos escalable. HOLAP = hibrido (detalle relacional, agregaciones en cubo). MOLAP recomendado cuando el rendimiento de consultas es critico y los datos cambian poco.

7. Modos Standby: HOT = Mirroring/AlwaysOn (failover automatico, consistencia transaccional). WARM = Log Shipping (failover manual, posible perdida de datos). COLD = Backup/Restore (requiere restauracion completa, lento).

8. 5FN (PJ/NF): Una tabla esta en 5FN si no puede descomponerse en tablas mas pequenas sin perder informacion al reconstruir con JOINs. Esoterica porque pocos casos practicos la requieren. Relevante en sistemas con restricciones de join complejas y Big Data.

9. RPO vs RTO: RPO = perdida maxima de datos aceptable. RTO = tiempo maximo de recuperacion. Mirroring: RPO bajo (segundos o cero), RTO bajo (segundos). Log Shipping: RPO alto (minutos-horas), RTO alto (horas-dias).

10. Indice Bitmap: Usa bits para representar valores de columnas con baja cardinalidad. Mas eficiente en espacio y velocidad para consultas ad-hoc en DW. B-tree es mejor para alta cardinalidad y busquedas puntuales.

Seccion 2: PL/SQL

Ejercicio 1 - Procedimiento:

CREATE OR REPLACE PROCEDURE obtener_estadisticas_empleado(
    p_dept_id IN NUMBER,
    p_total OUT NUMBER,
    p_promedio OUT NUMBER,
    p_maximo OUT NUMBER
)
IS
BEGIN
    SELECT COUNT(*), AVG(salario), MAX(salario)
    INTO p_total, p_promedio, p_maximo
    FROM empleado
    WHERE departamento_id = p_dept_id;
END obtener_estadisticas_empleado;

Ejercicio 2 - Funcion:

CREATE OR REPLACE FUNCTION categoria_cliente(
    p_cliente_id IN NUMBER
)
RETURN VARCHAR2
IS
    v_total_compras NUMBER;
BEGIN
    SELECT COALESCE(SUM(p.total), 0)
    INTO v_total_compras
    FROM pedido p
    WHERE p.cliente_id = p_cliente_id;

    IF v_total_compras > 10000 THEN
        RETURN 'VIP';
    ELSIF v_total_compras >= 1000 THEN
        RETURN 'Regular';
    ELSE
        RETURN 'Basico';
    END IF;
END categoria_cliente;

Ejercicio 3 - Trigger:

CREATE OR REPLACE TRIGGER trg_auditar_precio
    BEFORE UPDATE OF precio ON producto
    FOR EACH ROW
BEGIN
    IF :NEW.precio > :OLD.precio * 1.5 THEN
        raise_application_error(-20001,
            'El incremento de precio no puede exceder el 50%');
    END IF;
END trg_auditar_precio;

Seccion 3: Consultas

1. Top 5 clientes por gasto:

SELECT c.id, c.nombre, c.email,
       COUNT(DISTINCT p.id) AS total_pedidos,
       COALESCE(SUM(p.total), 0) AS gasto_total,
       COALESCE(AVG(p.total), 0) AS promedio_pedido
FROM cliente c
INNER JOIN pedido p ON c.id = p.cliente_id
WHERE c.activo = true
GROUP BY c.id, c.nombre, c.email
ORDER BY gasto_total DESC
LIMIT 5;

2. Ventas por categoria:

SELECT pr.categoria,
       SUM(dp.cantidad) AS cantidad_vendida,
       SUM(dp.cantidad * dp.precio_unitario) AS ingreso_total,
       (SELECT dp2.producto_id
        FROM detalle_pedido dp2
        INNER JOIN producto pr2 ON dp2.producto_id = pr2.id
        WHERE pr2.categoria = pr.categoria
        GROUP BY dp2.producto_id
        ORDER BY SUM(dp2.cantidad) DESC
        LIMIT 1) AS producto_mas_vendido
FROM producto pr
INNER JOIN detalle_pedido dp ON pr.id = dp.producto_id
GROUP BY pr.categoria;

3. Subconsulta correlacionada:

SELECT c.id, c.nombre, c.ciudad,
       SUM(p.total) AS gasto_total
FROM cliente c
INNER JOIN pedido p ON c.id = p.cliente_id
GROUP BY c.id, c.nombre, c.ciudad
HAVING SUM(p.total) > (
    SELECT AVG(gasto_ciudad)
    FROM (
        SELECT SUM(p2.total) AS gasto_ciudad
        FROM cliente c2
        INNER JOIN pedido p2 ON c2.id = p2.cliente_id
        WHERE c2.ciudad = c.ciudad
        GROUP BY c2.id
    ) sub
);

4. Mes con mayores ventas por ano:

WITH ventas_mensuales AS (
    SELECT EXTRACT(YEAR FROM p.fecha) AS anio,
           EXTRACT(MONTH FROM p.fecha) AS mes,
           SUM(p.total) AS total_ventas,
           ROW_NUMBER() OVER (
               PARTITION BY EXTRACT(YEAR FROM p.fecha)
               ORDER BY SUM(p.total) DESC
           ) AS rn
    FROM pedido p
    GROUP BY EXTRACT(YEAR FROM p.fecha),
             EXTRACT(MONTH FROM p.fecha)
)
SELECT anio, mes, total_ventas
FROM ventas_mensuales
WHERE rn = 1
ORDER BY anio;

5. Productos nunca vendidos:

SELECT pr.id, pr.nombre, pr.categoria
FROM producto pr
WHERE NOT EXISTS (
    SELECT 1
    FROM detalle_pedido dp
    WHERE dp.producto_id = pr.id
);

-- Tablas de Dimension
CREATE TABLE DIM_TIENDA (
    tienda_key INTEGER PRIMARY KEY,
    id_tienda INTEGER,
    nombre VARCHAR(100),
    direccion VARCHAR(200),
    ciudad VARCHAR(50),
    region VARCHAR(50)
);

CREATE TABLE DIM_PRODUCTO (
    producto_key INTEGER PRIMARY KEY,
    id_producto INTEGER,
    nombre VARCHAR(100),
    categoria VARCHAR(50),
    subcategoria VARCHAR(50),
    precio_actual NUMERIC(10,2)
);

CREATE TABLE DIM_TIEMPO (
    tiempo_key INTEGER PRIMARY KEY,
    fecha DATE,
    dia_semana VARCHAR(20),
    mes INTEGER,
    nombre_mes VARCHAR(20),
    trimestre INTEGER,
    anio INTEGER
);

CREATE TABLE DIM_VENDEDOR (
    vendedor_key INTEGER PRIMARY KEY,
    id_vendedor INTEGER,
    nombre VARCHAR(100),
    tienda_id INTEGER,
    fecha_contratacion DATE
);

-- Tabla de Hechos
CREATE TABLE FACT_VENTAS (
    tienda_key INTEGER REFERENCES DIM_TIENDA(tienda_key),
    producto_key INTEGER REFERENCES DIM_PRODUCTO(producto_key),
    tiempo_key INTEGER REFERENCES DIM_TIEMPO(tiempo_key),
    vendedor_key INTEGER REFERENCES DIM_VENDEDOR(vendedor_key),
    unidades_vendidas INTEGER,
    monto_total NUMERIC(12,2),
    descuento NUMERIC(12,2),
    ganancia_neta NUMERIC(12,2)
);
