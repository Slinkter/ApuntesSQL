?? Insight del Arquitecto: Más Allá de 3FN

¿Realmente necesitas BCNF? En la práctica, la mayoría de las bases de datos OLTP se diseñan hasta 3FN. Las formas normales superiores (BCNF, 4FN, 5FN) son más relevantes en sistemas especializados. El foco debe estar en entender cuándo agregar redundancia intencional (desnormalización) para mejorar rendimiento, especialmente en Data Warehouses.

1. Forma Normal Boyce-Codd (BCNF)

Definición: Una relación está en BCNF si todo determinante es una clave candidata.

Ejemplo: Estudiantes y Carreras



-- Tabla: Notas

-- PK: (código_estudiante, carrera)

-- Cada estudiante puede tener varias carreras

-- Cada carrera tiene un solo asesor

| código | carrera   | asesor  | nota |

|--------|-----------|---------|------|

| 20041  | Sistemas  | López   | 16   |

| 20041  | Contab.  | Salas   | 16   |

| 20052  | Psicología| Pérez   | 13   |

| 20051  | Contab.  | Rojas   | 12   |

-- Problema: Asesor determina carrera (cada asesor es único de una carrera)

-- Pero Asesor NO es clave primaria

-- Solución en BCNF:

Tabla1: Estudiante_Carrera(código, carrera, nota)

Tabla2: Asesor(carrera, asesor)

Caso donde BCNF ? 3FN:

  • Cuando hay múltiples claves candidatas
  • Y existe un determinante que no es clave primaria

2. Cuarta Forma Normal (4FN)

Definición: Una relación está en 4FN si está en BCNF y no contiene dependencias multivaluadas.

Dependencia Multivaluada:

Existe cuando para cada valor de A, hay un conjunto de valores de B Y un conjunto de valores de C, pero B y C son independientes entre sí.

Ejemplo: Cursos, Profesores, Textos



-- Tabla: Curso_Profesor_Texto (no en 4FN)

-- Un curso tiene múltiples profesores Y múltiples textos

| curso       | profesor | texto    |

|-------------|----------|----------|

| Admin       | García   | Drucker  |

| Admin       | García   | Peters   |

| Admin       | Arce     | Drucker  |

| Admin       | Arce     | Peters   |

| Admin       | Pérez    | Drucker  |

| Admin       | Pérez    | Peters   |

| Finanzas    | Méndez   | Jones    |

| Finanzas    | Méndez   | Chang    |

-- Si agregamos un profesor, debemos agregar TODAS las combinaciones con textos

-- Los profesores y textos son independientes!

-- Solución en 4FN:

Curso_Profesor(curso, profesor)

Curso_Texto(curso, texto)

3. Quinta Forma Normal (5FN)

Definición: Una relación está en 5FN si está en 4FN y no contiene dependencias de group/join (también llamadas dependencias de proyección-union).

Dependencia de Join:

Una tabla puede ser reconstruida unionizando tres o más tablas más pequeñas.

Ejemplo: Vendedor-Marca-Producto



-- Tabla original: Vendedor_Marca_Producto

| vendedor | marca   | producto |

|----------|---------|----------|

| Pérez    | Toyota  | Auto     |

| Pérez    | Toyota  | Camioneta|

| Pérez    | Nissan  | Auto     |

| Pérez    | Nissan  | Camioneta|

| Rojas    | Toyota  | Auto     |

-- Parece que se puede separar en 3 tablas...

-- Pero si las unis obtienes la original, está en 5FN

-- Si NO puedes reconstructir correctamente, NO está en 5FN

-- Solución (si aplica):

Vendedor_Marca(vendedor, marca)

Marca_Producto(marca, producto)

Vendedor_Producto(vendedor, producto)

4. Resumen de Formas Normales

F.N.Problema que resuelveCondición
1FNGrupos repetitivosValores atómicos
2FNDependencias parcialesEn 1FN + sin dep. parciales
3FNDependencias transitivasEn 2FN + sin dep. transitivas
BCNFDeterminantes no claveEn 3FN + todo determinante es clave
4FNDependencias multivaluadasEn BCNF + sin dep. multivaluadas
5FNDependencias de joinEn 4FN + sin dep. de join

5. Desnormalización

Desnormalización: Proceso de crear redundancia controlada para mejorar rendimiento de consultas.

Cuándo Desnormalizar:

  • Sistemas OLAP (Data Warehouse)
  • Lecturas frecuentes, escrituras pocas
  • Consultas complejas con muchos JOINs
  • Cuando la normalización causa problemas de rendimiento

Técnicas de Desnormalización:

TécnicaDescripción
Valores DerivadosGuardar cálculos (total, subtotal)
Unir EntidadesCombinar tablas relacionadas
Valores FijosHard code de catálogos pequeños
Repetir AtributosCopiar datos de detalle a maestro
Llaves de Corto CircuitoAgregar FK directa para evitar JOIN

Ejemplo de Desnormalización:



-- Normalizado:

Factura(id, id_cliente, fecha, total)

Detalle(id_factura, id_producto, cantidad, precio_unitario)

-- Desnormalizado (agregando campo derivado):

Factura(id, id_cliente, fecha, total, nombre_cliente)

-- Se evita un JOIN para mostrar el nombre del cliente

-- Con precaución:

-- Necesitas disparadores o aplicación para mantener consistencia

-- Si cambia el nombre del cliente, actualizar TODAS las facturas

?? Trade-off: Desnormalización mejora lectura pero complica escritura. Usa solo cuando el beneficio de rendimiento supera el costo de mantener consistencia.
