Insight del Arquitecto: HA no es solo tecnologia

La alta disponibilidad no es solo una decision tecnica -- es un compromiso de negocio. Cada "9" de disponibilidad cuesta exponencialmente mas. Pasar de 99.9% a 99.99% puede duplicar o triplicar el costo de infraestructura. La clave es alinear el SLA con el presupuesto y el impacto real del negocio. No todas las bases de datos necesitan 5 nueves.

1. Conceptos Generales de HA

Alta Disponibilidad

La alta disponibilidad es una solucion para minimizar o mitigar el impacto de paradas de sistemas. Idealmente, un sistema de base de datos deberia estar disponible las 24 horas del dia, cada dia. Implica redundancia de algun tipo y tecnicas para incrementar la disponibilidad.

Acuerdos de Nivel de Servicio (SLA)

Numero de 9s % Disponibilidad Tiempo de parada anual
2 99% 3 dias y 15 horas
3 99.9% 8 horas y 45 minutos
4 99.99% 52 minutos y 34 segundos
5 99.999% 5 minutos y 15 segundos

Metricas Clave

  • RPO (Recovery Point Objective): Cantidad maxima de datos que se puede perder (medido en tiempo). Ej: RPO de 5 minutos = se pierden hasta 5 minutos de datos.
  • RTO (Recovery Time Objective): Tiempo maximo para recuperar el servicio tras una falla. Ej: RTO de 1 hora = el sistema debe estar operativo en 1 hora.

2. Modos de Espera (Standby Modes)

Modo Descripcion Failover
HOT Standby El nodo secundario provee una copia transaccionalmente consistente de los datos del servidor primario. Deteccion automatica de errores y recuperacion. Automatico
WARM Standby Se mantiene una copia de los datos en el secundario, pero los datos pueden o no ser confirmados concurrentemente. El error y recuperacion puede no ser automatico. Manual
COLD Standby Servidor donde se pueden restaurar los datos. Se necesita OS apropiado, software y copia de seguridad. Puede llevar un tiempo considerable. Manual (lento)

4. Trasvase de Registro (Log Shipping)

Warm Standby

El Log Shipping es una tecnologia madura y estable que permite mantener una base de datos secundaria actualizada mediante la copia y restauracion periodica de los archivos de registro de transacciones de la base de datos primaria.

Funcionamiento:

  1. En el servidor primario se realizan copias de seguridad del log de transacciones.
  2. Los archivos de backup se copian a una carpeta compartida accesible por el secundario.
  3. En el servidor secundario, los archivos se restauran (con NORECOVERY) para mantener la base sincronizada.

Fortalezas:

  • Maduro, estable y simple de configurar.
  • No requiere hardware especial.
  • Multiples copias de respaldo (ilimitadas).
  • Las bases secundarias pueden consultarse (solo lectura) entre restauraciones.
  • Alertas configuradas si el trasvase no se completa en un umbral de tiempo (ej. LSAlert).

Debilidades:

  • Fallo (failover) de cliente manual.
  • Configuracion por base de datos (no protege bases de datos del sistema).
  • Perdida de datos potencial (RPO de minutos a horas).

Pasos de Configuración de Trasvase de Registro

La implementación del Trasvase de Registro consta de seis pasos esenciales realizados a través de SSMS o scripts:

  1. Habilitar Trasvase: Habilitar la base de datos principal en las propiedades del servidor primario (modo de recuperación completo/FULL).
  2. Configurar Copia de Seguridad (LSBackup): Indicar la carpeta local para los backups y una ruta de red compartida para que el secundario acceda. Conceder permisos de lectura/escritura a la cuenta del servicio SQL Server y del Agente. Definir la frecuencia (por defecto cada 15 minutos).
  3. Agregar Instancia Secundaria: Conectarse a la instancia que alojará la base de datos secundaria.
  4. Inicializar Base de Datos Secundaria: Generar un backup completo del principal y restaurarlo en el secundario (dejar la base en estado de restauración).
  5. Configurar Copia de Archivos (LSCopy): Definir la carpeta local en el servidor secundario donde se descargarán los archivos del log y configurar la retención (ej. 72 horas).
  6. Configurar Restauración (LSRestore): Establecer si la base se mantiene en modo sin recuperación (NORECOVERY) o en modo de espera (STANDBY / Read-Only, permitiendo desconectar usuarios durante la restauración).

Monitoreo y Conmutación por Código T-SQL

-- En el servidor primario: monitoreo
EXEC sp_help_log_shipping_monitor_primary
    'perdbprod01', 'adventureworks2012';
EXEC sp_help_log_shipping_primary_database
    'adventureworks2012';

-- En el servidor secundario: monitoreo
EXEC sp_help_log_shipping_monitor_secondary
    'perdbprod01\SQLAD', 'adventureworks2012';
EXEC sp_help_log_shipping_secondary_database
    'adventureworks2012';

-- Conmutación (Failover) manual (en caso de falla del primario)
RESTORE DATABASE AdventureWorks2012 WITH RECOVERY;

5. Base de Datos Reflejada (Database Mirroring)

Hot Standby

El Database Mirroring es una solucion de software para incrementar la disponibilidad de la base de datos. Mantiene una copia actualizada (reflejo) en un servidor aparte para conmutacion en caso de falla del servidor principal.

Roles de los servidores:

  • Servidor Principal: Base de datos de produccion. Los usuarios y aplicaciones se conectan normalmente.
  • Servidor Reflejo: Contiene la base de datos reflejo. Toma el control si el principal falla.
  • Servidor Testigo (opcional): Monitorea ambos servidores y permite failover automatico.

Modos de Operacion

Modo Sincronizacion Perdida de datos Failover
Alta Disponibilidad Sincronizado (con testigo) Cero Automatico
Alta Proteccion Sincronizado (sin testigo) Cero Manual
Alto Rendimiento Asincrono Posible (segundos) Manual

Pasos de Configuración de Reflejo de Base de Datos

La configuración de una sesión de creación de reflejo (mirroring) requiere un orden estricto:

  1. Sincronizar la Base de Datos: Configurar la base de datos principal en modo de recuperación completo (FULL). Realizar un backup completo y de registro de transacciones, y restaurarlos en el servidor reflejo utilizando la opción WITH NORECOVERY (dejar la base en estado de restauración).
  2. Propiedades de Reflejo: En SSMS, ir a las propiedades de la base de datos principal y seleccionar la pestaña "Mirroring" (Creación de reflejo) y pulsar en "Configurar seguridad...".
  3. Definir Servidor Testigo (Witness): Indicar si la sesión incluirá una instancia testigo (necesaria para conmutación por error automática en modo síncrono).
  4. Configurar Puertos y Extremos (Endpoints): Definir el puerto de escucha y el nombre del extremo en cada servidor. Por defecto se usa el puerto 5022 para el Principal, 5023 para el Reflejado, y 5024 para el Testigo (extremos de tipo TCP, habilitar en Firewall).
  5. Definir Cuentas de Servicio: Asignar las cuentas de dominio bajo las cuales corren las instancias. Esto configurará automáticamente los permisos GRANT CONNECT ON ENDPOINT correspondientes.
  6. Establecer el Modo de Funcionamiento: Elegir entre Alto Rendimiento (asíncrono), Seguridad Alta sin conmutación automática (síncrono sin testigo), o Seguridad Alta con conmutación automática (síncrono con testigo).
  7. Iniciar la Sesión: Pulsar "Iniciar creación de reflejo" para activar la sincronización.

Comandos de administración (T-SQL):

-- Conmutacion manual (modo sincronizado)
ALTER DATABASE AdventureWorks2012 SET PARTNER FAILOVER;

-- Conmutacion en falla (modo sincronizado, sin testigo)
ALTER DATABASE AdventureWorks2012 SET PARTNER OFF;
RESTORE DATABASE AdventureWorks2012 WITH RECOVERY;

-- Forzar servicio con perdida de datos (modo asincrono)
ALTER DATABASE AdventureWorks2012
    SET PARTNER FORCE_SERVICE_ALLOW_DATA_LOSS;
RESTORE DATABASE AdventureWorks2012 WITH RECOVERY;

-- Suspender sesion
ALTER DATABASE AdventureWorks2012 SET PARTNER SUSPEND;

-- Reanudar sesion
ALTER DATABASE AdventureWorks2012 SET PARTNER RESUME;

Nota importante: Database Mirroring fue descontinuado en versiones posteriores de SQL Server en favor de AlwaysOn Availability Groups. Sin embargo, su estudio es fundamental para entender la evolucion de las tecnologias HA.

6. Failover Clustering

Proteccion a nivel de instancia

Failover Clustering protege contra fallos a nivel de servidor (instancia completa). Un cluster es un grupo de servidores independientes que trabajan juntos para aumentar la disponibilidad. Si un nodo falla, otro nodo asume automaticamente la carga de trabajo.

  • Protege la instancia completa de SQL Server, no solo una base de datos.
  • Requiere almacenamiento compartido (SAN).
  • Failover automatico en segundos.
  • No hay perdida de datos (disco compartido).
  • Complejidad media de implementacion.

7. AlwaysOn Availability Groups

SQL Server 2012+

AlwaysOn es la evolucion de las tecnologias HA en SQL Server 2012. Combina lo mejor de Failover Clustering y Database Mirroring, permitiendo hasta 4 replicas secundarias con capacidad de lectura.

  • Grupo de bases de datos como unidad de failover.
  • Hasta 4 replicas secundarias (leibles).
  • Soporta modos sincrono y asincrono.
  • Failover automatico con hasta 2 replicas sincronas.
  • Requiere Windows Server Failover Cluster (WSFC).

8. Interoperabilidad entre Tecnologias

Log Shipping + Mirroring: La base de datos principal en una sesion de mirroring puede actuar tambien como base principal en una configuracion de log shipping. Las bases secundarias de log shipping deben residir en instancias distintas del servidor principal o reflejado.

Log Shipping + Replicacion: Pueden usarse conjuntamente, pero la replicacion no continua tras un failover de log shipping. En replicacion transaccional, el comportamiento depende de la opcion sync with backup.
