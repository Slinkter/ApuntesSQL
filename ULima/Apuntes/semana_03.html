Insight del Arquitecto: El Fundamento Relacional

El modelo relacional de Codd (1970): La genialidad de Codd fue eliminar los punteros fisicos. En los modelos jerarquico y de red, los datos se navegaban mediante punteros. El modelo relacional usa SOLO valores de atributos para establecer relaciones, lo que le da una elegancia matematica basada en teoria de conjuntos y logica de predicados.

Implicancia practica: Puedes cambiar completamente la estructura fisica de almacenamiento (de HDD a SSD, de fila a columnar) sin cambiar las consultas SQL. Eso es independencia de datos en accion.

1. Base de Datos Relacional (BDR)

Definicion

Base de Datos Relacional: Es una coleccion de relaciones o tablas de dos dimensiones para almacenar informacion. Puede ser accedida y modificada por sentencias del Lenguaje Estructurado de Consultas (SQL). Contiene una coleccion de tablas sin punteros fisicos y utiliza un conjunto de operadores relacionales.

2. Propiedades de las BDR

  • Acceso y modificacion mediante SQL (Structure Query Language).
  • Coleccion de tablas sin punteros fisicos — las relaciones se establecen por valores de atributos.
  • Utiliza un conjunto de operadores relacionales (seleccion, proyeccion, union, join, etc.).
  • Cada tabla tiene filas (tuplas) y columnas (atributos).
  • Se puede manipular datos de las filas usando comandos SQL.

  • Relacion (Tabla): Estructura bidimensional que almacena datos de una entidad.
  • Tupla (Fila): Cada registro individual. Ej: la fila de PEDRO.
  • Atributo (Columna): Caracteristica de la entidad. Ej: Nombre, Tipo, ZIP.
  • Primary Key (PK): Columna o combinacion que identifica de manera unica cada tupla. Subrayada en el diagrama.
  • Foreign Key (FK): Columna que referencia la PK de otra tabla. Implementa la asociacion entre tablas.

5. Reglas de Nomenclatura de Objetos

Reglas estandar ANSI para nombrar objetos en una BD:

  • Debe comenzar con una letra.
  • Puede tener una longitud de 1 a 30 caracteres.
  • Debe contener solamente los caracteres A-Z, a-z, 0-9, _, $ y #.
  • No debe ser igual al nombre de otro objeto dentro del mismo usuario del DBMS.
  • No debe ser una palabra reservada del DBMS.

7. Restricciones de la Base de Datos

Integridad y Validacion

7.1 Restricciones de Integridad

  • Integridad de Entidad (PRIMARY KEY): El Primary Key no puede ser NULL y su valor debe ser unico. Solo se permite una PK por tabla. Garantiza que no hayan filas duplicadas. Automaticamente crea un indice UNIQUE.
  • Integridad Referencial (FOREIGN KEY): Los valores de las Foreign Key deben relacionarse con una llave primaria (PK). Establece una relacion entre la PK en la misma tabla o con otras tablas. Debe existir un valor en la tabla Padre a donde hace referencia la FK.

Palabras clave de FOREIGN KEY:

  • FOREIGN KEY — En la tabla hijo se define que columna tiene la restriccion.
  • REFERENCES — Identifica a la tabla padre y la columna PK.
  • ON DELETE CASCADE — Elimina automaticamente filas hijo si se elimina la fila padre.

7.2 Restricciones de Columna (Dominio)

Restriccion Descripcion
NULL / NOT NULL Especifica si el atributo puede tener valores nulos
UNIQUE Todos los valores en la columna deben ser distintos
CHECK Condicion booleana que debe cumplir cada valor ingresado

Los valores que ingresan a una columna deben ser del mismo tipo que la definicion del dato (restriccion de dominio por tipo de dato).

7.3 Restricciones Definidas por el Usuario

Los valores de las columnas deben tener validaciones de acuerdo con las reglas de negocio. Por ejemplo: "el salario no puede ser menor al minimo legal", "la fecha de atencion debe ser posterior a la fecha de orden".

Resumen de restricciones: Las restricciones de integridad mantienen la correccion de los datos (PK no nula, FK valida). Las restricciones de columna controlan los valores permitidos (NULL, UNIQUE, CHECK). Las restricciones definidas por el usuario implementan reglas de negocio especificas.
