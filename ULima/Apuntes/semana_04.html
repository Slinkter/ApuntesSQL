DDL Relacional Puro: Restricciones y Secuencias

El Lenguaje de Definición de Datos (DDL) es la formalización del esquema conceptual. Permite estructurar la base de datos mediante:

  • CHECK: Valida que los valores insertados sigan reglas relacionales matemáticas (ej. precio > 0).
  • DEFAULT: Garantiza valores por defecto si no son provistos en el DML.
  • Secuencias e Identidades: El uso de GENERATED BY DEFAULT AS IDENTITY (estándar SQL:2003) se prefiere sobre SERIAL porque es compatible con el estándar ANSI y evita problemas de permisos de secuencia ocultos.

Migraciones DDL Zero-Downtime en la Nube

En bases de datos RDS/Aurora de producción con millones de registros, ejecutar ciertos DDLs (como ALTER TABLE ADD COLUMN con valores por defecto no nulos o CREATE INDEX) puede bloquear las operaciones de lectura/escritura de la tabla por largos períodos de tiempo.

Solución Cloud: Use CREATE INDEX CONCURRENTLY en PostgreSQL para crear índices sin bloquear escrituras, o herramientas de migración en línea como pg_repack y gh-ost para reconstruir esquemas de forma segura.

Objetivos de Aprendizaje

  • Crear tablas con tipos de datos apropiados y restricciones
  • Definir PRIMARY KEY y FOREIGN KEY en la creacion de tablas
  • Modificar tablas existentes con ALTER TABLE
  • Eliminar tablas correctamente usando DROP TABLE
  • Comprender el modelo de datos de la muebleria como caso de estudio

Insight del Arquitecto: DDL Transaccional en PostgreSQL

Una diferencia crucial: En Oracle, los comandos DDL hacen COMMIT implicito. En PostgreSQL, los comandos DDL son transaccionales — puedes hacer ROLLBACK de un CREATE TABLE, ALTER TABLE o incluso DROP TABLE si esta dentro de una transaccion. Esto cambia dramaticamente la estrategia de migraciones.

Consejo practico: Siempre envuelve cambios estructurales en transacciones explicitas (BEGIN; ... COMMIT;) y prueba con ROLLBACK en desarrollo antes de ejecutar en produccion.

1. Creacion de Tablas (CREATE TABLE)

DDL basico

Sintaxis General

CREATE TABLE nombre_tabla
    (columna1 tipo_dato [restriccion_columna],
     columna2 tipo_dato [restriccion_columna],
     ...
     [restricciones_de_tabla]);

Ejemplo 1: Restriccion a nivel de columna

CREATE TABLE orden_pedido
    (numero_orden NUMBER(7) PRIMARY KEY,
     fecha_colocacion DATE,
     fecha_llenado DATE,
     numero_embarques NUMBER(5),
     id_cliente NUMBER(7) REFERENCES cliente );

En este ejemplo, PRIMARY KEY y REFERENCES se definen directamente sobre la columna. Es una sintaxis compacta pero menos flexible para claves compuestas.

Ejemplo 2: Restricciones a nivel de tabla

CREATE TABLE orden_pedido
    (numero_orden NUMBER(7),
     fecha_colocacion DATE,
     fecha_llenado DATE,
     numero_embarques NUMBER(5),
     id_cliente NUMBER(7),
     PRIMARY KEY (numero_orden),
     FOREIGN KEY (id_cliente) REFERENCES cliente);

Diferencia clave: La sintaxis a nivel de tabla permite definir claves compuestas (PK con multiples columnas) y nombrar las restricciones. Es la forma recomendada para esquemas complejos.

2. Modelo de Datos de una Muebleria

Caso de estudio

El siguiente es el modelo relacional para un sistema de muebleria que usaremos como caso de estudio en DDL:

-- Entidades principales del modelo
cliente:      id_cliente, nombre, direccion, tipo, zip
orden_pedido: numero_orden (PK), id_cliente (FK),
              fecha_colocacion, fecha_llenado,
              numero_embarques
producto:     id_producto (PK), nombre_linea (FK),
              meta_venta, stock, precio, costo
linea_producto: nombre_linea (PK)
producto_ordenado: numero_orden (FK), id_producto (FK),
                   cantidad_ordenada

Nota: Este modelo usa nomenclatura Oracle (NUMBER, DATE, VARCHAR2). En los ejemplos siguientes se presenta la sintaxis original del material academico y luego su equivalente en PostgreSQL.

3. Creacion del Esquema Muebleria

Sintaxis Oracle (original del curso)

CREATE TABLE cliente (
    id_cliente NUMBER(7) PRIMARY KEY,
    nombre VARCHAR2(50) NOT NULL,
    direccion VARCHAR2(100),
    tipo CHAR(1) CHECK (tipo IN ('A','B','C')),
    zip VARCHAR2(5)
);

CREATE TABLE linea_producto (
    nombre_linea VARCHAR2(30) PRIMARY KEY
);

CREATE TABLE producto (
    id_producto NUMBER(7) PRIMARY KEY,
    nombre_linea VARCHAR2(30) REFERENCES linea_producto,
    meta_venta NUMBER(10,2),
    stock NUMBER(5) DEFAULT 0,
    precio NUMBER(10,2) CHECK (precio > 0),
    costo NUMBER(10,2)
);

CREATE TABLE orden_pedido (
    numero_orden NUMBER(7) PRIMARY KEY,
    fecha_colocacion DATE DEFAULT SYSDATE,
    fecha_llenado DATE,
    numero_embarques NUMBER(5),
    id_cliente NUMBER(7) NOT NULL REFERENCES cliente
);

CREATE TABLE producto_ordenado (
    numero_orden NUMBER(7),
    id_producto NUMBER(7),
    cantidad_ordenada NUMBER(5) CHECK (cantidad_ordenada > 0),
    PRIMARY KEY (numero_orden, id_producto),
    FOREIGN KEY (numero_orden) REFERENCES orden_pedido
        ON DELETE CASCADE,
    FOREIGN KEY (id_producto) REFERENCES producto
);

Equivalente en PostgreSQL

CREATE TABLE cliente (
    id_cliente SERIAL PRIMARY KEY,
    nombre VARCHAR(50) NOT NULL,
    direccion VARCHAR(100),
    tipo CHAR(1) CHECK (tipo IN ('A','B','C')),
    zip VARCHAR(5)
);

CREATE TABLE linea_producto (
    nombre_linea VARCHAR(30) PRIMARY KEY
);

CREATE TABLE producto (
    id_producto SERIAL PRIMARY KEY,
    nombre_linea VARCHAR(30) REFERENCES linea_producto,
    meta_venta NUMERIC(10,2),
    stock INTEGER DEFAULT 0,
    precio NUMERIC(10,2) CHECK (precio > 0),
    costo NUMERIC(10,2)
);

CREATE TABLE orden_pedido (
    numero_orden SERIAL PRIMARY KEY,
    fecha_colocacion DATE DEFAULT CURRENT_DATE,
    fecha_llenado DATE,
    numero_embarques INTEGER,
    id_cliente INTEGER NOT NULL REFERENCES cliente
);

CREATE TABLE producto_ordenado (
    numero_orden INTEGER,
    id_producto INTEGER,
    cantidad_ordenada INTEGER CHECK (cantidad_ordenada > 0),
    PRIMARY KEY (numero_orden, id_producto),
    FOREIGN KEY (numero_orden) REFERENCES orden_pedido
        ON DELETE CASCADE,
    FOREIGN KEY (id_producto) REFERENCES producto
);

4. Alteracion de Tablas (ALTER TABLE)

Modificacion de estructura

ALTER TABLE permite modificar la estructura de una tabla existente sin perder los datos:

Operacion Sintaxis
Agregar columna ALTER TABLE tabla ADD (columna tipo [DEFAULT expr] [restriccion]);
Modificar columna ALTER TABLE tabla MODIFY (columna NUEVO_tipo [DEFAULT expr] [restriccion]);
Eliminar columna ALTER TABLE tabla DROP COLUMN columna;

Ejemplos

-- Agregar una columna
ALTER TABLE orden_pedido
    ADD nombre_vendedor VARCHAR2(15);

-- Modificar el tipo de dato de una columna existente
ALTER TABLE orden_pedido
    MODIFY numero_embarques NUMBER(6);

-- Eliminar una columna
ALTER TABLE orden_pedido
    DROP COLUMN numero_embarques;

Advertencia: En Oracle, MODIFY puede fallar si la tabla tiene datos que no cumplen con el nuevo tipo. En PostgreSQL, ALTER COLUMN ... TYPE requiere una clausula USING para convertir datos existentes.

5. Eliminacion de Tablas (DROP TABLE)

Eliminacion de objetos

Sintaxis: DROP TABLE tabla [CASCADE CONSTRAINT];

  • Elimina la definicion de la tabla y todos sus datos.
  • La opcion CASCADE CONSTRAINT elimina todas las restricciones de integridad referencial que dependen de la tabla (FK de otras tablas que la referencian).
  • Si otras tablas tienen FK que referencian a esta tabla y no se usa CASCADE, el DROP fallara.

Ejemplo

-- Eliminar la tabla (falla si hay dependencias)
DROP TABLE orden_pedido;

-- Eliminar la tabla con todas sus dependencias
DROP TABLE orden_pedido CASCADE CONSTRAINTS;

-- En PostgreSQL equivalente:
DROP TABLE orden_pedido CASCADE;

Diferencia importante: En Oracle se usa CASCADE CONSTRAINTS (plural). En PostgreSQL es simplemente CASCADE. Ademas, PostgreSQL soporta DROP TABLE IF EXISTS para evitar errores si la tabla no existe.

6. Resumen de DDL en PostgreSQL vs Oracle

Comando Oracle Comando PostgreSQL
VARCHAR2(n) VARCHAR(n)
NUMBER(7) INTEGER
NUMBER(10,2) NUMERIC(10,2)
SYSDATE CURRENT_DATE
DROP ... CASCADE CONSTRAINTS DROP ... CASCADE

💡 Tips del Profesor

•

Use IDENTITY en lugar de SERIAL: En PostgreSQL 10+, use siempre GENERATED ALWAYS AS IDENTITY para evitar la manipulación desordenada de secuencias y garantizar compatibilidad con estándares.

•

Cuidado con los bloqueos de ALTER TABLE: Modificar tipos de columnas puede provocar una reescritura total de la tabla en disco, deteniendo la operación del negocio. Planifique DDLs disruptivos en horas de bajo tráfico.

•

Defina nombres explícitos para los constraints: No permita que el DBMS genere nombres automáticos (ej. table_column_key). Nombrar sus constraints manualmente facilita el mantenimiento y la lectura de logs de error.

✍️ Autoevaluación y Ejercicios Prácticos

1 ¿Cuál es la diferencia entre GENERATED ALWAYS y GENERATED BY DEFAULT para columnas de identidad?

GENERATED ALWAYS impide que el usuario proporcione un valor manualmente en un INSERT (bloquea la inserción explícita de IDs), mientras que GENERATED BY DEFAULT genera el valor automáticamente pero permite sobrescribirlo si es necesario.

2 ¿Por qué es riesgoso agregar una columna NOT NULL con DEFAULT en una tabla muy grande?

En versiones antiguas de PostgreSQL u otros motores tradicionales, esto requiere que el DBMS actualice físicamente cada fila del disco con el valor por defecto, lo que bloquea la tabla para escritura durante toda la operación.

3 ¿Qué comando de PostgreSQL permite crear un índice de forma asíncrona sin bloquear la tabla?

El comando es CREATE INDEX CONCURRENTLY nombre_indice ON tabla(columna);.
