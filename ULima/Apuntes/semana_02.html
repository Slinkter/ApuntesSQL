Insight del Arquitecto: La Abstraccion en Tres Niveles

Arquitectura ANSI/SPARC: Propuesta en 1975, establece que un sistema de BD debe separar la vision del usuario (externo), la vision logica global (conceptual) y la realidad fisica (interno). Esta separacion es la base de la independencia de datos.

Por que importa: Permite cambiar el almacenamiento fisico sin reescribir aplicaciones. En PostgreSQL, esto se traduce en la capacidad de agregar indices o cambiar tablespaces sin modificar una sola linea de codigo de aplicacion.

1. Datos, Informacion y Base de Datos

Repaso conceptual

Dato vs Informacion: Los datos son las unidades basicas de la informacion no procesada, que por si solos no ofrecen significado. Requieren de un analisis de su contexto para interpretar su significado. La informacion es el resultado del procesamiento de los datos, que nos entrega un resultado significativo para tomar decisiones.

Que es una Base de Datos?

  • Conjunto de datos persistentes utilizado por los sistemas de aplicaciones de una empresa.
  • Coleccion de datos interrelacionados creados con un DBMS.
  • Conjunto auto descriptivo de datos integrados.
  • Sirve para almacenar datos que luego pueden ser recuperados al hacer una consulta y convertirse en informacion.

2. Caracteristicas de una Base de Datos

  • CRUD: Create, Read, Update, Delete.
  • Consistencia: Los datos deberian ser los mismos en el transcurso del tiempo si no ha tenido modificaciones.
  • Validez: Datos validados.
  • Correccion facil de errores.
  • Velocidad en las operaciones.
  • ACID: Atomicidad, Consistencia, Aislamiento, Durabilidad.
  • Persistencia y Copias de Seguridad.
  • Portabilidad: Descargar datos relevantes de forma sencilla.
  • Seguridad.

3. Arquitectura de Tres Niveles

ANSI/SPARC
Nivel Descripcion Pregunta clave
Interno (Fisico) Describe la estructura fisica de almacenamiento: estrategia de almacenamiento, caminos de acceso, indices. Como se guardan los datos en disco?
Conceptual Describe la estructura de toda la base de datos para una comunidad de usuarios: vision global de los datos, interrelaciones, restricciones de integridad y seguridad. Que datos existen y como se relacionan?
Externo (Vistas) Incluye varios esquemas externos o vistas de usuario. Cada usuario ve solo los datos que necesita. Que ve cada usuario?

4. Independencia de Datos

Capacidad para modificar el esquema en un nivel del sistema de base de datos sin tener que modificar el esquema del nivel inmediato superior.

Tipo Descripcion
Independencia Logica Anadir, borrar o cambiar campos, registros o relaciones sin afectar aplicaciones existentes.
Independencia Fisica Cambiar caracteristicas fisicas, ubicacion de registros, esquemas de encriptacion sin afectar el nivel conceptual.

5. Tipos de Bases de Datos

Tipo Caracteristicas Ejemplos
Archivos Planos Archivos de texto sin ayuda para busqueda ni calculos. Utiles para informacion sencilla que raramente cambia. CSV, TXT
Jerarquicas Estructura de arbol. Funcionan bien si los datos tienen naturaleza jerarquica. IMS, XML, Registro de Windows
Relacionales Tablas con filas y columnas. Relaciones entre atributos comunes. Lenguaje SQL. Oracle, MSSQL, DB2, MySQL, PostgreSQL
Redes Nodos conectados mediante enlaces. Utiles para rutas mas cortas y flujos de red. Neo4J, CODASYL
Objetos (OODBMS) Abstracciones llamadas objetos. Bueno si la arquitectura de programacion usa objetos. O2, GemStone
Objeto-Relacional (ORDBMS) hibrido entre Relacional y Objetos. PostgreSQL, Oracle
Orientada a Documentos Documentos como registros no rigidos. Util para e-commerce. MongoDB
Dimensionales dimensiones en lugar de tablas. Busqueda de patrones en datos. Data Warehouses (OLAP)

6. Funciones del DBMS

  • Definicion de Datos (DDL): Crear, organizar y manipular la estructura de la base de datos.
  • Manipulacion de Datos (DML): Consultas, inserciones, actualizaciones y eliminaciones.
  • Administracion de Usuarios: Permisos, privilegios y recursos.
  • Administracion de Performance: Throughput, instancia, almacenamiento, memoria, bloqueos.
  • Control de Concurrencia: Gestion de bloqueos, commit y rollback.
  • Encriptacion de datos.
  • Administracion de Backup y Recovery: Backup, restore, recovery, archivos Log.
  • Optimizacion del desempeno.
  • Administrar el diccionario de datos.

  • Relacion (Tabla): Unidad basica de almacenamiento compuesta de filas y columnas.
  • Tupla (Fila): Un registro especifico que representa una entidad.
  • Atributo (Columna): Una propiedad de la entidad (nombre, telefono, etc.).
  • Dominio: Conjunto de valores permitidos para un atributo.
  • Primary Key (PK): Atributo o combinacion que identifica de manera unica un registro. No nula, inmutable, unica.
  • Foreign Key (FK): Atributo que referencia la PK de otra entidad para implementar asociaciones entre tablas.

9. Transacciones y ACID

Transaccion: Cualquier operacion de lectura o escritura realizada en la base de datos. Pasa por estados: activa, parcialmente comprometida, fallida, terminada, comprometida (COMMIT) o deshecha (ROLLBACK).

  • Atomicidad: Todo o nada. Si una parte falla, toda la transaccion se deshace.
  • Consistencia: La BD pasa de un estado valido a otro estado valido.
  • Independencia/Aislamiento: Las transacciones concurrentes no se afectan entre si.
  • Durabilidad: Una vez committeada, los cambios persisten incluso si hay falla.

10. Fases del Diseno de Base de Datos

Fase Actividades
Diseno Conceptual Recoleccion: identificacion de requerimientos, entidades y atributos. Refinamiento: cardinalidad, frecuencia de consultas, relaciones simples. Normalizacion.
Diseno Logico Determinar relaciones entre entidades. Asegurar que los requerimientos puedan ser soportados. Depende del tipo de DBMS.
Diseno Fisico Representacion fisica de entidades y atributos (tablas) a ser implementados en un DBMS especifico.

11. Metodologia Pratt-Adamski

  1. Representar las vistas de usuarios como un conjunto de relaciones.
  2. Normalizar las relaciones.
  3. Representar todas las claves (PK, FK).
  4. Determinar cualquier restriccion especial.
  5. Combinar el resultado de los pasos previos dentro del diseno final.

Supuestos: Los requerimientos son conocidos y se documentara durante el proceso de diseno.
