💡 Insight del Arquitecto: La Teoría de Codd

El modelo relacional: En 1970, Edgar F. Codd (IBM) publicó su trabajo fundamental "A Relational Model of Data for Large Shared Data Banks". Este modelo se basa en la teoría de conjuntos y lógica de predicados — no en punteros físicos como los modelos jerárquicos o de red.

Por qué importa: El modelo relacional es independiente del hardware y del software. El lenguaje declarativo (SQL) permite al usuario especificar QUÉ quiere, no CÓMO obtenerlo. Esto es posible gracias al optimizador del DBMS.

1. Arquitectura de tres niveles

Niveles de abstracción

Arquitectura ANSI/SPARC: Tres niveles de abstracción que proporcionan independencia de datos.

  • Nivel Interno (Físico):
    • Describe la estructura física de almacenamiento
    • Estrategias de indexación, paths de acceso
    • ¿Cómo se guardan los datos en disco?
  • Nivel Conceptual (Lógico):
    • Describe la estructura de TODA la base de datos
    • Integridad, seguridad, relaciones entre datos
    • ¿Qué datos existen y cómo se relacionan?
  • Nivel Externo (Vistas):
    • Múltiples vistas de diferentes usuarios
    • Cada aplicación ve solo lo que necesita
    • ¿Qué ve cada usuario específico?

  • Relación (Tabla): Colección de tuplas (filas) con atributos (columnas)
  • Tupla (Fila): Un registro específico de la tabla
  • Atributo (Columna): Una propiedad de la entidad
  • Dominio: Conjunto de valores permitidos para un atributo
  • Grado: Número de atributos
  • Cardinalidad: Número de tuplas

4. Claves en el Modelo Relacional

  • Primary Key (PK):
    • Identifica cada fila de manera única
    • No puede ser NULL
    • Debe ser única
    • ideally inmutable (no cambia en el tiempo)
  • Foreign Key (FK):
    • Establece relación entre tablas
    • Referencia a una PK de otra tabla
    • Mantiene integridad referencial
  • Clave Candidata: Atributo(s) que podrían ser PK

6. Transacciones y Propiedades ACID

Transacción: Conjunto de operaciones que se ejecutan como una unidad lógica de trabajo.

  • Atomicity: Todo o nada — si una parte falla, toda la transacción se deshace
  • Consistency: La BD pasa de un estado válido a otro estado válido
  • Isolation: Las transacciones concurrentes no se afectan entre sí
  • Durability: Una vez committeada, los cambios persisten incluso si hay falla

⚠️ Importante en PostgreSQL: Los comandos DDL (CREATE, ALTER, DROP) en PostgreSQL son transaccionales. Puedes hacer ROLLBACK de un CREATE TABLE si está dentro de una transacción.

7. Componentes del DBMS

  • Gestor de Datos: Lee/escribe datos en disco
  • Gestor de Transacciones: Controla ACID
  • Gestor de Concurrencia: Maneja bloqueos
  • Gestor de Recuperación: Backup/Recovery, WAL
  • Optimizador de Consultas: Determina el mejor plan de ejecución
  • Gestor de Catálogo: Almacena metadatos

8. Metodología de Diseño Pratt-Adamski

  1. Representar las vistas de usuarios como relaciones
  2. Normalizar las relaciones (1FN, 2FN, 3FN, BCNF)
  3. Representar todas las claves
  4. Determinar restricciones especiales
  5. Combinar resultados en el diseño final
