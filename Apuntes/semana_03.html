1. Fundamentos del Modelo ER

Conceptos por E.F. Codd (1970)

El Modelo Entidad-Relación (ER) fue propuesto por Peter Chen en 1976. Se fundamenta en:

  • Entidad: Objeto con existencia propia y distinguible. Ej: Cliente, Pedido, Producto
  • Atributo: Característica o propiedad de una entidad. Ej: Nombre, Edad, Precio
  • Relación: Asociación o conexión entre entidades. Ej: Cliente REALIZA Pedido

Concepto clave: Una entidad es un objeto que existe y es distinguible de otros objetos. Los atributos describen las características de las entidades. Las relaciones describen cómo las entidades se asocian entre sí.

2. Tipos de Entidades

  • Fuertes: Tienen existencia independiente. Su PK no depende de otra entidad.
  • Débiles: Su existencia depende de una entidad fuerte. No tienen PK propia.

4. Claves en el Modelo Relacional

  • Primary Key (PK): Identifica cada tupla de forma única. No puede ser NULL ni duplicada.
  • Foreign Key (FK): Apunta a una PK en otra tabla. Mantiene integridad referencial.
  • Clave Candidata: Atributo que podría ser PK. Es única y no nula.
  • Superclave: Conjunto de atributos que identifican唯一.

📝 Resumen: El modelo ER permite representar el mundo real mediante entidades (objetos), atributos (propiedades) y relaciones (asociaciones). La cardinalidad define cuántos elementos de una entidad se relacionan con otros. Las claves (PK/FK) garantizan la integridad y permiten las relaciones entre tablas.

EJERCICIO RESUELTO

Enunciado:

Diseña el modelo ER para una universidad: Estudiantes, Cursos, Profesores. Un estudiante puede inscribirse en muchos cursos. Un curso puede tener muchos estudiantes. Un profesor puede impartir muchos cursos.

Solución:
- ESTUDIANTE (PK id_estudiante, nombre, email)
- CURSO (PK id_curso, nombre, creditos)
- PROFESOR (PK id_profesor, nombre, especialidad)
- INSCRIPCION (FK id_estudiante, FK id_curso) - relación N:M
- IMPARTE (FK id_profesor, FK id_curso) - relación 1:N

QUIZ DE REPASO

Nivel 1 — Comprensión

P1: ¿Qué es una entidad?

R1: Un objeto con existencia propia y distinguible de otros (ej: Cliente, Producto).

Nivel 2 — Aplicación

P2: ¿Qué representa una relación 1:N?

R2: Que un elemento de una entidad se relaciona con muchos de otra (ej: Cliente - Pedidos).

Nivel 3 — Análisis

P3: ¿Por qué una tabla sin PRIMARY KEY es problemática?

R3: Porque no hay forma única de identificar cada fila, permitiendo duplicados y relaciones ambiguas.
