  • Diferenciar dato, información y conocimiento

1. Del Dato al Conocimiento

Conceptos Fundamentales
  • Dato: Representación simbólica de un hecho. Objetivo, sin contexto. Ej: 3.50
  • Información: Dato procesado con contexto. Reduce incertidumbre. Ej: "El precio del producto es 3.50"
  • Conocimiento: Información analizada para tomar decisiones. Ej: "El precio subió 75% en 3 años, debemos ajustar estrategia"

Concepto clave: Los datos son el insumo básico. La información surge al procesarlos. El conocimiento emerge al analizar la información para la toma de decisiones.

2. El Problema: Sistemas Tradicionales de Archivos

Antes del enfoque de BD

Antes de las bases de datos, cada aplicación tenía sus propios archivos:

  • Redundancia: Misma información duplicada en múltiples archivos
  • Dependencia programa-dato: Código tied to estructura física
  • Inconsistencia: Actualizar en un lugar pero no en otro
  • Acceso limitado: Sin forma estándar de compartir datos

3. La Solución: Enfoque de Base de Datos

Analogía: El DBMS actúa como un bibliotecario experto que conoce la ubicación exacta de cada libro (dato) y puede recuperarlo rápidamente sin que el usuario necesite saber dónde está físicamente almacenado.

  • Base de Datos: Colección de datos lógicamente relacionados
  • DBMS: Software intermediario entre usuarios y datos
  • Independencia: Cambios físicos no afectan aplicaciones
  • Seguridad: Control de acceso centralizado

5. Evolución de las Bases de Datos

  • 1960s: Modelos Jerárquicos (IMS) y Redes (CODASYL)
  • 1980s: Modelo Relacional (Oracle, DB2) - Estándar industria
  • 1990s: Orientados a Objetos y Objeto-Relacional
  • 2000s: NoSQL, Big Data, flexibilidad
  • 2010s: NewSQL (TiDB, CockroachDB), Cloud Databases
  • 2020s: Serverless, Edge Computing, AI/ML integration

📝 Resumen: Los datos son el insumo básico que al procesarse se convierten en información, y al analizarse para la toma de decisiones se transforman en conocimiento. Los sistemas tradicionales de archivos presentan problemas de redundancia e inconsistencia, que el enfoque de bases de datos resuelve mediante un DBMS que proporciona independencia, integridad y seguridad.

EJERCICIO RESUELTO

Enunciado:

Una empresa tiene 3 departamentos (Pedidos, Contabilidad, Nóminas). Cada uno tiene sus archivos. El archivo Customer_Master existe en Pedidos y Contabilidad.

Preguntas:

  1. ¿Qué problema causa la redundancia?
  2. Si cliente cambia dirección, ¿qué puede pasar?
  3. Si Contabilidad añade campo "descuento", ¿qué pasa en Pedidos?

Respuestas:
1. Redundancia: Datos duplicados en múltiples lugares
2. Inconsistencia: Si actualizan en un dept pero no en otro, datos desfasados
3. Dependencia: App de Pedidos deja de funcionar con campo desconocido

QUIZ DE REPASO

Nivel 1 — Comprensión directa

P1: ¿Cuál es la diferencia entre dato e información?

R1: Dato es un hecho bruto sin contexto (ej: 3.50). Información es dato procesado con significado (ej: el precio es 3.50).

Nivel 2 — Aplicación

P2: ¿Por qué la redundancia de datos es un problema?

R2: Porque puede causar inconsistencias - el mismo dato en diferentes lugares diverge con el tiempo.

Nivel 3 — Análisis

P3: ¿Qué pilar de administración se vulnera cuando un hacker accede a datos de clientes?

R3: La Seguridad - el acceso no autorizado viola este pilar.

Nivel 4 — Síntesis

P4: Si una empresa migra de archivos planos a una base de datos relacional, ¿qué beneficios inmediatos obtiene en términos de los 3 pilares?

R4: Disponibilidad: Acceso estructurado | Integridad: Constraints y normalización | Seguridad: Control de acceso centralizado
